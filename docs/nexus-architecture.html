Design Principle: Files-as-Truth

Markdown files on disk are always the authoritative source. The SQLite index, Tantivy search index, and knowledge graph are all fully rebuildable from the filesystem via nexus forge reindex. This means if any index is corrupted or lost, zero data loss — just rebuild. The Forge advisory lock prevents concurrent writers.

Multi-UI, Single Backend

All four UI surfaces (CLI, TUI, MCP, Desktop) share the same StorageEngine, Kernel, and AI engine. There's no separate server or API layer — each binary directly instantiates the crates it needs. The MCP server is launched via nexus mcp and communicates over stdio JSON-RPC.

Key Data Flows
How data moves through the system for the most important operations
1

File Change Detection & Reconciliation

When a user edits a markdown file on disk, the watcher detects the change and triggers a full reconciliation cycle — updating SQLite, Tantivy, and the knowledge graph.

User edits notes/hello.md on disk ↓ notify detects change (debounce 300ms) ↓ Watcher emits StorageEvent::FileModified { path } ↓ StorageCorePlugin receives via kernel event bus ↓ Reconciler: hash file → compare with index → update: ├→ SQLite index (blocks, links, tags, props) ├→ Tantivy search index (full-text) └→ KnowledgeGraph (link edges) ↓ Publish StorageEvent::Reconciled → TUI / CLI refresh
2

RAG Query (AI-Powered Q&A)

When a user asks a question via nexus ai ask, the RAG pipeline retrieves relevant blocks from the knowledge base, embeds them, and generates an AI response with source citations.

User: nexus ai ask "How does the plugin system work?" ↓ RAG::query(question) ↓ 1. SearchIndex::search(question) → top 5 block-level chunks 2. Vectorstore: embed question → cosine similarity search 3. Rerank by relevance 4. Build prompt: <context> blocks + <question> ↓ 5. Stream response from AI provider (Anthropic/OpenAI/Ollama) ↓ Output: Answer + sources (file paths + block IDs)
3

MCP Tool Invocation (AI Agent Access)

External AI agents (Claude Code, Cursor) invoke MCP tools over stdio JSON-RPC to read, write, and query the knowledge base.

Claude Code invokes: nexus_list_tasks { completed: false } ↓ MCP Server (stdio): receives CallToolRequest ↓ NexusMcpServer routes to handler: list_tasks() ↓ StorageEngine::query_tasks(filter) ↓ SQLite: SELECT * FROM tasks WHERE completed = 0 ↓ Return TaskRecord[] as JSON → stdout (JSON-RPC response) ↓ Claude receives tool results → uses in its response
4

WASM Plugin Execution

Community plugins run in a sandboxed WASM environment with capability-gated access to host functions.

User: nexus plugin call my-plugin some-command --args '{...}' ↓ PluginManager::call(plugin_id, command, args) ↓ PluginLoader::get(plugin_id) → check capabilities ↓ WasmSandbox: instantiate WASM module (wasmtime) ├ Fuel limit enforced ├ Memory limit enforced └ Host functions: storage, events, settings, KV ↓ WASM module executes → returns JSON result ↓ Output: text/json/table (formatted per --format flag)
5

Theme Resolution (Desktop App)

When the user toggles dark mode, the theme engine resolves CSS variables through a cascade and injects them into the React frontend.

User clicks "Dark mode" in Tauri app ↓ React → invoke Tauri IPC: set_mode("dark") ↓ ThemeEngine::apply_theme(mode: "dark") ↓ Resolution cascade: base_defaults (100 CSS vars) + theme_overrides(dark) + platform_overrides (OS accent color) + snippet_toggles + plugin_overrides ↓ Result: VariableMap { --color-bg: "#1E1E1E", ... } ↓ Serialize JSON → send to React → inject as CSS custom properties
6

Full-Text Search

Search queries hit the Tantivy inverted index at block level — returning sub-document results with snippet previews.

User: nexus content search "knowledge graph" ↓ SearchIndex::search(query, limit: 10) ↓ Tantivy inverted index (.forge/search/) BM25 scoring at block level (not note level) ↓ Return top N results with: ├ file path + block ID ├ snippet preview (highlighted) └ relevance score ↓ Output: formatted per --format flag (text/json/table)
Plugin System
5 core plugins (native Rust) + WASM sandbox for community extensions. Two trust tiers with 12 capability gates.

Two-Tier Architecture

Core plugins are compiled Rust with full kernel access — they run inside the process with no sandbox overhead. Community plugins are WASM modules sandboxed via wasmtime with fuel metering, memory limits, and capability-gated access to host functions. This gives maximum performance for built-in features while keeping third-party code isolated.

Core Plugins (5)

Storage Core Plugin

com.nexus.storage

Bridges the forge file watcher to the kernel event bus. Detects filesystem changes and translates them into typed events that drive search reindexing and graph updates.

Lifecycle: on_init verifies .forge dir exists · on_start starts watcher + bridge thread · on_stop signals + joins thread

com.nexus.storage.file_created com.nexus.storage.file_modified com.nexus.storage.file_deleted com.nexus.storage.file_renamed com.nexus.storage.indexing.started com.nexus.storage.indexing.completed

Security Core Plugin

com.nexus.security

Manages the security subsystem. Publishes audit events for security-relevant activity. Provides centralized access logging and risk assessment.

Lifecycle: on_init logs subsystem init · on_start publishes started event · on_stop publishes stopped event

com.nexus.security.started com.nexus.security.stopped com.nexus.security.audit.*

Database Core Plugin

com.nexus.database

Manages the database engine (20+ property types, formula language, relations). Publishes events for record CRUD and schema changes.

Lifecycle: on_init logs engine init · on_start publishes started · on_stop publishes stopped

com.nexus.database.started com.nexus.database.stopped

Git Core Plugin

com.nexus.git

Git integration via libgit2. Verifies the forge root is within a git repository. Enables git-aware features (status, log, diff, blame). Optional — succeeds gracefully if no repo found.

Lifecycle: on_init verifies git repository exists at forge root

IPC commands reserved for future (pending serde::Serialize derives)

AI Core Plugin

com.nexus.ai

AI engine integration. Auto-detects configured provider (Anthropic, OpenAI, Ollama) from environment variables and OS keyring. Enables embeddings, chat completions, and RAG queries. Optional — features disabled if no provider found.

Lifecycle: on_init detects AI provider from env vars + keyring

IPC commands reserved for future (pending async dispatch support)

CLI Plugin Commands

Management

  • nexus plugin install <dir> — Install from directory
  • nexus plugin uninstall <id> — Unload plugin
  • nexus plugin list — List all loaded plugins
