# nexus-plugins

> Kind: lib · IPC plugin id: — (registers no IPC handlers of its own; it *is* the dispatch/loader machinery) · CorePlugin: no · Has settings: per-plugin `[settings]` JSON-Schema (no crate-level config) · As of: 2026-05-25

## Overview

`nexus-plugins` is the **plugin runtime** for the microkernel. It owns everything between a directory of plugin files on disk and a running, capability-gated handler the rest of the system can call: manifest parsing and validation, the WASM sandbox (wasmtime), the host-function ABI, the plugin loader and lifecycle registry, per-plugin settings (JSON-Schema), hot-reload (notify), manifest signing (ed25519, BL-099), at-rest encryption of granted capabilities (ChaCha20-Poly1305, BL-101), and the concrete `IpcDispatcher` implementation that backs `KernelPluginContext::ipc_call`. The crate description sums it: "manifest parsing, WASM sandbox, host functions, plugin loader, settings, hot-reload."

The crate realises the **plugin-tiers** model from the architecture directly. A plugin is one of three runtime tiers, resolved in `manifest.rs` into `PluginRuntime::{Native, Wasm, Script}` and reified at load time as a `PluginBackend::{Core, Community, Script}`. **Core** plugins are native Rust (`Box<dyn CorePlugin>`), registered at bootstrap via `register_core`, bypass the WASM sandbox, and receive every capability unconditionally. **Community WASM** plugins are wasmtime-sandboxed, fuel/epoch-metered, memory-capped, and capability-gated at the host-function boundary. **Script** (JS) plugins carry no backend runtime state at all — `PluginBackend::Script` is a marker; their handlers execute in the Tauri WebView, and the loader only tracks their manifest, settings, and event subscriptions. The single `WasmSandbox::dispatch` ABI (`nexus_alloc` + `nexus_dispatch` + packed ptr/len return) is the boundary for the WASM tier.

On the "kernel routes, plugins owns lifecycle" split: this is accurate and visible here. The kernel defines the `IpcDispatcher` trait and the `KernelPluginContext::ipc_call` chokepoint but registers no handlers; the **concrete dispatcher lives in this crate** as `SharedPluginLoader` (a `Mutex<PluginLoader>` newtype implementing `IpcDispatcher`), and the **lifecycle registry** is `PluginLoader` (low-level) wrapped by `PluginManager` (high-level facade adding hot-reload, safe-mode, and crash/timeout quarantine). `CompositeIpcDispatcher` then stitches the two separate registries — community (`PluginManager`) and core (`SharedPluginLoader`) — together so a WASM plugin holding `ipc.call` can reach a core plugin's command without merging the registries.

Three cross-cutting security mechanisms are implemented in-crate and worth flagging up front: **HIGH-risk capability consent** (a community plugin's HIGH-risk caps are default-denied unless present in `granted_caps.json`, pinned to the plugin version), **at-rest encryption** of that grants file (BL-101 — ChaCha20-Poly1305 with a per-plugin keyring-stored key), and **manifest signing** (BL-099 — ed25519 verification against a trusted community keyring, opt-in via `require_signatures`). All three degrade safe: a missing/tampered grants blob resets to deny-all and forces re-consent; an unverifiable signature fails loud.

## IPC handlers

This crate **registers no IPC handlers of its own** — there is no `PLUGIN_ID` for it and no `[[registrations.ipc_command]]` block. Instead it *implements the dispatch machinery*:

- **`SharedPluginLoader as IpcDispatcher`** is the concrete dispatcher that `KernelPluginContext::ipc_call` routes through. `dispatch` resolves `(target, command)` under the loader mutex via `resolve_ipc`, **releases the loader lock**, detects same-thread reentrancy through a thread-local `DispatchGuard` (returns `IpcError::ReentrantCall` instead of deadlocking), then locks only the per-plugin backend mutex and dispatches. It instruments lock-wait + handler time and warns past `SLOW_SYNC_DISPATCH_WARN`. `dispatch_async` hands back a future for core async handlers, holding the loader lock only long enough to construct it.
- **`resolve_ipc`** finds the `handler_id` for a command, searching `ipc_commands`, then falling back to `ui_commands`, `ui_panels`, and `ui_settings_tabs` (so UI handlers are reachable by id).
- **`CompositeIpcDispatcher`** wraps a primary (community) dispatcher and a write-once `FallbackCell` (core). Only `IpcError::PluginNotFound` falls through to the fallback; `CommandNotFound` is returned as-is so a community plugin can't shadow a core command.
- The cap matrix surface (`add_cap_requirement`, `register_handler_*`, `required_caller_caps[_for_args]`, `is_handler_internal_only`) is consulted by the kernel context *before* dispatch; the completeness invariant (every live handler classified) is asserted by a bootstrap integration test, not at runtime.

CLI subcommands and the WASM `host::invoke_command` host function route through the same `dispatch_ipc`/dispatcher path.

## Capabilities

The capability granting model is implemented in `loader::build_capabilities` plus `grant_capability`/`revoke_capability` and the `grants_crypto` module:

- **Declaration.** A manifest's `[capabilities]` block lists `required` and `optional` capability strings. `validate` rejects any string `Capability::from_str` doesn't recognise.
- **Granting (load time).** Core plugins receive **every** `Capability::ALL` unconditionally (trusted shell). Community plugins receive the union of `required` + `optional`, **minus any HIGH-risk capability not present in `granted_caps.json`** (F-5.1.1). Denied HIGH-risk caps are logged at `audit = true`.
- **Persistence.** `granted_caps.json` lives in the plugin directory, pinned to the plugin **version** (a version bump resets grants → re-prompt). It is encrypted at rest (BL-101): an AEAD blob with magic header `NXENC1` + 12-byte nonce + ChaCha20-Poly1305 ciphertext. The 32-byte key is stored per-plugin in the OS keyring under `nexus.plugin_grants:<plugin_id>`. A decryption failure (rotated key or tampered file), a version mismatch, or a parse/UTF-8 error all reset to deny-all and force re-consent. `NEXUS_NO_KEYRING=1` falls back to plaintext JSON (leading `{`) with a `tracing::warn!`; reads dispatch on the first byte (`N` → encrypted, `{` → plaintext).
- **Enforcement.** WASM host functions check `caller.data().capabilities.contains(cap)` and call `audit::log_capability_denied` on denial (`host::kv_get`→`kv.read`, `kv_set`→`kv.write`, `emit_event`→`events.publish`, `read_file`→`fs.read`, `write_file`→`fs.write`, `invoke_command`→`ipc.call`, `notify`→`ui.notify`; `get_settings` is ungated as first-party). Cross-plugin IPC additionally enforces caller-side caps via the kernel context's `required_caller_caps[_for_args]` + `is_handler_internal_only` (P1-02 core-trust-only handlers).
- **Live revocation (BL-096).** `revoke_capability` mutates the running plugin's wired `KernelPluginContext` cap set in place (no restart), rewrites `granted_caps.json`, emits an audit entry, and publishes `com.nexus.kernel.capability_revoked` plus a universal-activity-timeline entry. Only HIGH-risk caps are revocable at runtime; non-HIGH-risk caps are auto-granted from the manifest.

## Settings / Config

There is no crate-level `Config` struct — settings are **per-plugin**, owned by `SettingsManager`:

- A manifest's optional `[settings]` block names a relative `schema` file (JSON Schema). At load, the loader reads it and calls `register_schema`. `validate` rejects a manifest whose declared schema file is absent.
- `load_settings` reads `<plugin_dir>/settings.json`; a missing file seeds defaults from the schema's top-level `properties[].default` (so `required` fields don't fail first-load). Every load/save runs `jsonschema` validation; without a registered schema everything passes.
- `save_settings` validates *before* writing — invalid settings never touch disk. `update_settings` additionally refreshes the shared `settings_cache` (`Arc<RwLock<String>>`, mirrored into the sandbox so `host::get_settings` reads the authoritative view) and fires `on_settings_changed` if declared.
- **WASM budgets** are manifest-declared in `[wasm]`: `memory_mb` (default 16, validated 1–256), `fuel` (default 10M, must be >0 for community plugins), `max_execution_ms` (default 5000). P1-09 clamps each against the kernel-wide `WasmCapsCeiling` at load (`clamp_wasm_cfg`), logging a warn on every clamp; `fuel == 0` from a community plugin is treated as an attempt to escape metering and forced to the ceiling.
- **Signing config** is kernel-level, mirrored in via `set_require_signatures`/`set_signature_verifier`; the trusted keyring is `~/.nexus/keys/community.json` + `revoked.json`.

## Events

- **Subscriptions.** A manifest's `[[registrations.event_subscriber]]` entries wire to the kernel `EventBus` at load via `parse_event_filter` (`""`/`"*"`→`All`; known kernel variant names→`Variant`; `".*"` suffix→`CustomPrefix`; else→`CustomExact`). `poll_events` drains each subscription and dispatches matching events to the plugin's handler, returning handler responses so the host can re-emit any `{events:[...]}` side channel. Disabled subscription ids persist to `<plugin_dir>/subscriptions.json` (corrupt file → preserved as `.corrupt-<unix>`, falls back to all-enabled).
- **Plugin-emitted events.** `host::emit_event` publishes to the kernel bus (`publish_plugin`) and forwards to the frontend via `PluginEventForwarder`; `host::notify` forwards a `ui.notification` toast.
- **Capability mutations** publish `com.nexus.kernel.capability_granted` / `capability_revoked` on the bus plus a universal `ActivityEntry` (`ActivitySurface::Capability`).
- **Hot-reload signaling.** `HotReloader` watches the plugins dir (notify, debounced) and emits `ReloadEvent { plugin_id, wasm_path }` for `.wasm` changes (plugin_id derived from the parent directory name; deletions skipped). `PluginManager::poll_reloads` drains and reloads; reload failures mark the plugin `PluginStatus::Crashed`.

## Internals & notable implementation details

- **WASM instantiation + limits.** `WasmSandbox::new` builds a wasmtime `Engine` with `wasm_simd` + `wasm_bulk_memory`, conditionally enabling `consume_fuel` and `epoch_interruption`. A `StoreLimits` enforces the `memory_mb` cap. `dispatch` resets the fuel budget each call (so a long-lived plugin doesn't accumulate usage) and arms a wall-clock watcher: a spawned thread sleeps `max_execution_ms` then `increment_epoch()` once; an `AtomicBool` cancels a late increment so it can't trip the next call. `map_trap_error` maps `Trap::OutOfFuel`/`Trap::Interrupt` → `ExecutionTimeout`, everything else → `ExecutionFailed`.
- **Dispatch ABI.** Args are serialised to JSON, `nexus_alloc(len)` reserves WASM memory, bytes are copied in, `nexus_dispatch(handler_id, ptr, len) -> u64` is called, the `u64` is unpacked as `(result_ptr<<32) | result_len`, and the result region is read back and JSON-deserialised.
- **Host functions** (`host_fns.rs`) registered on the `Linker`: `log` (rate-limited, lossy UTF-8), `kv_get`/`kv_set`, `emit_event`, `write_file`, `read_file`, `invoke_command`, `get_settings`, `notify`. Error codes: `HOST_OK=0`, `HOST_ERROR=-1`, `HOST_CAPABILITY_DENIED=-1001`, `HOST_BUFFER_OVERFLOW=-1002`. `write_file` uses `ForgePathValidator::validate_for_write` to close the canonicalize-parent-then-open TOCTOU race (F-5.3.1) and mkdirs only under the canonical target chain; `read_file` canonicalizes and prefix-checks against the (non-empty) forge root.
- **Reentrancy + concurrency.** A thread-local `ACTIVE_DISPATCHES` stack + `DispatchGuard` distinguishes a true same-thread reentrant call (→ `ReentrantCall`) from ordinary cross-thread contention (queues on the per-plugin `Mutex<PluginBackend>`). The backend mutex is held across the sync handler call because `CorePlugin::dispatch` takes `&mut self`.
- **Hot-reload (issue #74).** `reload_plugin` sets a `reloading` `AtomicBool` (concurrent IPC → `PluginReloading`) via an RAII guard, reads fresh WASM bytes, **re-evaluates capabilities from disk** (`refresh_capabilities` re-reads `granted_caps.json`), builds the new sandbox **before** stopping the old one (build failure leaves the old running; one 100ms retry for transient truncated-file errors), swaps in via `replace_sandbox`, clears quarantine, and re-injects the cached `IpcDispatcher` + `PluginEventForwarder`.
- **Quarantine watchdogs.** Two independent mechanisms: a persistent crash counter (`.nexus-crashes` text file, F-8.2.1 `max_crashes`) bumped on load failure and checked in `load_all`; and a consecutive-timeout streak (`timeout_streak`/`quarantined` atomics, PRD-04 §13 `max_timeout_streak`) tracked by `record_dispatch_result`/`check_quarantine`. Both reset via `nexus plugin reset <id>`.
- **Lifecycle timeout (BL-095).** `run_hook_with_timeout` runs core `on_init`/`on_start` on a named worker thread with an mpsc deadline (default 30s); on timeout the thread is detached so a hung hook can't block bootstrap (`LifecycleTimeout`). `Duration::ZERO` runs inline (tests).
- **Manifest signing (BL-099).** `verify_manifest_signature`: no signature + `require_signatures=false` → OK; no signature + required → reject; signature present → always verified (even with the gate off, so a malformed sig fails loud). Verification canonicalises by stripping the `[signature]` block, then `ed25519_dalek::VerifyingKey::verify_strict` against the trusted keyring; revocation list checked before lookup; only `ed25519` supported.
- **Validation rules (`validate`).** Rule 0 caps each registration vec at 1024 entries (issue #85 OOM guard); reverse-DNS id pattern; semver version; dependency ranges (valid `VersionReq`, reverse-DNS dep id, no self-dependency); known capability strings; `handler_id` uniqueness across non-IPC kinds (IPC commands may alias per ADR 0021, but IPC command *ids* must be unique); community = exactly one of `[wasm]`/`[script]` + module-file existence + memory/fuel bounds; core = no `[wasm]`/`[script]`; settings-schema-file existence. `api_version` major must equal `PLUGIN_API_VERSION_MAJOR` (F-9.2.1).
- **Scaffolding.** `scaffold` emits starter projects from embedded templates for `Core`/`Community` (Cargo + `manifest.toml` + `src/lib.rs`) and `Script` (`plugin.json` + `index.ts` + `package.json` pinning `@nexus/extension-api ^1.0.0` + `tsconfig.json` + `README.md`); validates plugin-id and requires an absent/empty output dir.
