# Rust Core Plugin Dependency Extract

Ground-truth extraction of declared + runtime + compile-time dependencies for
the 23 native Rust core plugins. Generated by reading manifest builders,
`ipc_call(...)` call sites, `subscribe(...)` filters, and per-crate
`Cargo.toml` files directly.

## Methodology notes

- **Declared `dependsOn`** — The `PluginManifest` schema (see
  `crates/nexus-plugins/src/manifest.rs`) does **not** define a `dependsOn`
  field. Core-plugin manifests are built inline by `core_manifest_with_ipc(...)`
  in `crates/nexus-bootstrap/src/plugins/mod.rs:112`, which emits only
  `[plugin]`, `[lifecycle]`, and `[[registrations.ipc_command]]` tables — no
  dependency declarations are possible from this builder. Grepping the entire
  workspace for `depends_on|dependsOn|depends-on` returns zero matches in
  `crates/`. Ordering is enforced purely by the call sequence in
  `register_all` (`crates/nexus-bootstrap/src/plugins/mod.rs:44`). For every
  plugin below, **Declared dependsOn:** is therefore "not supported by manifest
  schema" — listed once here to avoid repetition.
- **Infrastructure libs (skipped per scope):** `nexus-crdt`, `nexus-kv`,
  `nexus-remote`, `nexus-panic-log` are libraries, not registered core
  plugins. `nexus-kernel`, `nexus-types`, `nexus-plugin-api`, `nexus-plugins`
  are the kernel + leaf crates. `nexus-cli`, `nexus-tui`, `nexus-mcp` (binary
  layer), `nexus-bootstrap`, `nexus-fuzz` are frontends.

## Cross-cutting findings

- **No plugin declares `dependsOn` because the manifest schema does not support it.** Every dependency is implicit — either compile-time (Cargo path-dep, e.g. `nexus-storage → nexus-database/nexus-formats`) or runtime (`ipc_call` target plugin id, e.g. `nexus-editor → com.nexus.storage`). Boot ordering is hand-tuned in `crates/nexus-bootstrap/src/plugins/mod.rs::register_all` (comments at lines 53–82 document the rationale: security first, storage next, ai-runtime before ai for shared worker pool, collab last so the relay bridge sees every preceding plugin).
- **Implicit runtime deps via `ipc_call` count: 12 of 23** plugins call into at least one other plugin: editor, terminal, git (none — but receives), ai, ai-runtime, agent, skills, workflow, mcp, acp, audio, collab (publishes only). Plugins with **no** outbound `ipc_call`: security, storage, formats, database, git, templates, comments, linkpreview, notifications, theme, lsp, dap.
- **Compile-time Cargo path-deps that look like implicit hierarchies:**
  - `nexus-storage` depends on `nexus-database` + `nexus-formats` at compile time. Initial read flagged this as a leak; subsequent review (Phase 4.8) showed it is intentional — both target crates declare themselves as pure-logic libraries (`nexus-formats/src/lib.rs:5`, `nexus-database/src/lib.rs:3-8`) and the architecture deliberately puts `nexus-storage` above them as the SQLite-owning subsystem. `dep_invariants.rs` enforces the layering by forbidding `rusqlite` in `nexus-database`. Not a leak; see DEPENDENCIES.md §6 for the corrected analysis.
  - `nexus-ai` depends on `nexus-ai-runtime` at compile time and on `nexus-security`.
  - `nexus-git` and `nexus-audio` both compile-time-link `nexus-security` (likely for capability/credential types).
- **Surprises:**
  - `nexus-comments`, `nexus-linkpreview`, `nexus-templates` are *fully* leaf plugins — no IPC out, no subs, only depend on `nexus-plugins`.
  - `nexus-mcp` core plugin is itself trivial; the heavy IPC fan-out (`storage`, `skills`, `git`, `security`, `ai`) lives in the MCP **server binary** under the same crate, which is a frontend per the scope rules.
  - `nexus-storage` subscribes to `com.nexus.git.commit` (BL-114 indexing wake-up). This is a publisher→subscriber loop with `nexus-git`, but the coupling is loose: the subscriber path no-ops if git isn't loaded.
  - `nexus-agent` and `nexus-workflow` both have **dynamic dispatch** — the target plugin id is data, not a constant. This means their effective runtime dep set is unbounded (any plugin a user wires into a workflow step or tool call).
  - `nexus-skills` calls back into `nexus-agent::session_run`, and `nexus-agent` calls into `nexus-skills::triggered_by/compose/render` — a mutual runtime cycle that boot order alone doesn't enforce; both must be present for either to function fully.
