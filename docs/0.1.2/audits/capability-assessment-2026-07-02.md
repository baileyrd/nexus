## TL;DR — four cross-cutting patterns

**1. The last mile is the gap.** Roughly a fifth of the findings are not
missing features but **complete, tested machinery with no consumer**: plugin
hot-reload (C80), the ACP host (C34), the agent snapshot/rollback store (C29),
the process-manager FSM + pre-command runner (C54), sandboxed terminal
sessions (C55), `OpLog::missing_for` late-join sync (C59), terminal scrollback
restore (C53), `ThemeWatcher` (C87), the deep-link pipeline (C71), the
session-restore `eState` channel (C52), the ADR 0028 runtime observe/control
surface (C79), the reserved quick-switcher command (C5). The
feature-per-effort ratio of finishing these is far higher than greenfield
work.

**2. Shipped settings that lie.** Six persisted settings are read by no code:
"Automatically update internal links" (C2), "Deleted files → Move to system
trash" (C3), "Spellcheck" + languages (C4), `hot_reload_enabled` (C80),
"Default location for new attachments" (C1), and the daily-notes pair (C6).
Each promises behavior the product silently does not deliver — worse than a
missing feature, because the user thinks it is on. Wire them or remove them
(same reasoning the 2026-07-01 audit applied to the settings stub pages, but
these six are live toggles, not stubs).

**3. Ingestion and egress are the thinnest walls.** The engine is strong, but
getting content **in** (no image paste/drop anywhere — three independent
surveyors converged on it, C1; one importer total, C9; no web clipper, C10; no
voice capture despite a shipped audio subsystem, C11) and **out** (PDF export
prints the whole workbench, C65; the styled HTML exporter has no IPC handler,
C66; exports mangle wikilinks/callouts/frontmatter, C67) trails the core by a
wide margin.

**4. Frontend-parity drift.** The four-frontend promise is eroding at the
edges: memory has no CLI/TUI surface though docs claim one (C39), comments are
shell-only (C74), the Dream Cycle scheduler runs only under the TUI (C46),
hybrid/semantic search landed shell-only (C78), workflow triggers run only
while the desktop shell is open (C76), and MCP clients can reach neither agent
sessions nor workflows (C75). A recurring root cause: subsystems keep
shipping their "primary" frontend and dropping the other three.

### C1. Notes have no image/attachment pipeline: markdown images never resolve, and there is no paste-screenshot or OS file-drop to create attachments

`capability-gap` · value **high** · effort **medium** · sweep `shell-ux` · **independently corroborated** by the `editor-devtools, product-pkm` sweeps

In the markdown reading view (shell/src/plugins/nexus/editor/markdownRender.ts) `![](img.png)` passes through marked+DOMPurify with no src rewriting, so forge-relative images render as broken img tags — the CSP allows asset:/http://asset.localhost img-src (shell/src-tauri/tauri.conf.json:31,43) but nothing in shell/src calls convertFileSrc — and live preview explicitly ships mark-only image styling ("v1: mark-only. The block-widget swap lands in Phase 2", cm/livePreviewDecorations.ts:615) with that Phase 2 tracked in no backlog item or issue. There are zero clipboardData/onPaste or Tauri onDragDropEvent handlers in shell/src, so pasting a screenshot or dropping an OS file does nothing, opening a .png from the file tree lands in a bare-text CodeMirror buffer (editor/codeMode.ts), and the nexus.settings.files.defaultAttachmentLocation setting (SettingsPanelView.tsx:977) is read by no code. The plumbing already exists — com.nexus.storage::read_file/write_file carry raw bytes (crates/nexus-storage/src/ipc.rs:140,296) and the canvas file-node overlay already renders forge images as data: URLs from those bytes (canvas/CanvasOverlay.tsx:592-660,821) — so the work is an editor-side src rewriter (asset protocol or the proven data-URL path), a paste/drop handler that writes bytes to the attachment folder and inserts a link, and the live-preview image widget. This is distinct from the tracked "attachment intelligence (PDF/OCR extraction into the index)" direction, which parses existing attachments for search and neither renders nor creates them.

> **Evidence:** shell/src/plugins/nexus/editor/markdownRender.ts (full read — wikilinks/callouts/pills handled, no img token or src rewriting; DOMPurify.sanitize at line 413); grep for 'convertFileSrc|asset://|onDragDropEvent|tauri://drag|clipboardData|onPaste' across shell/src returned zero hits; shell/src-tauri/tauri.conf.json CSP img-src allows asset: but nothing consumes it; grep 'binary|base64' across crates/nexus-storage/src/ipc.rs and handlers/*.rs → only a non-UTF-8 comment in handlers/files.rs:90; settings define an attachment folder (shell/src/plugins/core/settings/SettingsPanelView.tsx:973-977 'Default location for new attachments') that no code reads (grep 'defaultAttachmentLocation' → settings panel only); shell/src/plugins/nexus/editor/codeMode.ts routes unknown extensions as plain text. Searched under 'attachment', 'image', 'asset', 'paste', 'drop' to confirm no other crate/plugin provides this.

### C2. Renaming or moving a note never rewrites inbound links, silently breaking wikilinks — despite a shipped "Automatically update internal links" setting that no code reads

`capability-gap` · value **high** · effort **medium** · sweep `storage-search` · **independently corroborated** by the `editor-devtools` sweep

StorageEngine::rename_entry (crates/nexus-storage/src/lib.rs:1195-1237) does fs::rename and re-keys only the renamed file's own index row, FTS rows, and graph node; it never rewrites the referencing markdown files, so under file-as-truth every inbound wikilink/markdown link is stale on disk immediately, and flips to phantom in the index as soon as each referencing file is re-indexed (resolve_link, index.rs:536, matches by path/stem only) or the index is rebuilt. This is live user-facing breakage: the P4-07 rename/move tab commands (shell/src/plugins/nexus/editor/index.ts:1428-1476) and the files-tree drag/rename all call rename_entry with no link handling and no prompt. The settings panel already ships a wired, value-persisting "Automatically update internal links" toggle whose key nexus.settings.links.autoUpdate appears nowhere else in the codebase (SettingsPanelView.tsx:1007-1017), and whose own description promises a rename-time prompt that also does not exist — P4-06/P4-07 were marked shipped 2026-05-18, so no open tracker covers this. The pieces to implement it exist: the links table records every referencing file and link text, and the IPC-exposed replace_in_files (BL-078, find_replace.rs:524) provides a write-path-safe multi-file rewrite.

> **Evidence:** crates/nexus-storage/src/lib.rs:1195-1237 (rename_entry: fs::rename + re-key own rows only; no link rewrite); crates/nexus-storage/src/index.rs:647-653 (invalidate_links_to flips inbound links to phantom on delete — same fate on rename); shell/src/plugins/core/settings/SettingsPanelView.tsx:1007-1015 (stub toggle 'nexus.settings.links.autoUpdate' — grep shows this key appears nowhere else in shell/ or crates/). Searched 'update.?links|rewrite.?links|renameLinks|fix.?links' repo-wide: only the stub toggle, an archived help doc, and unrelated SQL in index.rs.

### C3. Deletes are always permanent despite a shipped "Deleted files → Move to system trash" setting; add .trash staging + restore for notes/folders

`capability-gap` · value **high** · effort **medium** · sweep `storage-search`

Every delete path destroys file bytes immediately: com.nexus.storage delete_file runs std::fs::remove_file (crates/nexus-storage/src/lib.rs:984-989), delete_entry runs remove_dir_all soft-deleting only SQLite index rows (lib.rs:1254-1281), and the shell file tree (shell/src/plugins/nexus/files/index.ts:294-319 → kernelClient.ts:90-95) and CLI nexus delete (crates/nexus-cli/src/commands/content.rs:172-198) both route there — the confirm dialog literally says "This cannot be undone." Yet the Settings panel already ships a "Deleted files" select (nexus.settings.files.deletedFilesDestination, SettingsPanelView.tsx:1070-1085) whose default value is "Move to system trash" with a "Move to .trash in forge" option — a key read nowhere in the repo (one grep hit repo-wide), so the UI promises recovery behavior no backend implements, and shell/src-tauri has no OS-trash plugin. Base records are the proven pattern: base_record_soft_delete/base_record_restore (lib.rs:905/930) plus a full Trash view with restore and "Delete forever" (shell/src/plugins/nexus/bases/), leaving notes — the primary content type — as the only data that dies unrecoverably. Wiring the existing setting to a forge-level .trash/ move (index-excluded like .forge/) or OS trash, plus restore/empty-trash IPC handlers and a CLI verb, closes the gap; this is distinct from the tracked cp-stub:file-recovery snapshot-daemon stub page and from the git version-history future direction (AutoCommitter only helps git opt-in users).

> **Evidence:** crates/nexus-storage/src/lib.rs:984-989 (delete_file → fs::remove_file), lib.rs:1254-1281 (delete_entry → fs::remove_dir_all); shell/src/plugins/nexus/files/index.ts:307 and files/kernelClient.ts:90-92 (UI delete → storage delete_entry directly); grep '(?i)trash' in shell/src-tauri → no matches (no OS-trash Tauri plugin). Contrast: base-record trash exists (docs/0.1.2/crates/nexus-storage.md handler rows base_record_soft_delete/restore; shell/src/plugins/nexus/bases/basesStore.test.ts:119). Searched trash|recycle|soft-delete repo-wide: soft-delete exists only for index rows and base records, never for file bytes.

### C4. Editor spellcheck toggle persists but is consumed by nothing — CodeMirror leaves the webview's native checker hard-disabled

`capability-gap` · value **high** · effort **small** · sweep `editor-devtools` · **independently corroborated** by the `shell-ux` sweep

The Settings panel ships a "Spellcheck" toggle (default ON, copy: "Turn on the spellchecker") plus a languages select that persist nexus.settings.editor.spellcheck/spellcheckLanguages to configStore, but no code consumes either key — the editor's live config keys (nexus.editor.*, editorConstants.ts:118-155) have no spellcheck entry, and the vendored @codemirror/view 6.41.1 hard-sets spellcheck="false"/autocorrect="off" on the contentDOM with no EditorView.contentAttributes override anywhere in shell/src, so the webview's native checker never runs and there is no dictionary/hunspell integration in crates/. First increment: consume the existing toggle via EditorView.contentAttributes {spellcheck:'true', lang} — this lights up native checking on WebView2/WKWebView immediately; note webkit2gtk additionally needs spell checking enabled on the WebKit context (enchant), which may require a small Tauri-side setting for Linux, with a custom-dictionary layer as a follow-up.

> **Evidence:** shell/src/plugins/nexus/editor/cm/extensions.ts:87-158 (baselineExtensions — no contentAttributes/spellcheck); shell/src/plugins/nexus/editor/editorConstants.ts:118-155 (live editor config keys, no spellcheck); shell/src/plugins/core/settings/SettingsPanelView.tsx:817-836 (toggle writes nexus.settings.editor.spellcheck; grep shows no consumer outside this file). Searched 'spell|spellcheck|hunspell|dictionary|grammar' across crates/ and shell/src — only settings stubs, spellCheck={false} attributes on inputs, and archived Obsidian plugin research match.

### C5. File quick-switcher (Ctrl+P) with per-forge recent files — reserved in code, anticipated in three places, never built

`ux-gap` · value **high** · effort **small** · sweep `product-pkm`

Nexus has no way to open a note by typing its name — the highest-frequency navigation primitive in Obsidian/Logseq — even though the codebase anticipates it in three places: Ctrl+P is bound as a command-palette alias with a comment "to be reclaimed when a file quick-open plugin lands" (shell/src/plugins/nexus/commandPalette/index.ts:36-39), the editor empty-state "Go to file" link falls back to the command palette (EditorView.tsx:749-751), and Settings ships a non-functional "Quick switcher" stub page whose nexus.settings.quickSwitcher.* keys nothing reads (SettingsStubPages.tsx:453-480). The nearest existing surfaces don't cover it — nexus.search/searchPanel are Tantivy content-FTS panes and the project's own Obsidian-parity research rates the quick switcher 🟡 "maps loosely" to Search — and recents exist only as forge-level workspace paths in the launcher, with no per-forge recently-opened-files list anywhere. The building blocks are already in place (generic nexus.pick modal, the palette's subsequence matcher, com.nexus.storage's list_dir IPC handler and StorageEngine::list_files over the SQLite files table), making this a small shell plugin: a file-mode palette on Ctrl+P seeded with recents, with create-on-Enter for no-match queries. Caveat: the stub settings page itself is tracked in gap-analysis-2026-07-01.md ("implement or remove"), but the switcher feature carries no BL-*/WI-* marker, issue, or plan.

> **Evidence:** shell/src/plugins/nexus/commandPalette/index.ts:36-39 (Ctrl+P alias comment anticipating a 'file quick-open plugin'); shell/src/plugins/nexus/commandPalette/match.ts:14 (fuzzy match over commands only); shell/src/plugins/nexus/editor/EditorView.tsx:751 ('Go to file' → command palette); shell/src/plugins/core/settings/SettingsStubPages.tsx:453-472 (quick-switcher settings keys nexus.settings.quickSwitcher.* that nothing reads); shell/src/plugins/nexus/launcher/index.ts:1-17 (recents are forge-level workspace paths only). Searched: 'quickOpen|go to file|goToFile|fuzzy|ctrl+p', 'openByPath' (programmatic command only, templates/index.ts:118), 'recent' across shell/src/plugins/nexus — no file switcher or per-forge recent-files feature anywhere.

### C6. Daily notes are CLI-only and internally inconsistent: no shell open-today command or calendar navigation, and both daily-notes settings surfaces are dead

`ux-gap` · value **high** · effort **medium** · sweep `product-pkm`

Daily notes — the core ritual of Logseq, Reflect, and Obsidian — exist only as `nexus content daily`, which hardcodes `notes/daily/%Y-%m-%d.md` with a fixed skeleton and bypasses the templates system entirely (crates/nexus-cli/src/commands/content.rs:384-420, scoped that way by closed BL-001); the desktop shell has no open-today command, hotkey, prev/next-day navigation, or daily calendar (BasesCalendar renders .base records only), leaving only the generic multi-prompt "New from template…" flow as a non-idempotent workaround. > **Evidence:** crates/nexus-cli/src/commands/content.rs:383-415 (CLI-only, hardcoded path + skeleton, no template use); grep -i 'daily|today' over shell/src/plugins/nexus returns no daily-note command; shell/src/plugins/core/settings/SettingsStubPages.tsx:247-282 (unwired nexus.settings.dailyNotes.dateFormat/fileLocation/templateLocation); crates/nexus-workflow/templates/daily-journal.workflow.toml (cron writes a fixed skeleton, no open/navigate UX); docs/0.1.2/application-capabilities.md:74 (Calendar view exists only inside bases). > **Evidence:** crates/nexus-storage/src/tasks.rs:13-21 (ParsedTask fields), :47-54 (TaskFilter = completed + file_path only); crates/nexus-storage/src/handlers/tasks.rs (query_tasks/toggle_task IPC exist); grep 'query_tasks|toggle_task' over shell/src → zero matches; crates/nexus-tui/src/ui/tasks.rs (TUI pane exists); crates/nexus-cli/src/args.rs:131-146 (CLI tasks/task-toggle). ### C8. Saved searches / smart views: persist and pin named FTS + scoped queries (Bases covers only frontmatter properties)

`feature-opportunity` · value **medium** · effort **medium** · sweep `storage-search`

> **Evidence:** shell/src/plugins/nexus/search/searchStore.ts (67 lines, transient state only; grep saved|history|recent → no hits), shell/src/plugins/nexus/searchPanel/searchPanelStore.ts (same), shell/src/plugins/nexus/bookmarks/index.tsx:16-20 (BookmarkEntry = { relpath, label, createdAt } — files only); grep '(?i)saved.?search|saved.?quer' repo-wide → only archived ADR/PRD prose. ### C10. Web capture ingestion path: a "clip URL to markdown note" IPC verb (with the agent's dormant web.fetch capability as a second consumer)

`feature-opportunity` · value **high** · effort **medium** · sweep `product-pkm` · **independently corroborated** by the `interop-ecosystem` sweep

Nexus has no way to turn a web page into a note: nexus-linkpreview's single IPC verb `fetch` deliberately returns only OG/Twitter-card metadata (regex-based, 5 s timeout, 512 KB cap — lib.rs:1-49), nexus-formats converts markdown→HTML for export but has no HTML→markdown import, and the CLI import command handles only Notion zips. A `clip_url` IPC verb (readability-style extraction → markdown with url/date frontmatter → write via com.nexus.storage) would follow the project's own extension recipe (application-capabilities.md:223) and instantly serve the command palette, CLI, MCP, workflows, and agents — matching the top ingestion path of Obsidian/Notion/Reflect, and it is not listed among the project's deliberate non-goals or any tracked backlog item.

> **Evidence:** crates/nexus-linkpreview/src/lib.rs:1-40 (scope explicitly limited to OG metadata cards for the canvas; 5 s timeout, 512 KB body cap); crates/nexus-formats/src/ (markdown/notion/canvas modules only — no HTML→markdown import). Searched: 'clipper|readability|read.?later|html_to_markdown|turndown|url_to|import_url|scrape' repo-wide → matches only in archived research docs (docs/archive/pre-0.1.2/research/obsidian-community-plugins.json). docs/0.1.2/application-capabilities.md:202-204 confirms link preview is the only web-facing content feature.

### C17. Watcher file events and ReconcileRequested are published but never consumed for index updates — external edits leave SQLite/FTS/graph stale and the AI daemon re-embeds stale content

### C89. No plugin-author dev loop: scaffold emits no tests/CI, the typed PluginAPI stub is shell-internal, script plugins have no watch mode, and most API surfaces lack worked examples

`ux-gap` · value **medium** · effort **medium** · sweep `interop-ecosystem`

`nexus plugin scaffold --type script` emits exactly five files (plugin.json, index.ts, package.json with only `build`/`typecheck` scripts, tsconfig.json, README.md — crates/nexus-plugins/src/scaffold.rs) and the WASM template even less, so plugin authors get no test file, no test runner, no CI workflow, and no mock host — even though the shell already maintains a typed `stubPluginAPI` helper (shell/src/testing/typedStubs.ts) that is not shipped in @nexus/extension-api (package.json `files`: dist/src/README only). There is also no dev iteration loop for script plugins: shell/src/main.tsx loads community bundles once at startup via communityPluginLoader.ts, and hot-reload (crates/nexus-plugins HotReloader, notify-debouncer-mini) exists only for kernel-side WASM; the CLI has no `plugin dev` verb. The in-repo examples cover commands, notifications, input.prompt, a declarative panel (hello-world), settings access (hello-js), and fenced-code/viewRegistry rendering (mermaid), but kernel event subscription, platform.fs, uri handlers, kv storage, and activity-bar contributions have no worked example, and the authoring guide survives only in docs/archive/pre-0.1.2/shell/writing-a-plugin.md (docs/0.1.2/plugins/community.md has a ~12-line quickstart). With #187 conformance landed and the marketplace (WI-44) queued as the next ecosystem step, an exported test-stub package, scaffold test+CI files, a watch/dev mode, and per-surface examples are the untracked precondition to third-party plugin growth.

> **Evidence:** crates/nexus-plugins/templates/script/package.json (scripts: build, typecheck only); crates/nexus-plugins/src/scaffold.rs:60-126 (emitted files: manifest/lib.rs or plugin.json/index.ts/package.json/tsconfig/README — no tests or CI); shell/src/testing/typedStubs.ts (shell-internal only; packages/nexus-extension-api/package.json ships just dist+src); shell/src/host/communityPluginLoader.ts (no watch/hot-reload — grepped watch|reload|hot); examples inventory: plugins/{hello-js,hello-nexus,first-party-*}, shell/src/plugins/community/{hello-world,mermaid}; the authoring guide lives only in the archive (docs/archive/pre-0.1.2/shell/writing-a-plugin.md) — docs/0.1.2/plugins/community.md has a 12-line quickstart.

### C90. Generic webhook notification transport (Slack/ntfy/Gotify/Matrix/arbitrary JSON POST) — explicitly deferred in BL-135 but never tracked

`feature-opportunity` · value **medium** · effort **small** · sweep `interop-ecosystem`

The notifications dispatcher supports exactly four transports — Desktop, Discord, Telegram, SMTP email (Channel enum, crates/nexus-notifications/src/lib.rs:123-134; ChannelsConfig, config.rs:252-262) — and tests explicitly pin rejection of other channel names (config.rs:534, nexus-cli notify.rs:145), so Slack/ntfy/Gotify/Matrix/home-automation users cannot receive unattended agent or workflow notifications; no workaround exists since workflows have no HTTP step and no arbitrary-POST IPC verb is registered in cap_matrix.toml. The archived BL-135 closure note (docs/archive/pre-0.1.2/PRDs/backlog/BL-135.md:39) deferred "Slack / Matrix / ntfy / web-push" transports "until someone asks" but filed no backlog item, and neither the 2026-07-01 gap analysis nor any open issue tracks it. A single generic webhook transport (configurable URL, optional headers/body template) would reuse the existing blocking_webhook_client() plumbing (lib.rs:251) and inherit BL-135 source routing, severity/quiet-hours filtering, the BL-136 inbox, and the workflow notify step unchanged; the Channel enum's wire form is documented append-only, so adding a variant (e.g. a single Webhook variant keyed to named [channels.webhook.<name>] config blocks) is contemplated by the design, though the two negative tests pinning channel-name rejection would need updating.

> **Evidence:** crates/nexus-notifications/src/lib.rs:123-134 (Channel enum: Desktop/Discord/Telegram/Email — doc notes wire form is append-only); crates/nexus-notifications/src/config.rs (ChannelsConfig has only discord/telegram/email blocks); grepped `slack|ntfy|gotify|matrix|generic.*webhook` in crates/nexus-notifications/src — only the Discord-specific webhook exists; crates/nexus-workflow/src/handlers/run.rs:611-669 (notify step routes into the same fixed channel set).
