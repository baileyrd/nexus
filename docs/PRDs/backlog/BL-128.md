Adds a second graph layer orthogonal to the existing document-link graph: typed entities (`person`, `place`, `project`, …) with typed directional relations (`knows`, `lives_in`, `works_on`, …) and confidence scores. Entities live as plain markdown files under `<forge>/entities/` — same file-as-truth invariant as the rest of storage. FAISS-backed semantic recall layers on top via a new `com.nexus.ai::entity_recall` handler that reuses the shared chunk vectorstore (entity files are auto-embedded by the existing indexing daemon alongside every other markdown file); no parallel FAISS index needed.

**What landed.**

- **`entity_index` module** (`crates/nexus-storage/src/entity_index.rs`). Parses every `*.md` under `<forge>/entities/` into an in-memory `EntityIndex` per IPC call (sub-millisecond for O(dozens) of entity files; the per-call rebuild keeps the wire shape stable for a future watcher-cached layer). Frontmatter shape: `entity_type`, `aliases`, `description`, `relations: [{target, type, confidence}]`. Missing `description` falls back to the first non-empty body paragraph truncated at 240 chars. Aliases that collide with another entity's canonical id silently lose; alias targets in relation edges are resolved to canonical ids at query time, mirroring the wikilink resolution convention.
- **Canonical vocabularies.** `ENTITY_TYPES` (11 entries: `person`, `preference`, `fact`, `event`, `place`, `project`, `organisation`, `concept`, `skill`, `media`, `self_knowledge`). `RELATION_TYPES` (40 entries grouped by family, location, work, knowledge, media, ownership, temporal, causality, generic). `normalize_relation_type(input)` lowercases, replaces `' '` / `'-'` with `'_'`, looks up a 35-entry alias table (`"spouse"` → `"married_to"`, `"reports to"` → `"managed_by"`, `"works for"` → `"employed_by"`, etc.); unknown inputs fall back to `"related_to"`. Applied on the write path (`entity_upsert`) so the on-disk file always carries canonical vocabulary regardless of what the LLM or user submitted.
- **Five IPC handlers** under `com.nexus.storage`:
  - `entity_search` (handler 64) — substring scoring over `id` (exact 100, contains 75), `aliases` (exact 90, contains 60), `description` (contains 40). Optional `entity_type` filter; empty query returns lexicographically-first hits up to `limit`.
  - `entity_get` (handler 65) — canonical-id-then-alias lookup, full payload + outgoing relations.
  - `entity_relations` (handler 66) — `outgoing` / `incoming` / `both` directional traversal, stable ordering by `(from, to, kind)`.
  - `entity_upsert` (handler 67) — file-as-truth write-through. Renders the payload via `render_entity_markdown`, writes through `crate::atomic_write` (temp-fsync-rename in `<forge>/.forge/temp`) so a concurrent read sees old-or-new content, never a torn write. Validates `id` is a bare file stem (rejects `'/'` / `'\'` / `'..'`); rejects empty `id` or `entity_type`. Returns `{ relpath, replaced }`.
  - `entity_find_duplicates` (handler 68) — Jaccard token similarity over `id + aliases + description`, restricted to same `entity_type` pairs. Threshold defaults to `0.92` (Thoth's review threshold). Pairs sorted by descending similarity then ascending `(a, b)`.
- **Bootstrap helpers + CLI.** `nexus-bootstrap::storage` exports typed client wrappers for all five storage handlers + DTOs. New `nexus graph entity` subcommand tree: `list [--type T] [-l N]`, `show <id>`, `search <query> [--type T] [-l N]`, `related <id> [--direction outgoing|incoming|both]`, `duplicates [--threshold 0.92]`. All five honour `--format json|jsonl` for scripting.
- **FAISS-backed semantic recall** (`com.nexus.ai::entity_recall`, handler 23). Pipeline: embed `query` through the configured provider → query the shared chunk vectorstore with a 5x oversample so multi-block entities aren't squeezed out → keep hits whose `file_path` starts with `entities/` → group by file, keeping the max score per entity → resolve each surviving stem through `com.nexus.storage::entity_get` → return `EntityRecallHitRow`s ranked by descending cosine similarity. Empty query returns `{ results: [] }` (not an error) so the agent fallback path treats "no matches" identically to "embedder missing". Stems whose file lives under `entities/` but lacks `entity_type` frontmatter are dropped silently (recall must not surface non-entities). The new `EntityRecallHitRow` is shape-compatible with the substring path's `EntitySearchHitRow` apart from the `score` units, so the agent's renderer is a single pure function over both wire shapes.
- **Agent integration.** `nexus-agent::core_plugin::compose_entity_preamble` tries `com.nexus.ai::entity_recall` first (15s timeout — embedding-based, slower); falls back to `com.nexus.storage::entity_search` (5s, substring-only) on any error: missing embedder, IPC plumbing failure, no AI provider, zero hits. Renders the top 5 as a `"Known entities relevant to this goal (from the forge's `entities/` directory):"` block spliced into the planner system prompt next to the existing memory + skills preambles. A forge without an `entities/` directory or without an AI provider pays nothing.
- **13 IPC wire types** under `feature = "ts-export"` derive: storage — `EntitySearchArgs` / `EntitySearchHitRow` / `EntitySearchResult`, `EntityGetArgs` / `EntityRecordRow` / `EntityRelationRow` / `EntityGetResult`, `EntityRelationsArgs` / `EntityRelationsResultRow` / `EntityRelationsResult`, `EntityUpsertRelationRow` / `EntityUpsertArgs` / `EntityUpsertResult`, `EntityFindDuplicatesArgs` / `EntityDuplicatePairRow` / `EntityFindDuplicatesResult`; ai — `EntityRecallArgs` / `EntityRecallHitRow` / `EntityRecallResult`. `scripts/check_ipc_drift.sh` regenerated all 19 TS bindings + 19 JSON Schemas across the two plugin namespaces.

**Definition of done coverage**:
- ✅ `nexus-storage` parses `entity_type` + `relations` frontmatter (deliberately as a file-backed `EntityIndex` rather than a SQLite-table extension — entities re-index sub-millisecond at the expected scale).
- ❌ Petgraph edges extended with `type: String` / `confidence: f32` — **dropped from BL-128 scope after re-evaluation**. The existing `entity_relations` IPC handler already exposes typed traversal over the entity edge set, and the only callers needing typed edges are entity-aware (agent prompt assembly, CLI `related`, BL-129 Dream Cycle). Folding typed entity edges into the document `petgraph` would either (a) require every existing `EdgeData` consumer to handle a new variant for a property they don't read, or (b) introduce ambiguity in `graph_neighbors` between wikilink hops and entity relations. Both worse than keeping the two graphs separate. If a future caller genuinely needs union traversal, file a fresh BL.
- ✅ FAISS embedding index over entity content — landed via the new `com.nexus.ai::entity_recall` handler. Layers on the existing shared chunk vectorstore (the indexing daemon already embeds every markdown file under the forge, entity files included), so no parallel index needed. The agent's `compose_entity_preamble` prefers this path and falls back to the substring ranker when no embedder is configured.
- ✅ `normalize_relation_type` table covering the 40-relation vocabulary (35-entry alias table + identity passthrough + `"related_to"` fallback).
- ✅ Five new `com.nexus.storage` IPC handlers (`entity_search`, `entity_get`, `entity_relations`, `entity_upsert`, `entity_find_duplicates`) + one new `com.nexus.ai` handler (`entity_recall`).
- ❌ `nexus.entityGraph` shell plugin — **dropped from BL-128 scope after re-evaluation**. Pure UX surface over capabilities already exposed by the CLI + the agent preamble path. The five typed IPC handlers + their TS bindings stay available for any future shell plugin that wants to ship; the BL-128 ticket isn't held open for a UX-only follow-up that has no functional gap to close. If a user signals demand, file a fresh BL.
- ✅ Agent integration: `compose_entity_preamble` splices entity hits into the planner system prompt — recall-first via `entity_recall`, substring-fallback via `entity_search`.
- ✅ CLI: `nexus graph entity list|show|search|related|duplicates` (Duplicates added beyond the original DoD because BL-129 needs it; original DoD's four verbs all present).
