# BL-098: `com.nexus.security` IPC handlers ‚Äî credential vault via IPC

**Status:** closed ‚Äî 2026-05-06

**Source**: Security Integration Assessment (2026-05-06) ‚Äî gap #3 (highest leverage)
**Files**: `crates/nexus-security/src/{ipc.rs (new),core_plugin.rs,lib.rs,Cargo.toml}`, `crates/nexus-bootstrap/src/lib.rs`

`com.nexus.security` previously dispatched zero handlers ‚Äî every credential-using plugin had to link `nexus-security` directly, violating the microkernel boundary. New `ipc.rs` exposes four wire-mirror type pairs (`Get/Set/Delete/ListSecretNames` with TS export). `SecurityCorePlugin` now holds a `CredentialVault` and an in-memory `known_names: HashSet<String>`; dispatch routes handler IDs 1‚Äì4 to vault operations with `"{plugin_id}:{name}"` namespacing so plugins can't read each other's secrets. Disabled-mode policy: `get_secret` returns `{value: null}`, `delete_secret` returns `{ok: false}`, `set_secret` surfaces `KeyringDisabled` as a hard error (caller must know the secret wasn't persisted). `list_secret_names` filters `known_names` by `plugin_id` prefix ‚Äî names from prior sessions are retrievable by exact name but not enumerable (OS keyring limitation). 5 new dispatch tests covering disabled-mode behaviour, missing args, and namespace isolation. Bootstrap switched from `core_manifest()` to `core_manifest_with_ipc()` with the four `(name, id)` pairs. `core_plugin` module made `pub` so bootstrap can reach the `HANDLER_*` constants. Unblocks BL-090 (SSH passphrase caching for `nexus-git`).


## BL-098follow-up: capability grant/revoke emitter (runtime path) ‚úÖ (2026-05-08)

**Source**: BL-052 deferral list ‚Äî "would need to thread the universal topic through the kernel security path; scoped out as a follow-up because the security audit log already lives in SQLite (`nexus-security` audit table) and a non-trivial subset of grants happens before the bus is fully wired"
**Files**: `crates/nexus-plugins/src/loader.rs` (`grant_capability` now mirrors `revoke_capability`'s posture: writes the grant, then audits + publishes a kernel-bus event + emits a universal-activity entry; new helper `publish_capability_activity(bus, kind, plugin_id, capability)` factors out the `ActivityEntry` construction shared between both verbs); `crates/nexus-plugins/tests/capability_revocation.rs` (+3 tests covering the new emit paths)
**Related**: BL-096 (live runtime revocation, which already published `com.nexus.kernel.capability_revoked`), BL-052 (universal activity timeline + `ActivityOrigin::Capability` / `ActivitySurface::Capability` already on the wire)

The BL-052 deferral list named this as scoped-out because parts of the consent flow happen before the event bus is wired ‚Äî install-time HIGH-risk grants flow through `granted_caps.json` writing during plugin loading, ahead of bus availability. This pass ships the *runtime* path: any time a user toggles a capability through Settings, the CLI's `nexus plugin grant` / `revoke` verbs, or the BL-096 live-revocation flow, the change emits a kernel-bus event AND a universal-activity entry. Install-time grants stay in the SQLite audit table only.

- **Symmetry between `grant_capability` and `revoke_capability`.** Pre-pass, `revoke` published `com.nexus.kernel.capability_revoked` (BL-096) but `grant` was emit-silent ‚Äî only the audit log saw it. After this pass, both verbs run the same shape: persist to disk, audit-log via `nexus_kernel::audit::log_capability_*`, publish to the kernel bus, fan to the universal activity topic. A subscriber watching one side now sees the other.
- **`publish_capability_activity` helper.** New free function in `loader.rs` that builds an `ActivityEntry` with `ActivityOrigin::Capability` + `ActivitySurface::Capability` (both wire-format variants already exist from BL-052), session id `cap:<plugin>:<capability>` (so the activity timeline can group entries about the same cap together), and prompt `"<kind> <capability> for <plugin>"`. Publishes to `ACTIVITY_APPENDED_TOPIC` (`com.nexus.activity.appended`) ‚Äî the kernel-owned shared topic that lets cross-plugin emitters fan into a single timeline.
- **Synthetic source-plugin-id `com.nexus.kernel`.** Same pattern BL-095 introduced for `plugin_lifecycle_timeout`: there's no real plugin with that id, but the bus's namespace anti-spoof check is purely string-based, and the `ACTIVITY_APPENDED_TOPIC` is on the kernel-owned shared list so cross-namespace publishing is allowed. Subscribers see `emitting_plugin: "com.nexus.kernel"` and can attribute the entry to the kernel-tier surface.
- **Non-HIGH-risk caps stay silent.** Both `grant_capability` and `revoke_capability` short-circuit when the capability isn't HIGH-risk (auto-granted from the manifest, no consent required). The skip happens before the new emit path, so granting / revoking `FsRead` doesn't pollute the activity timeline.
- **Bootstrap-time grants explicitly stay out.** The deferral note's bootstrap-timing concern is real ‚Äî `granted_caps.json` reads + initial-grant decisions happen before `PluginLoader::set_event_bus` runs. Wiring those into the bus would require restructuring boot order; the SQLite audit table already captures them, so the effort is deferred unless a real gap surfaces.

**Tested**: `cargo test -p nexus-plugins --test capability_revocation` 6/6 (was 3, +3); `cargo test -p nexus-plugins --lib` 137/137 (no regressions); `cargo check --workspace` clean. (One pre-existing test failure ‚Äî `wasm_capability_denial` ‚Äî fails to compile because `wat` is misconfigured in `Cargo.toml`'s `bench.0.wat` key; unrelated to this work, present on `main` before this commit, verified via `git stash` round-trip.) The 3 new tests:
- `revoke_capability_publishes_bus_event_and_activity_entry` ‚Äî registers a plugin, subscribes to the kernel-prefix bus and to `ACTIVITY_APPENDED_TOPIC`, calls `revoke_capability(NetHttp)`, asserts both the kernel-tier `capability_revoked` event and a universal-activity entry with prompt `"revoked net.http for <plugin>"` land on the bus.
- `grant_capability_publishes_bus_event_and_activity_entry` ‚Äî same shape against `grant_capability`; asserts `capability_granted` + `"granted net.http for <plugin>"`.
- `non_high_risk_grant_emits_nothing` ‚Äî sanity guard that granting `FsRead` (non-HIGH-risk) short-circuits before the emit path so the activity timeline doesn't get polluted.

**Definition of done coverage**: ‚úÖ runtime grant/revoke emit to the universal activity topic. The bootstrap-timing piece (install-time consent flow) stays deferred per the original closure note rationale.
