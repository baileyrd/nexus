## Phase 1.1 follow-up â€” `RelayServer::shutdown` (closed 2026-05-16)

The Phase 1.1 relay's `serve_listener` originally `tokio::spawn`'d each accepted peer into a detached task; aborting the accept loop left existing peer connections alive. Phase 1.1 follow-up tracks per-peer tasks in a `tokio::task::JoinSet` and adds a broadcast-channel shutdown signal:

- New `RelayServer::shutdown()` broadcasts `()` to every running task.
- `serve_listener` selects on the shutdown signal vs. `listener.accept()`; on shutdown it calls `JoinSet::shutdown().await`, which aborts every per-peer task *and waits for them to drain* before returning `Ok(())`.
- Caller pattern: keep an `Arc<RelayServer>` clone for `.shutdown()`, await the `serve_listener` JoinHandle to confirm peer tasks exited. Test helpers in `tests/reconnect.rs` wrap this in a `RelayHandle::stop().await`.

This unblocks the previously-deferred buffered-events-replay integration test. New `reconnect_replays_buffered_events_after_relay_outage` boots a relay, connects alice + bob, calls `relay.stop()` (closing both sockets), publishes a presence event on alice's bus during the outage (her supervisor's feeder buffers it), brings a fresh relay up on the same port, and asserts the buffered event arrives on bob's bus once both supervisors reconnect. End-to-end coverage now pins the BL-143 DoD's "Disconnect / reconnect handled gracefully (buffered ops replayed on reconnect)" invariant.

`wait_for_state` was refactored to take a pre-built `EventSubscription` so callers can subscribe *before* the supervisor starts. The broadcast bus drops events with no subscribers, so subscribing after `ReconnectingClient::start` raced the initial `connecting`/`connected` transitions.

All 8 Phase 1.1 relay tests still pass â€” the shutdown plumbing is additive; the previous "drop the listener and let peers exit naturally" path still works for tests that don't explicitly call `shutdown`.

## Phase 1.5 â€” reconnect resilience (closed 2026-05-16)

New `nexus_collab::reconnect_client::ReconnectingClient` is a self-supervising counterpart to the one-shot `CollabClient`. Its supervisor task connects to the relay, runs the bus bridge inside, and on disconnect (or initial-connect failure) sleeps for an exponential-backoff delay (`initial_delay` â†’ Ă—`backoff_factor` â†’ capped at `max_delay`) before reconnecting. Backoff resets to `initial_delay` after any successful session so a long-lived bridge that finally drops doesn't inherit a stale max sleep.

Outbound buffering across reconnects: the supervisor owns a bounded `VecDeque` filled by per-filter feeder tasks. When a session is live the session task pops items and ships them to the relay; when disconnected the deque grows up to `buffer_capacity` (default 256), dropping the oldest entry on overflow. On reconnect the buffer drains naturally â€” items pushed during the outage are the first thing the new WS sees. A `Notify` wakes the drain task whenever a producer pushes, so the consumer never busy-spins.

Connection lifecycle is broadcast on `com.nexus.collab.connection` (new `CONNECTION_STATE_TOPIC` constant) with payloads `{"state": "connecting" | "connected" | "disconnected"}`. The shell can subscribe for a "reconnecting" badge in Phase 2.

Bootstrap + CLI now use `ReconnectingClient` by default:
- `nexus_bootstrap::collab::start_if_enabled` swaps `CollabClient::connect(...) + pending().await` for `ReconnectingClient::start(...)`. New optional `[collab].initial_delay_ms` / `max_delay_ms` / `buffer_capacity` config keys override the defaults.
- `nexus collab join` uses `ReconnectingClient` too â€” the CLI message changes from "connected as alice (N peers)" to "bridge online (auto-reconnect enabled)" since the supervisor reconnects independently of the CLI's lifetime.

Test surface (Phase 1.5 additions): 4 new `reconnect_client::tests` (next_backoff doubling-to-cap, outbound-buffer drop-oldest, outbound-buffer notify-wakes-pop, connection-state wire codes) + 2 new `tests/reconnect.rs` integration tests (initial-handshake state transitions, bad-token retry-with-backoff). The "kill relay + restart" replay test is omitted because the Phase 1.1 server doesn't track per-peer task handles â€” aborting the accept loop leaves existing connections alive. Adding a `JoinSet`/shutdown channel to `RelayServer` is a Phase 1.1 follow-up.

## Phase 1.4 â€” CLI verbs + keyring token storage (closed 2026-05-16)

New `nexus collab` subcommand group in `nexus-cli`:
- `nexus collab serve [--port <p>] [--token <t>] [--save-token]` â€” binds a `RelayServer` on `0.0.0.0:<p>` (default 7700), resolves the shared secret from `--token` / keyring (in that order), and blocks until SIGINT. Prints the listen URL + join hint to stderr.
- `nexus collab join <ws-url> [--token <t>] [--peer-id <id>] [--display-name <n>] [--save-token]` â€” parses `ws://host:port[?token=â€¦]`, builds a full CLI runtime, connects a `CollabClient` against the runtime's `EventBus`, blocks until SIGINT, then `client.shutdown()`s gracefully so the relay sees a clean WS close frame. `peer_id` defaults to `$USER`; `display_name` defaults to title-cased `peer_id`.
- `nexus collab token set <value>` / `nexus collab token clear` â€” manage the `nexus.collab.token` entry via `nexus_security::CredentialVault`. Missing-entry on clear is not an error.

Shared URL helper now lives in `nexus_collab::url::parse` (re-exported as `parse_ws_url`); both the CLI and the existing `nexus-bootstrap` `[collab]` config loader use it. The parser handles `ws://host:port`, `ws://host:port/`, `ws://host:port?token=â€¦`, `ws://host:port/path?token=â€¦` and rejects the wrong scheme / missing port / empty host.

Token resolution precedence: explicit `--token` > `?token=` query string > keyring entry. The CLI emits an actionable error message when none produces a non-empty value.

Test surface: 9 new `url::tests` round-trip cases (with-/without-path, missing port, wrong scheme, unknown query keys, token-only) and 5 new CLI `commands::collab::tests` covering the token precedence and default identity helpers. The 8 relay + 7 client_bridge + 5 presence_bridge integration tests + 5 bootstrap collab tests all stay green â€” 54 collab-related tests in total.

## Phase 1.3 â€” presence protocol (closed 2026-05-16)

New `nexus_collab::presence` module ships the bus-topic constants + payload types:
- `PresenceEvent { user_id, display_name, cursor }` with optional `PresenceCursor { relpath, block_id }` (Phase 2 extends with character-offset / selection-range fields additively).
- `PRESENCE_TOPIC` = `com.nexus.collab.presence` â€” peer-authored cursor / focus updates.
- `PEER_JOINED_TOPIC` / `PEER_LEFT_TOPIC` = `com.nexus.collab.peers.joined` / `â€¦peers.left` â€” synthesised by the bridge from `ServerMessage::PeerJoined` / `PeerLeft` so the shell's peers panel can subscribe without reaching into the wire protocol.
- `COLLAB_TOPIC_PREFIX` = `com.nexus.collab.` so the outbound subscription catches presence + any future collab topic with one filter.

`CollabClient` refactor:
- `CollabClientConfig::outbound_filter` â†’ `outbound_filters: Vec<EventFilter>`. The default subscribes to **two** filters â€” `OPS_TOPIC_PREFIX` (CRDT ops) and `COLLAB_TOPIC_PREFIX` (presence + future collab topics) â€” and spawns one outbound task per filter sharing the WS sink.
- Inbound republishes use `EventBus::publish_core(COLLAB_BRIDGE_PLUGIN_ID, â€¦)` to bypass the kernel's namespace anti-spoof check (the bridge legitimately stamps editor-namespace topics with the collab plugin id).
- New `COLLAB_BRIDGE_PLUGIN_ID = "com.nexus.collab.bridge"` sub-namespace distinguishes bridge-republished events from locally-authored ones; the outbound subscriber filters by `event.emitting_plugin == COLLAB_BRIDGE_PLUGIN_ID` and skips, breaking the broadcastâ†’republishâ†’outbound loop without per-topic special casing. Locally-authored presence under `COLLAB_PLUGIN_ID` flows through normally.
- Broadcast-channel shutdown lets every outbound task observe one `()` and exit; `shutdown` now drains the writer Vec then closes the sink so the relay sees a proper WS close frame.

Test surface (Phase 1.3 additions): 5 new tests in `tests/presence_bridge.rs` covering presence round-trip with cursor, presence without cursor, peer-joined surfacing on existing peers' buses, peer-left on disconnect, and the anti-loop invariant ("3-peer fanout: each receiver gets exactly one presence event"). Plus 6 new unit tests on the `presence` module payload round-trips.

## Phase 1.2 â€” kernel-bus client bridge (closed 2026-05-16)

`nexus_collab::CollabClient` opens one WebSocket connection to the relay, completes the Hello handshake, and runs two background tasks:
- Outbound: subscribes to `EventFilter::CustomPrefix("com.nexus.editor.ops.")` and forwards every event as `ClientMessage::Envelope { topic, payload }`. The subscription filter is configurable via `CollabClientConfig::outbound_filter` so a future presence/settings bridge can spin up parallel clients without changing the wire shape.
- Inbound: receives `ServerMessage::Envelope` frames; republishes the payload on the kernel bus via `EventBus::publish_plugin` under `com.nexus.editor` (matching the existing `CrdtPublisher` namespace convention so the kernel's anti-spoof check accepts it). Site-based self-echo dedup drops envelopes whose `op.id.site` equals the configured `local_site_id`.

Bootstrap wiring lives in `crates/nexus-bootstrap/src/collab.rs`. `start_if_enabled(forge_root, bus)` reads `[collab]` from `<forge>/.forge/config.toml` (fields: `enabled`, `relay_url`, `token`, `peer_id`, `display_name`) and, when an ambient tokio runtime is reachable via `Handle::try_current`, spawns the bridge fire-and-forget. Missing runtime / disabled config / malformed URL / failed connect all log and skip â€” collab is opt-in and never fatal.

Test surface: 15 unit tests on protocol + auth + client helpers, 7 integration tests in `tests/client_bridge.rs` covering handshake echo, initial-peer snapshot, outboundâ†’other-peer-bus delivery, site-based self-echo drop, cross-site pass-through, outbound-filter scope, and clean shutdown; 8 relay-server tests carried forward from Phase 1.1. 9 bootstrap collab-module tests covering URL parsing, config load, and skip paths.

Deferred to follow-up phases: passing the CrdtPublisher's `SiteId` into the bridge (bootstrap currently leaves `local_site_id = None`, relying on the relay's peer-id echo suppression); plumbing the bridge JoinHandle into Runtime for explicit shutdown (1.5 reconnect resilience picks this up).

## Phase 1.1 â€” relay server scaffold (closed 2026-05-16)

New `nexus-collab` crate ships the WebSocket relay server primitive:
- `protocol.rs` â€” tagged `ClientMessage` / `ServerMessage` enums (Hello / Envelope / PeerJoined / PeerLeft / Error). Topic-agnostic so the relay stays stable as op + presence topics evolve.
- `auth.rs` â€” `Token` newtype with constant-time `verify`; empty-token construction rejected.
- `server.rs` â€” `RelayServer::serve_listener` over `TcpListener`; per-peer read/write task pair fanning through `tokio::sync::broadcast`; self-echo suppression by originator `peer_id`; `accept_async_with_config` capped at 16 MiB/frame.
- 8 integration tests on `127.0.0.1:0` covering two-peer handshake, three-peer fanout, peer-left on drop, envelope echo policy, and the bad-token / first-frame-not-Hello / duplicate-peer-id / empty-peer-id rejection paths. 10 unit tests on protocol + auth round-trips.

Remaining sub-tasks (none committed yet, design noted in BACKLOG planning thread): 1.2 client bridge into the kernel event bus; 1.3 presence event shape on `com.nexus.collab.presence`; 1.4 `nexus collab serve` / `join` CLI verbs + nexus-security token persistence; 1.5 reconnect resilience + gossip catch-up via `OpLog::missing_for`. Phase 2 (shell `nexus.collab` plugin: peers panel + remote-cursor annotations) and the 10-test cross-Runtime DoD suite remain open.

BL-074 shipped the CRDT primitives and the in-process ops bus (`com.nexus.editor.ops.<relpath>`). Two sessions on the *same machine* (two Nexus windows opening the same forge) already exchange ops via the kernel event bus. What's missing is a *network transport* so two users on different machines can edit together in real time â€” the "multiplayer" scenario Zed built as a core primitive.

**Phase 1 â€” local-network WebSocket transport (MVP):**
- `nexus collab serve --port 7700` starts a WebSocket relay in `nexus-collab`; relays `com.nexus.editor.ops.*` envelopes between connected peers.
- `nexus collab join ws://host:7700` â€” local instance connects to a relay; the collab plugin bridges the WebSocket into the local kernel event bus.
- Presence: connected peers broadcast a `com.nexus.collab.presence` event with `{ user_id, display_name, cursor: { relpath, block_id } }`; the shell renders remote cursors as coloured overlays in the editor (existing `AnnotationSystem` extended with ephemeral remote-cursor annotations).
- Authentication: shared secret in the URL (`ws://host:7700?token=â€¦`) or `nexus-security` keyring.

**Phase 2 â€” shell UI:**
- Collaboration panel plugin (`nexus.collab`) in the shell: shows connected peers, their active files, cursor positions.
- "Share this forge" button â€” starts a relay and copies the join URL to clipboard.
- Remote cursor annotations in the CM6 editor (coloured caret + display name tooltip).

**Phase 3 (deferred) â€” hosted relay / channels:**
- Zed-style persistent channels with voice (WebRTC) and screen sharing are out of scope for Phase 1/2. Defer until Phase 2 is validated.

**Definition of done (Phase 1 + 2):**
- Two machines on the same LAN can concurrently edit a shared forge with CRDT convergence
- Remote cursors visible in the editor within 200 ms of movement
- Presence panel lists connected peers and their active files
- Disconnect / reconnect handled gracefully (buffered ops replayed on reconnect)
- `nexus collab serve` and `nexus collab join` CLI verbs documented in `nexus help collab`
- 10 integration tests: relay message routing, presence broadcast, cursor annotation render, disconnect-reconnect op replay, auth token rejection
