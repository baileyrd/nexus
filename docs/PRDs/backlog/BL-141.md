# BL-141: Multibuffer / multi-excerpt view

**Status:** Phase 1 closed (2026-05-16); Phases 2 + 3 filed as follow-ups (see below)

## Phase 1 closure notes (2026-05-16)

Read-only multibuffer surface shipped. A caller can request a
synthetic session assembled from excerpts of one or more source
files, render those excerpts as `BlockType::Excerpt` nodes carrying
the captured snapshot text + metadata, and rely on the kernel to
read source files through `com.nexus.storage::read_file` (so
capability checks + path resolution apply).

**Shipped:**

- New `BlockType::Excerpt { source_relpath, line_start, line_end, label }`
  variant in `crates/nexus-editor/src/block.rs`. Markdown serialize
  groups it with the other "no canonical markdown form" variants —
  synthetic sessions never persist, so the serialize path only runs
  for diagnostic `get_markdown` calls.
- New `Session.is_synthetic: bool` field. `apply_transaction` and
  `save` reject synthetic sessions with a clear "read-only
  multibuffer" error referencing BL-141 Phase 2. The on-disk source
  files behind the excerpts can never be silently corrupted through
  the multibuffer surface.
- New `com.nexus.editor::open_excerpts` IPC handler (id 14, async
  dispatch). Accepts `{ items: [{ relpath, line_start, line_end,
  label? }, ...] }`; reads each unique source file once via the
  storage IPC (with a local-fs fallback for context-less unit tests);
  merges adjacent/overlapping ranges within the same file in
  first-appearance order; returns a standard `EditorSnapshot` keyed
  by a `multibuffer://<uuid>` synthetic relpath.
- Bootstrap manifest gained the new command mapping; `cap_matrix.toml`
  classifies it as `unrestricted` (source reads route through
  `com.nexus.storage::read_file`, which already carries fs caps).
- Shell client: `EditorKernelClient.openExcerpts(items)` +
  `ExcerptRequest` interface in
  `shell/src/plugins/nexus/editor/kernelClient.ts`.
- 6 IPC tests in
  `crates/nexus-bootstrap/tests/editor_excerpts_ipc.rs` cover
  construction, multi-file reads, empty-items rejection, overlapping-
  range merge (within + across files), read-only guard on
  `apply_transaction` + `save`, and per-excerpt label preservation.

**Deferred to follow-ups:**

### BL-141 Phase 2 — read-write multibuffer (Medium)

Today edits in an excerpt return an error. Phase 2 translates text
ops at "multibuffer position N" into character-position edits against
the corresponding source file's `Session` and propagates the result
back into the multibuffer's snapshot. Touches the transaction model
(operations need a source-file projection), the session lookup
(multibuffer session needs to acquire + dispatch to source sessions),
and `save` (saves every dirty source). Also adds bidirectional sync
so source-file edits update the multibuffer's snapshots while the
view is open.

### BL-141 Phase 3 — consumer wiring (Small per consumer)

The three consumers listed in the original DoD now have a target
they can call. Each is its own shell-side feature:

- **Diagnostics panel "Open all in multibuffer"** — gated on a
  diagnostics panel existing as a shell plugin (unclear whether
  shipped today; survey before scoping).
- **Find-all-references** — wire a CM6 command that calls
  `nexus-lsp::references` (already implemented) and routes the
  result into `editor.openExcerpts(...)`.
- **Rename refactoring preview** — same shape over
  `nexus-lsp::rename`'s `WorkspaceEdit` response.

### BL-141 Phase 1.5 — CM6 rendering (Small)

The synthetic session opens in the standard `EditorView` today,
which renders each excerpt's snapshot text as a paragraph. Phase
1.5 adds CM6 decorations for the per-excerpt header
(`{source_relpath}#L{line_start}-L{line_end}` + optional label) so
the multibuffer looks like a multibuffer rather than a wall of
text.

## Original scope (preserved for context)

**Source**: Zed capability assessment — see [../research/zed-capability-assessment.md](../research/zed-capability-assessment.md). Filed 2026-05-15.
**Effort**: Medium–Large. Extends the block-tree model; touches editor open flow and LSP wiring.
**Crates**: `nexus-editor` (excerpt block type, synthetic session), `nexus-storage` (multi-file read), shell editor plugin.

Zed's multibuffer is a single editable tab containing *excerpts* (ranges) from multiple different files — useful for project-wide diagnostics (every error, editable in place), find-all-references results, and rename refactoring preview. Nexus's block-tree model is excerpt-friendly: a synthetic document composed of `ExcerptBlock` nodes, each referencing `(relpath, line_start, line_end)`, can be constructed without changing the on-disk model.

**Scope:**

- New `ExcerptBlock` variant in `nexus-editor`'s block types: `{ source_relpath, line_start, line_end, content_snapshot }`.
- New `com.nexus.editor::open_excerpts` IPC handler: accepts `[{ relpath, line_start, line_end, label? }]`, constructs a synthetic read-write buffer, returns a session ID. Edits within an excerpt dispatch through `apply_transaction` on the source file's session; saves to the source file. Read-only mode available for reference excerpts (e.g. definition peek).
- Shell editor plugin: `nexus.editor.openExcerpts(items)` — opens the synthetic buffer in a new tab with a `📄 N files` title.
- Consumers:
  - **Diagnostics panel** ("Open all in multibuffer" button) — replaces the current single-file-at-a-time diagnostics flow.
  - **Find-all-references** LSP result — open references as an excerpt view.
  - **Rename refactoring** preview — show all affected locations before confirming.

**Definition of done:**
- `open_excerpts` returns a synthetic session; reads show correct content from source files
- Edits within an excerpt round-trip through `apply_transaction` on the source file and persist on save
- "Open diagnostics in multibuffer" renders all LSP errors across the project in one tab, each error editable
- Excerpt separators show file path + line range header
- 8 unit tests: construction, multi-file reads, edit round-trip, read-only guard, empty excerpt list, overlapping ranges dedup
