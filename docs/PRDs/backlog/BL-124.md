# BL-124: `useFrameSnapshot` adoption in `EditorView` (Phase 2 of TYPING-LATENCY-PLAN)

**Status:** closed — 2026-05-14

**Source**: Typing-latency analysis — `docs/roadmap/TYPING-LATENCY-PLAN.md`
**Files**: `shell/src/plugins/nexus/editor/EditorView.tsx` (per-relpath selectors via `useFrameSnapshot`, `totalTabs` derived from a narrow selector); `shell/src/stores/useFrameSnapshot.ts` (new optional `rebuildKey` parameter so the controller can re-bind on prop change); `shell/tests/editor-view-frame-snapshot.test.ts` (new — 4 tests pinning the per-relpath narrowing contract)
**Related**: BL-110 (the `useFrameSnapshot` hook this PR wires into the editor — reference site was `FileStats.tsx`); BL-122 (the perf harness baseline this PR doesn't move directly — its win is on the React-render side, not the kernel-side scenarios)

The motivating observation: pre-BL-124 `EditorView` reads `useEditorStore((s) => s.tabs)` and derives the active tab inside a `useMemo`. Every leaf subscribes to the entire `tabs` array; every `setContent` produces a new `tabs` reference; every keystroke in any leaf re-renders every other leaf's React tree. With multiple tabs open this multiplies the React commit work by N.

**What landed.**

- **Per-relpath narrowing in `EditorView`.** Replaces `useEditorStore((s) => s.tabs)` with a two-element `useFrameSnapshot` over `[s.tabs.find(t => t.relpath === relpath) ?? null, s.tabs.length]`. The active-tab selector returns the same object reference whenever this tab's slice didn't change — even when other tabs mutated — because the store's `setContent` does `tabs.map(t => t.relpath === r ? { ...t, content: v } : t)`, leaving non-target tabs by reference. So the FrameSnapshot tuple is `===` stable across keystrokes on other tabs, and the leaf doesn't re-render at all.
- **`useFrameSnapshot` gains an optional `rebuildKey` parameter.** Pre-PR the hook captured `entries` on first render only (the docstring promised "Pair with `useMemo` if a selector needs to depend on props" but the implementation's `useMemo([])` ignored entries changes). `EditorView`'s selectors close over the `relpath` prop, and a leaf can be re-targeted to a different file without unmounting, so we need a way to re-bind. The new parameter passes through to the inner `useMemo`'s deps; default (`undefined`) preserves BL-110 reference behaviour.
- **`activeRelpath` and `isDirty` excluded from the snapshot.** The BL DoD listed them as suggested entries, but `EditorView` itself never renders either — `activeRelpath` is only read inside event handlers via `getState()`, and the dirty dot flows through `WorkspaceRenderer.useLeafDirty` per-leaf subscription. Including them in EditorView's snapshot would force a re-render on every session-revision bump (i.e. every keystroke) for no visible change. The closure note documents the deviation; `WorkspaceRenderer.useLeafDirty` is unchanged.
- **Setter actions stay as direct `getState()` reads.** `setMode` is action-only — its identity never changes, so a direct `useEditorStore.getState().setMode` keeps the FrameSnapshot scoped to the value selectors that actually need rAF coalescing.
- **Children (`TabBody`, `ViewHeader`, `BreadcrumbSegments`, `ModeToggle`) keep their prop-driven shape.** They receive the narrow slice from EditorView via props; when this leaf's selectors don't fire (other tab changed), EditorView doesn't re-render, so the children don't either. No per-child hook needed — the narrowing happens at the EditorView boundary.

**Tests.** `shell/tests/editor-view-frame-snapshot.test.ts` (new) verifies the contract at the FrameSnapshot layer rather than rendering EditorView end-to-end through React (which would need to mount CodeMirror, the runtime, the storage IPC stub, etc. — too heavy for a unit test):

1. **Render-count narrowing (DoD bullet 3).** Two `FrameSnapshot` controllers wired to tab A and tab B with the EditorView selector shape. 10 `setContent` calls against tab A produce exactly 10 flushes on A's controller and **zero** on B's. The tuple `[tab-object, tabs.length]` is identity-stable for B across A's mutations because (a) B's tab object is preserved by `tabs.map`'s identity-passthrough on non-target entries and (b) `tabs.length` is unchanged.
2. **Identity preservation, `setContent` (DoD bullet 4).** `setContent('notes/a.md', ...)` doesn't replace tab B's object — pinned with `===`. The store already did this; the test stops a future refactor from quietly breaking the narrowing win.
3. **Identity preservation, `setMode`.** Same `===` check for `setMode('notes/a.md', 'source')`. Modes are mutated through the same `tabs.map` pattern.
4. **`rebuildKey` re-bind.** Pins the BL-124 hook enhancement: a fresh `FrameSnapshot` against a different `relpath` reads the new tab's slice. The React layer (`useMemo([rebuildKey])`) handles the controller swap; the test pins the underlying factory behaviour.
