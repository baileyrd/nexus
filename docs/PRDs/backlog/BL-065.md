# BL-065: Windows pre-command support (cmd.exe / PowerShell)

**Status:** closed — 2026-05-07

**Source**: Terminal Integration Assessment (2026-05-06) — Phase Q follow-up
**Files**: `crates/nexus-terminal/src/{precmd.rs, lib.rs}`
**Related**: PRD-09 §4.4; Phase Q (POSIX-only sentinel approach shipped)

The pre-command pipeline used a POSIX-only sentinel (`printf '<SENTINEL> %d\n' $?`) that has no working equivalent in cmd.exe or PowerShell. BL-065 makes the wrapper shell-family aware so a saved command running on a Windows host can chain pre-commands like a saved command on Linux.

- **`ShellFamily` enum.** Three variants today: `Posix`, `Cmd`, `PowerShell`. `Default::default()` is `Posix` so existing callers get the historical behaviour without code changes.
- **`detect_from_path(&str)`.** Picks the family by basename:
  - Splits on either `/` or `\` so a Windows path like `C:\Program Files\PowerShell\7\pwsh.exe` parses correctly even on a Linux dev box. `Path::new` would treat `\` as a literal character on POSIX and miss the basename.
  - Strips a trailing `.exe` (case-insensitive — Windows paths often arrive uppercase).
  - Lowercase-matches against `cmd` / `pwsh` / `powershell`; everything else falls through to `Posix` (the safe default for unknown shells).
- **Per-family `wrap_step(command, sentinel)`.** Builds the byte-payload sent into the PTY:
  - `Posix`: `{cmd}\nprintf '{sentinel} %d\n' $?\n` — unchanged from Phase Q.
  - `Cmd`: `{cmd}\r\necho {sentinel} %ERRORLEVEL%\r\n` — `%ERRORLEVEL%` expands inline against the previous command's exit code; CRLF matches what cmd.exe expects from a typed line.
  - `PowerShell`: `{cmd}\r\nWrite-Host "{sentinel} $LASTEXITCODE"\r\n` — `$LASTEXITCODE` is the equivalent of `$?` for native commands; `Write-Host` writes to stdout the way `echo` does on POSIX (vs. `Write-Output` which feeds the pipeline). PowerShell's `$?` is a boolean, not a code, so `$LASTEXITCODE` is the right expansion for surfacing native-tool failures.
- **Same parser, same waiter.** All three families emit lines of the form `<sentinel> <integer>`, so `parse_sentinel_exit_code` and `wait_for_sentinel` don't need to fork. The "skip the input echo, take the actual output line" logic that existed for POSIX continues to work — the echo line for cmd / pwsh contains the literal token (`%ERRORLEVEL%` / `$LASTEXITCODE`) which doesn't parse as an integer, so the waiter naturally skips it.
- **`PreCommandOptions.shell_family`.** New field with default `Posix`; callers that drive a Windows session set it explicitly (or via `ShellFamily::detect_from_path(saved.shell)`). The procmgr layer wiring is left for a follow-up — this commit is the precmd-layer infrastructure.

**Tests.**

- 9 new unit tests in `nexus-terminal::precmd::tests`:
  - `shell_family_detect_recognises_cmd_with_and_without_extension` — full Windows path, bare `cmd`, bare `cmd.exe`, uppercase `CMD.EXE`.
  - `shell_family_detect_recognises_powershell_and_pwsh` — pwsh on Linux (`/usr/bin/pwsh`), bare `pwsh`, `pwsh.exe`, full `C:\Program Files\PowerShell\7\pwsh.exe`, legacy `powershell.exe`.
  - `shell_family_detect_falls_through_to_posix_for_known_unix_shells` — bash / zsh / dash / fish / sh / ksh.
  - `shell_family_detect_unknown_shell_defaults_to_posix` — empty string + `/opt/weird/myshell` both Posix.
  - `wrap_step_posix_uses_printf_with_dollar_q` — POSIX wrapper shape pinned (`printf '__sentinel %d\n' $?`, trailing `\n`).
  - `wrap_step_cmd_uses_echo_errorlevel_and_crlf` — cmd shape pinned, asserts no `$?` leak.
  - `wrap_step_powershell_uses_write_host_with_lastexitcode` — pwsh shape pinned, asserts no `%ERRORLEVEL%` leak.
  - `wrap_step_output_shape_round_trips_through_parse_sentinel` — proves all three families produce a line the existing parser accepts (the structural invariant that lets the waiter stay family-agnostic).
  - `pre_command_options_default_is_posix_family` — back-compat invariant: existing callers that don't set `shell_family` get `Posix`.
- All 6 pre-existing precmd tests still pass; full nexus-terminal suite passes (314 tests). Clippy clean on changed code.

**Deliberately deferred:**

- **Live cmd.exe / pwsh execution tests.** The repo's primary CI runner is WSL2 / Linux; cmd.exe and pwsh aren't available there. Pinning the wrappers via string-shape unit tests proves the load-bearing behaviour (the line each shell would emit is parseable). When a Windows runner lands, the `unix_only` gate becomes a `windows_only` gate on three new live-execution tests with the same shape as the existing bash ones.
- **State-inheritance verification on Windows.** Same blocker. The DoD asks for "PATH changes, env exports, directory changes carry forward" tests; those need a real cmd.exe or pwsh under test. The wrapper architecture preserves state-inheritance by construction (each step is a separate line in the same shell instance, just like POSIX), so the structural guarantee holds — the test would just be confirming what the design implies.
- **Procmgr-layer plumbing.** `ManagedConfig` doesn't currently carry shell family information through to `run_pre_commands`. Today, callers compose the option themselves: `PreCommandOptions { shell_family: ShellFamily::detect_from_path(&saved.shell), ..Default::default() }`. A future change can either add a derived `family()` method on `ManagedConfig` or have the procmgr default the family from the saved command's shell field; that's a wiring decision, not a precmd-layer concern.
