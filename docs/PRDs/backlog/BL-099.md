# BL-099: Plugin manifest signing verification

**Status:** closed — 2026-05-06

**Source**: Security Integration Assessment (2026-05-06) — gap #1; PRD-02 §5
**Files**: `crates/nexus-plugins/src/{signing.rs (new),manifest.rs,loader.rs,lib.rs,Cargo.toml}`, `crates/nexus-plugins/tests/manifest_signing.rs (new)`, `crates/nexus-kernel/src/config.rs`, `crates/nexus-bootstrap/src/lib.rs`, `crates/nexus-cli/src/{main.rs,commands/plugin.rs}`, workspace `Cargo.toml`

`nexus-plugins/src/signing.rs` is a new module with `PluginSignatureVerifier`, `KeyringFile`, `RevocationList`, `TrustedKey`, and the `canonicalize_manifest_for_signing` helper. Verification is `ed25519_dalek::VerifyingKey::verify_strict` over the canonical manifest bytes (manifest with the `[signature]` block stripped from the line that opens it through the next bare-`[` section header or EOF). The verifier loads its trusted keyring from `~/.nexus/keys/community.json` and a CRL from `~/.nexus/keys/revoked.json`; missing files are treated as empty (a manifest referencing a missing key fails with the deterministic `KeyNotTrusted` rather than an i/o error).

`PluginManifest` gained an optional `signature: Option<PluginSignature>` field; the TOML shadow type accepts a `[signature]` block with `algorithm`, `signer_key_id`, `signature` keys. `PluginLoader::load` runs `verify_manifest_signature` between api-version and dependency checks. The rule is: a signature, if present, is *always* verified (so a malformed signature is a loud failure even when the gate is off); when `KernelConfig::require_signatures = true`, an unsigned community plugin returns `ManifestInvalid` with the BL-099 prefix.

User-facing surface: `nexus plugin verify <path> [--keys-dir <dir>]` parses the manifest at `<path>/manifest.toml`, runs the verifier, and prints `OK — <id> signed by '<key_id>' (ed25519)` on success. Bootstrap plumbs `kernel.config().require_signatures` through `PluginLoader::set_require_signatures`.

**Module placement.** The PRD specified `nexus-security/src/signing.rs`; the module shipped in `nexus-plugins` instead because nexus-security already depends on nexus-plugins (via `CorePlugin`), and pulling the verifier out of the loading path would have required an injectable trait callback to break the cycle. Putting the verifier in nexus-plugins also keeps the "manifest parser" and "manifest signature checker" in the same crate, which is where future signature-format changes will land first.

**Default-off shipping posture.** `KernelConfig::require_signatures = false` per the PRD's backward-compat clause. Flip to `true` once a signed-plugin distribution channel exists (still gated on the marketplace work, WI-44).

Twelve tests cover the verifier (round-trip success, tampered manifest, untrusted/revoked/unsupported-algorithm/empty-keyring failure paths) plus four end-to-end loader tests in `tests/manifest_signing.rs` (signed-trusted accepts, signed-untrusted rejects, unsigned-with-gate-on rejects, signature-of-canonical-bytes round-trip).

**Deferred from the original DoD:** none — the full DoD shipped, just in a different module layout than the PRD's text suggested.
