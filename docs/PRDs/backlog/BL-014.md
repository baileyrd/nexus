Subcommands: `nexus db import-csv <FILE> --fields <a,b,c> [--has-header] [--column-mapping <JSON|@file>]` reads the CSV file, builds a `csv_import` (handler id 1) request, and prints `Imported N record(s).` plus a skipped count when non-zero; `nexus db export-csv [--records <JSON|@file>] [--fields <a,b,c>] [--output <FILE>]` builds a `csv_export` (id 2) request, decodes the `Vec<u8>` byte array out of the response, and writes raw CSV bytes to `--output` or stdout; `nexus db eval-formula <EXPR> --fields <JSON|@file>` builds a `formula_eval` (id 3) request and pretty-prints the response JSON via `serde_json::to_string_pretty`. The inline-or-`@file` convention (`@<path>` reads from disk; otherwise the literal string passes through) lets large argument bodies live on disk without needing shell quoting. Pure-function arg builders (`build_csv_import_args` / `build_csv_export_args` / `build_eval_formula_args`) are split out from the handlers so they're testable without a runtime. `apply_view` (handler id 4) is intentionally not surfaced â€” its arg shape requires a base file + schema context and is exercised by the shell renderer rather than being a useful CLI today. `list` / `show <base>` are deferred pending a `com.nexus.database::list_bases` handler that does not yet exist; both would be net-new database-engine work and were ruled out of scope for this BL. The lower-level IPC-only twin of `nexus bases`, which operates over the filesystem layer (`.bases` directories); both stay.
