# BL-073: Block auto-stamping on first reference

**Status:** closed — 2026-05-06

**Source**: Editor Integration Assessment (2026-05-06) — gap #4
**Files**: `crates/nexus-editor/src/core_plugin.rs`
**Related**: ADR 0017 (stable block IDs); `stamp_block` handler (id 11)

Stable block IDs no longer require an explicit `stamp_block` round-trip before a wikilink can target a block. Two new triggers stamp on first reference:

1. **`resolve_block_link` (session path).** When the resolved block has `stable_id = None`, the handler rekeys the positional id to a fresh v4 UUID, increments the session revision, and (in the async path) publishes a `com.nexus.editor.changed.<relpath>` event so subscribers reload. The lookup response carries the *new* stable id in both `block.id` and `block.stable_id`. The filesystem-fallback path (closed session, no kernel context) deliberately does **not** auto-stamp — silently rewriting an on-disk file from a read-shaped IPC call would be a surprise; the user keeps the session open if they want the stamp persisted on the next save.

2. **`apply_transaction` (inbound-link trigger).** After the transaction's ops apply, a post-pass walks every `UpdateAnnotations` / `UpdateBlockContent` op for newly-introduced `BlockRef` and `Wikilink` annotations and stamps the targets they point at. `BlockRef` carries the target id directly; `Wikilink` recovers the `#^<uuid>` fragment from the host block's freshly-mutated content (the parser strips fragments from the annotation payload, so byte-slicing the source text is the only path). Stamping uses `BlockTree::rekey` and is best-effort — a missing target or rekey collision (impossibly rare for v4) is silent so a metadata-only side effect can't invalidate a transaction the user already committed.

The new `extract_wikilink_block_uuid` helper handles `[[path#^uuid]]`, `[[path#^uuid|display]]`, heading-only fragments (no stamp), path-only links (no stamp), and non-uuid fragments (no stamp). All four shapes are covered by unit tests, plus `apply_transaction_auto_stamps_block_ref_target` and `apply_transaction_auto_stamps_wikilink_fragment_target` exercising the post-apply scan end-to-end.

**Deferred from the original DoD:**

- *"writes the comment to the file"* — kept as the existing pattern: stamp lives in the in-memory session, the next `save` writes `<!-- ^<uuid> -->` to disk through `com.nexus.storage`. No call path silently mutates the file.
