# BL-027: Multi-agent orchestration / delegation

**Status:** closed — 2026-04-29

**Delivered:** PRD-15 §10's `AgentOrchestrator::delegate / parallel / pipeline` lands as a new module on `nexus-agent` with four IPC handlers wired onto `com.nexus.agent`. Pre-BL-027 a `LlmAgent` planned + executed solo; the orchestrator now hands subtasks between archetypes (Researcher → Writer → Coder) with shared scratch state and an append-only trace log. **Library** ([`crates/nexus-agent/src/orchestrator.rs`](../../crates/nexus-agent/src/orchestrator.rs)): generic `AgentOrchestrator<D: ChatDriver + Clone + 'static, P: ToolDispatcher + Clone>` holding the driver factory, dispatcher, an `Arc<RwLock<HashMap<String, serde_json::Value>>>` scratch map, and an `Arc<Mutex<Vec<TraceEntry>>>` trace log. `delegate(archetype, goal, skills)` builds an archetype `LlmAgent` via `build_archetype`, plans via `Agent::plan`, executes via `PlanExecutor::run`, and appends a `TraceEntry { stage_idx, archetype, goal, started_at_ms, ended_at_ms, status, summary }` whether the stage succeeded, failed at the planner, or failed mid-execution. `parallel(jobs)` fans out via `futures::future::join_all` and returns observations in input order; failed jobs surface as a synthetic failed `Observation` so the positional vector stays aligned and the trace carries the per-stage status. `pipeline(stages)` runs sequentially, stashing each observation's textual summary under `"prev"` and substituting `{{prev}}` into the next stage's `goal_template` before planning; first failure halts and returns the partial vector (not `Err`) so the caller still sees pre-failure work. **IPC** ([`crates/nexus-agent/src/core_plugin.rs`](../../crates/nexus-agent/src/core_plugin.rs)): four new append-only handler ids — `HANDLER_DELEGATE: u32 = 9`, `HANDLER_PARALLEL: u32 = 10`, `HANDLER_PIPELINE: u32 = 11`, `HANDLER_TRACE_GET: u32 = 12`. DTOs: `DelegateArgs { archetype, goal }`, `ParallelArgs { jobs: Vec<ParallelJob> }`, `PipelineArgs { stages: Vec<PipelineStage { archetype, goal_template }> }`, response wrapper `TraceResponse { entries: Vec<TraceEntry> }`. The plugin caches one orchestrator behind a `std::sync::Mutex<Option<Arc<…>>>` so trace + scratch state survive across handler calls — the cache is wired the first time any of the four handlers fire and reuses the same `AiChatBridge` + `KernelToolBridge` adapters the existing `plan` / `run` handlers use (both bridges are now `#[derive(Clone)]` to satisfy the orchestrator generics). The kernel-context wiring lives in `dispatch_async` so a non-orchestrator handler still sees the same eager-context error as before. **Coverage**: `cargo test -p nexus-agent` 25 passing (was 21; +4 net). Three new in-module unit tests in `orchestrator.rs` (`delegate_runs_and_appends_trace`, `scratch_state_round_trips`, `trace_ordering_reflects_append_order`) plus one new e2e test in [`crates/nexus-agent/tests/orchestrator_e2e.rs`](../../crates/nexus-agent/tests/orchestrator_e2e.rs) (`pipeline_substitutes_prev_and_returns_two_observations`) wiring a branching `ChatDriver` that swaps replies based on the system prompt — researcher returns a one-step plan with description `"find data X"`, writer returns `"summary of: {prev}"`, and the test asserts (a) two observations come back, (b) both succeed, (c) the second `TraceEntry.goal` no longer contains `{{prev}}` and grew beyond the template prefix. `cargo clippy -p nexus-agent --all-targets -- -D warnings` clean; `bash scripts/check_ipc_drift.sh` clean (no schema drift — these handlers don't expose ts-rs types). **Out of scope**: skill layering on `delegate` (the `_skills` arg is reserved; per the BL-027 task allowance the orchestrator forwards `None` to `build_archetype` today and callers thread skill-resolved prompts through the goal text); replay hooks (PRD-15 §12 — deferred until orchestration UX settles); reactive rules (PRD §6.3 — same).
