**Delivered:** AI-MEMORY-LAYER-PLAN piece 2 lands as a two-phase pipeline split across the AI core plugin (Rust) and a default-off shell plugin (TypeScript). Phase 1 (`enrich_file`) reads the markdown file via storage IPC, asks the chat provider for tags + summary in two sequential `chat()` calls with tight system prompts, runs `rag::semantic_search` against the body to surface `[[basename]]`-style related notes (with the input file itself filtered out and results deduped), and returns an `EnrichmentProposal` WITHOUT touching the file. Phase 2 (`enrich_apply`) re-reads the file, verifies a body-hash drift guard, merges the proposal into the YAML frontmatter, and writes back via storage IPC. The shell plugin sits between them: it subscribes to a new `files:saved` event, throttles per-file with a 5-second debounce so rapid consecutive saves coalesce, dispatches `enrich_file`, surfaces the proposal in an inline accept-gate panel, and routes the user's Accept click through `enrich_apply`. **Backend** ([`crates/nexus-ai/src/enrichment.rs`](../../crates/nexus-ai/src/enrichment.rs), 470 lines): `EnrichmentProposal { path, body_hash, tags: Vec<String>, summary: String, related: Vec<String> }` is the cross-IPC payload (serde-derived; not ts-rs-exported in v1 — the shell side mirrors the shape with a hand-written interface). `body_hash(content)` strips leading frontmatter then SHA-256s the body, so re-applying a proposal whose frontmatter we just updated does NOT invalidate later proposals on the same body — the idempotency invariant the PRD called out. `strip_frontmatter` is intentionally minimal (recognises a `---\n…\n---\n` block on the very first line; treats unterminated blocks as body so we never accidentally swallow real content). `merge_frontmatter(original, proposal)` parses the existing `tags:` field (inline list `[a, b]` OR block list `\n  - a\n  - b`), unions with the proposal's tags case-insensitively (existing first, then new), drops the existing `tags` / `summary` / `related` lines from the frontmatter, then re-emits them as inline lists; every other key the user authored is preserved verbatim, and a `summary` containing `:` / `#` / `"` is YAML-quoted. `propose(ctx, ai, embedder, path)` runs the pipeline; tags are filtered to single-word lowercase entries (no whitespace, only alphanumerics + `-` / `_`), capped at 5; summaries are stripped of surrounding quotes/whitespace then char-truncated to 200; the related list is deduped, capped at 5, and asks for `MAX_RELATED + 5` chunks from `semantic_search` so the self-filter has headroom. `apply(ctx, proposal)` returns `(applied: bool, reason: Option<String>)` — a hash mismatch is a soft `applied = false` ("body changed since proposal") rather than an error, so the shell can re-propose without surfacing a stack trace. **Handlers** ([`crates/nexus-ai/src/core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs)): `HANDLER_ENRICH_FILE = 15` (args `{ path: String }` → JSON-serialised `EnrichmentProposal`) and `HANDLER_ENRICH_APPLY = 16` (args `{ proposal: EnrichmentProposal }` → `{ applied: bool, reason: ?String }`) wired into the `dispatch_async` match between the existing `HANDLER_SEMANTIC_SEARCH = 13` and `HANDLER_INDEX_STATUS = 14` arms; both require the AI plugin context (built in `wire_context`) and `enrich_file` additionally requires both an AI chat provider AND an embedding provider (matches the dual-prereq the BACKLOG entry called out). **Manifest** ([`crates/nexus-bootstrap/src/lib.rs`](../../crates/nexus-bootstrap/src/lib.rs) `com.nexus.ai` block): `enrich_file` and `enrich_apply` registered alongside the existing 13 handlers; CLI / TUI / MCP / shell all reach them through the canonical `context.ipc_call(...)` path with no per-frontend opt-in. **Shell plugin** ([`shell/src/plugins/nexus/enrich/`](../../shell/src/plugins/nexus/enrich/), 5 files): `enrichStore.ts` (zustand store with `pending: EnrichmentProposal | null`, `applying: boolean`, `error: string | null` + setters; v1 shows one proposal at a time per-file — the runtime overwrites the slot whenever a fresher proposal arrives so the user always sees the latest suggestion for the most recently saved file). `enrichRuntime.ts` is the core: `attachRuntime(api)` subscribes to `files:saved` (a new event), filters to markdown extensions (`.md` / `.mdx` / `.markdown`), and uses a module-scope `Map<filePath, timeoutHandle>` to enforce the 5-second per-file debounce — repeated saves of the same file within the window collapse into one `enrich_file` request, while saves of *other* files still fire on their own clocks. `applyPending(api)` issues `enrich_apply`, dismisses the panel + shows an info toast on success, and surfaces `result.reason` inline on a soft refusal. Failures from `enrich_file` are stashed as a dismissable error rather than thrown — enrichment is opt-in background luxury and must never interfere with the save itself. `EnrichAcceptGate.tsx` is a fixed bottom-right toast-style panel (centred 360px wide, z-index 9999) that renders the file basename, the suggested summary, the tags as `#tag` chips, and the related-notes line; two buttons (`Accept` / `Dismiss`) with the `Accept` button greyed out while `applying` is in flight. `enrichApi.ts` is the module-scope handle pattern (mirrors `recallApi.ts`) so the slot-rendered component can reach the API from a click handler. `index.ts` declares the plugin manifest (`activationEvents: ['onStartup']`, default-off via `DEFAULT_OFF_PLUGINS`), wires everything together in `activate()`, and registers the gate component in the chrome `overlay` slot at priority 27 (one above BL-044's recall overlay so a stacked open keeps the accept-gate visible). **Editor save event** ([`shell/src/plugins/nexus/editor/index.ts`](../../shell/src/plugins/nexus/editor/index.ts)): the COMMAND_SAVE handler now `api.events.emit('files:saved', { relpath })` after each `markSaved` call — three sites (kernel-session save, untitled-markdown write+acquire, non-markdown plain write) so opt-in features can react without coupling to the editor internals. **Catalog**: registered in `DEFAULT_OFF_PLUGINS` next to `recallPlugin` with comment rationale (default-off because it issues an AI chat call + an embedding+vector lookup per saved markdown file; requires both a chat provider AND an embedding provider configured); default-off count comment bumped from 15 → 16. **Inbox-scope decision** (the task's open question): v1 does NOT scope by inbox tag. The BACKLOG entry called out "On save of an inbox-tagged file, run an enrichment pass" but the existing `AiConfig` has no `memory.inboxPath` analogue on the Rust side and the BL-044 inbox-scope filter lives entirely on the shell side; rather than re-discover that wire, v1 enriches every saved markdown file and lets the user dismiss anything that looks irrelevant. The 5-second debounce keeps the cost manageable; a follow-up can subscribe to BL-043's `memory.inboxPath` config in the runtime and skip `enrich_file` for non-inbox files. **Coverage**: `cargo test -p nexus-ai` 142 passing (was 129 before BL-045; +13): 9 unit tests in `enrichment::tests` covering body-hash idempotency under frontmatter mutations, body-hash sensitivity to body changes, `strip_frontmatter` on no-fm / unterminated / well-formed inputs, `merge_frontmatter` creating fresh frontmatter on a file with none, preserving unrelated keys + dedupe-merging tags, replacing block-list tags with an inline list, and quoting summaries containing `:`; 4 helper tests covering `parse_tag_response` (clean tags + 5-tag cap with multi-word filter), `clean_summary` (quote strip + char truncation), and `related_from_matches` (self-drop + dedupe across multiple chunks of the same file). 4 new dispatch-level tests in `core_plugin::semantic_search_dispatch_tests` covering handler-id pinning (`HANDLER_ENRICH_FILE = 15`, `HANDLER_ENRICH_APPLY = 16` — bumping these is a wire break), `enrich_file` arg validation (missing `path`), missing AI provider, and `enrich_apply` arg validation (missing `proposal`). `cargo clippy -p nexus-ai --all-targets -- -D warnings` clean. `cargo test -p nexus-bootstrap` clean (handlers correctly resolve through the manifest). `pnpm --filter nexus-shell typecheck` + `lint` clean (no new warnings beyond the pre-existing pile); `pnpm --filter nexus-shell test` 521 passing + 1 skipped (no shell-side tests added in v1 — the runtime's debounce + IPC dispatch are easier to validate end-to-end than to mock Map-backed timers + zustand at the unit level; a follow-up can add `enrichRuntime.test.ts` mirroring `recallRuntime.test.ts`'s mocked-PluginAPI shape). No `EnrichmentProposal` ts-rs derive in v1 (the shell hand-mirrors the shape) → `scripts/check_ipc_drift.sh` not required. **Known limitations**: (1) inbox-scope is unwired (see "Inbox-scope decision" above); (2) no shell-side unit tests for the runtime debounce / IPC fan-out — the parity tests with BL-044's recall runtime are deferred to a follow-up; (3) the accept-gate UI is a single-pending slot — if the user saves three files in rapid succession, only the most-recent proposal is visible (older ones are silently overwritten); a queue or per-file panel stack would be a polish pass; (4) `enrich_file` issues two sequential `chat()` calls (tags then summary); a single combined prompt with structured-output parsing would halve provider latency but couples the two prompts and complicates retry logic — v1 prefers reliability over latency; (5) the runtime fires on every saved markdown file regardless of size — a 100KB note still costs one chat round-trip + one embedding + one vector lookup, throttled to once per 5s but not skipped; a body-length floor is a trivial follow-up.
