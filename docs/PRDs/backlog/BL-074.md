# BL-074: Collaborative editing — CRDT layer

**Status:** closed — 2026-05-14

**Source**: Editor Integration Assessment (2026-05-06) — gap #5; ADR 0026 documents the phase plan
**Crates**: `nexus-crdt` (new); `nexus-editor` (`OpObserver` hook); `nexus-bootstrap` (`CrdtPublisher` orchestrator + `start_pull_landing_subscriber`); `nexus-cli` (`nexus crdt merge-driver` + `install-merge-driver` shim); shell `nexus.crdtConflict` plugin
**Related**: BL-007 (CRDT-over-Git transport — fully subsumed by this BL per its closure note); PRD-08 collaborative-editing spec; stable block IDs (ADR 0017)

Every DoD bullet shipped between 2026-05-08 and 2026-05-09; this umbrella entry consolidates them after the 2026-05-14 backlog audit found BL-074 still in `BACKLOG.md` despite all phases + follow-ups being closed.

**Shipped scope:**

- **Phase 1 — CRDT foundation** (2026-05-08). New `nexus-crdt` crate with `SiteId` / `Lamport` / `OpId` / `VersionVector`, `OpLog` (idempotent by `OpId`, with gossip slicing via `missing_for`), and `CrdtDoc` (`apply_local` / `apply_remote` over `nexus_editor::Operation` with vector-clock causality). `text::RgaText` — full RGA sequence CRDT for in-block text, tested standalone. Conflict surface: `ConcurrentBlockEdit` and `StructuralDeleteEdit`.
- **Phase 2 — silent text merge** (2026-05-08). `CrdtDoc` eagerly maintains a per-block `RgaText` mirror materialised at construction from baseline content using deterministic synthetic `OpId`s (`merge::baseline_op_id`, `merge::subop_id`); both peers materialise identical RGAs from equal `BlockTree`s, so concurrent ops gossiped between them converge. `CrdtOp` gained `rga_ops: Vec<RgaTextOp>` carrying the position-free RGA translation authored at `apply_local` time. Concurrent text ops replay `rga_ops` on the local RGA and rebuild `block.content` from `rga.render()` — `Conflict::ConcurrentBlockEdit` now only surfaces for whole-block replacements (`UpdateBlockContent` / `UpdateAnnotations`) the RGA can't merge.
- **Phase 3 — sync infrastructure** (2026-05-08). `wire` module with topic `com.nexus.editor.ops.<relpath>` and `OpEnvelope { op: CrdtOp }` JSON payload. `sync` module with `DocHandle` (`Arc<Mutex<CrdtDoc>>`) and `SyncLoop` that owns a kernel `EventSubscription` and drains it into `CrdtDoc::apply_remote`; self-echo ops dropped.
- **Phase 4 — persistence + git merge primitive** (2026-05-08). `state::PersistedCrdt` schema-versioned envelope around `CrdtState` (site, lamport, log, per-block meta, RGA — no tree). Path layout: `<forge>/.forge/.editor/crdt/<sha-of-relpath>.json`. `content_hash_hex` integrity tag. `CrdtDoc::state()` / `from_state(tree, state)` snapshot/restore pair; `from_state` tolerates compatible markdown drift. `OpLog::merge` idempotent-union primitive — what the BL-007 git merge driver registers as the conflict resolver for the state file. 43 unit tests across the crate.
- **Editor wiring** (2026-05-08). `nexus-editor::OpObserver` callback trait — `EditorCorePlugin` invokes the hooks from `finish_open*` / `handle_sync_content` / `handle_close*` / `handle_apply_transaction`. `nexus-bootstrap::crdt_publisher::CrdtPublisher` maintains `HashMap<relpath, CrdtDoc>` + `SiteId`, calls `crdt.apply_local` for each tx op, publishes each `CrdtOp` on `wire::ops_topic(relpath)` via the shared `EventBus`. Open/close flow loads `state::PersistedCrdt` if present and `content_hash` matches; close flow atomic-writes `CrdtDoc::state()` via tmp+rename. Wired into `build_*_runtime` so all invokers (CLI, TUI, MCP, Tauri shell) get publishing + persistence by default. Periodic checkpoints + undo/redo propagation included.
- **BL-007 git merge driver shim** (2026-05-08). `nexus crdt merge-driver` + `nexus crdt install-merge-driver` CLI verbs register the driver in `.git/config` and `.gitattributes` so a `git pull` that touches `.forge/.editor/crdt/*.json` runs `OpLog::merge` as the union resolver. Idempotent install.
- **BL-007 pull-landing transport loop** (2026-05-08). `CrdtPublisher::reload_after_external_change(relpath)` re-reads the merged state file, applies absorbed remote ops via `apply_remote`, publishes envelopes on the ops topic, and surfaces structural conflicts. `start_pull_landing_subscriber` thread subscribes to `com.nexus.git.commit`, drains every event into per-relpath reloads; thread holds a `Weak<Inner>` and self-exits when the publisher drops. Closes the BL-007 transport loop end-to-end for the in-process editor.
- **Op-log compaction wiring** (2026-05-09). `CrdtDoc::compact_to(stable_vv)` wraps the Phase 4 `OpLog::prune_dominated` primitive; `CrdtPublisher` snapshots the doc's VV at session-open as the conservative "stable VV" oracle (anything dominated by it was on disk before the session started, so the prune floor still reports those ids as seen for any peer that loads the persisted state). On `on_session_closed` the publisher prunes against that floor before writing. Single-replica forges collapse old ops aggressively across reopens; multi-peer forges keep ops authored or absorbed during the session because they exceed the open-time VV — the deliberate trade-off documented in the BL.
- **State-file git-tracking policy** (2026-05-09). `Forge::init` writes a default `.forge/.gitignore` that excludes rebuildable / per-machine state and leaves `.forge/.editor/crdt/*.json` tracked so the merge driver sees it. `nexus crdt enable-transport` runs the same gitignore step and `install-merge-driver --apply` for forges created before this change. Both steps idempotent.
- **Conflict UI — toast + per-block resolver modal** (2026-05-09). New `nexus.crdtConflict` plugin under `shell/src/plugins/nexus/crdtConflict/` subscribes to the `com.nexus.editor.crdt.conflict.` topic prefix, summarises the `ConflictEnvelope` payload (counts of `concurrent_block_edit` vs `structural_delete_edit`), and renders an interactive modal with side-by-side local / remote content and three action buttons (Keep local, Use remote, Open file). "Use remote" on a `concurrent_block_edit` dispatches a fresh `UpdateBlockContent` transaction through `com.nexus.editor::apply_transaction` so the user's choice lands as a normal local op (the CRDT publisher records it, undo can roll it back). `structural_delete_edit` is read-only in v1 — the modal renders a description of who deleted vs edited plus the surviving edit's content and points the user at "Open file" for manual resolution. Wire shape extended additively: `ConflictEnvelope.conflicts` now carries `ConflictDetail` (a `Conflict` flattened into the same JSON shape plus optional `local_content`, `remote_content`, `delete_origin` fields); pre-BL-074 subscribers reading just `kind` / `block_id` keep parsing unchanged. The publisher's `build_conflict_detail` populates the snapshots: live tree content for the local side, the remote op's `new_content` (UpdateBlockContent) / `block.content` (InsertBlock) for the remote side, and origin = whichever side issued the delete for `structural_delete_edit`. Shell-side: new `conflictStore.ts` (queue+current pattern), `applyResolution.ts` (extracted helper for unit-testable IPC dispatch), and `ConflictModal.tsx` registered as an overlay view at priority 90.
- **Tauri popout-window ops forwarding** — turned out to be already covered by `bridge::kernel_subscribe`'s per-window scoping; no extra code shipped.

**Tests:**
- `nexus-crdt`: 49 unit tests covering the OpLog idempotency / gossip slicing, RGA text merge, `apply_local` / `apply_remote` causality, `PersistedCrdt` round-trip, `OpLog::merge` union semantics, `ConflictEnvelope` wire shape (including the legacy-payload decode test that pins back-compat for pre-BL-074 subscribers).
- `nexus-bootstrap --lib`: 33 tests covering the publisher orchestrator, the pull-landing reload, and `build_conflict_detail`.
- Shell suite at 1212 tests stayed green through every closure pass; the conflict-modal closure added 9 conflict-store + 6 apply-resolution + 1 publisher-detail + 1 wire-envelope + 1 legacy-decode test.

**Pre-existing limitations preserved:**
- **Reparenting / move-loop detection** — pre-existing CRDT limitation, separate from BL-074. Not auto-resolvable in v1; manual editor resolution remains the only path.
- **Auto-resolving delete-vs-edit** — `structural_delete_edit` stays read-only in the modal; re-creating a deleted block or re-issuing a delete after re-creation requires more thought than v1 ships.
