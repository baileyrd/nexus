# BL-063: Terminal FTS5 scrollback index

**Status:** closed — 2026-05-07

**Source**: Terminal Integration Assessment (2026-05-06); PRD-09 §19.3
**Files**: `crates/nexus-terminal/src/{persist.rs, core_plugin.rs, lib.rs}`, `crates/nexus-bootstrap/src/lib.rs`, `shell/src/plugins/nexus/terminal/{CrossSearchView.tsx, CrossSearchPaneView.tsx, index.ts, terminal.css}`, two new ts-rs bindings (`CrossSessionSearchArgs.ts`, `ScrollbackHit.ts`)
**Related**: PRD-09 §19.3; `SqliteSessionStore` shipped Phase M; BL-062 eviction persister wires the same store this index sits on top of

Closes the gap where evicted scrollback was unsearchable. Adds an FTS5 virtual table over `(session_id, line_text, ts_ms, line_index)`, indexed on every persistence write. New IPC handler + ⌘⇧F sidebar panel surfaces hits across every session in the workspace.

- **Schema.** `scrollback_fts USING fts5(session_id UNINDEXED, line_text, ts_ms UNINDEXED, line_index UNINDEXED, tokenize='porter')`. Only `line_text` is searchable; the metadata columns are stored as `UNINDEXED` so SELECTs can constrain by session/time without paying tokenisation cost. The `porter` tokenizer powers prefix queries (`error*`) the cross-session search UI relies on. Lives in the same `SQLite` file as `terminal_sessions`, but is intentionally rebuildable from on-disk scrollback blobs — when a backup-export mechanism eventually lands it can skip the FTS table.
- **Indexing.** `save_scrollback` calls a private `reindex_scrollback` helper that:
  - Splits the byte blob on `\n`.
  - ANSI-strips each line (BL-062's existing `strip_ansi`).
  - Drops blank / whitespace-only lines (~3× row count savings on a typical build log).
  - DELETEs old rows for the session id, then bulk INSERTs new ones — all in a single `unchecked_transaction` so a concurrent reader never sees a half-populated index.
  - Records `unix_now_millis()` once per save. Per-line timestamps would require richer scrollback metadata than what's on disk; the per-save granularity is fine for `since_ts` filtering ("hits from the last hour").
- **Query API.** `SqliteSessionStore::cross_session_search(query, is_regex, session_ids?, since_ts?, limit)` returns `Vec<ScrollbackHit { session_id, text, ts_ms, line_index }>`. The `is_regex=false` path goes through FTS5's `MATCH` for full-text indexing. The `is_regex=true` path is a regex_lite scan over the rows narrowed by `session_ids` / `since_ts` — slower but correct, and the user only reaches it for non-trivial regexes the FTS5 tokenizer can't express. Empty / whitespace-only queries short-circuit to `Vec::new()` rather than returning every row.
- **`delete` clears FTS rows.** When `delete(id)` runs, the FTS rows for that session are dropped alongside the metadata row and the on-disk blob — no stale results from sessions the user explicitly removed.
- **IPC.** New `com.nexus.terminal::cross_session_search` handler (id 25 — DoD-suggested 24 was already taken by BL-064's `suggest`; ids are append-only). Args wrap the store API; `limit` defaults to 100 at the dispatch layer when the caller omits it. Returns a clear "session store not attached" error when bootstrap didn't open one.
- **Plugin wiring.** `TerminalCorePlugin` gained a `with_session_store(Arc<Mutex<SqliteSessionStore>>)` builder. Bootstrap opens the store once and uses the same `Arc` for both the BL-062 eviction persister and the new search handle, so an evicted scrollback is immediately searchable.
- **Shell `CrossSearchView`.** New sidebar leaf revealed by the `nexus.terminal.crossSearch.show` palette command, default keybinding ⌘⇧F (Cmd+Shift+F on macOS, Ctrl+Shift+F elsewhere). Live-search input debounced at 200 ms with a 2-character floor. Regex checkbox flips `is_regex`. Results grouped by `session_id` with truncated UUID prefixes for readability; full id surfaces on hover via the row's `title` attribute. Errors / loading / empty states all surface inline.

**Deliberately deferred:**

- **Jump-to-line affordance.** The hit carries `line_index` so a "jump to this line in the original session" surface is one click away — but most evicted sessions can't have their scrollback opened back up in xterm without restoring state we don't track yet. The shell view shows the line text directly (good enough for the "I remember running X" use case); the jump path lands when session-restore lands.
- **Periodic incremental indexing.** Today, scrollback is only indexed on eviction (the `save_scrollback` callsite). A live session's output isn't searchable until it's evicted. That's the right tradeoff for a forge with mostly-stopped sessions; if a workload needs live indexing, the drainer thread can periodically `save_scrollback` (with a bounded blob) to make the latest tail searchable. PRD-09 §19.3 doesn't require this, and the I/O cost of frequent re-indexing on a 10 MiB blob is non-trivial.
- **FTS5 ranking.** Hits are returned in `(ts_ms DESC, line_index ASC)` order, not by relevance. FTS5's `bm25()` ranking would be a one-liner addition but the result-list UX is more useful as "newest results first" than "most relevant" for terminal output, where a fresh error usually outranks an old one regardless of token weighting.
