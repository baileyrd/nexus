# BL-037: AMB activity timeline

**Status:** closed — 2026-04-30

**Delivered:** AI-AMBIENT-COPILOT-PLAN pattern 7 lands as a per-forge JSONL log + a default-off shell pane that mirrors what every AI surface in the app just did. **Recorder** ([`crates/nexus-ai/src/activity_log.rs`](../../crates/nexus-ai/src/activity_log.rs)): new module owning `ActivityEntry { id, timestamp (RFC3339 UTC), session_id, surface, provider, model, prompt, files, tool_calls, outcome, error, duration_ms }` plus the `ActivitySurface` (`chat` / `ask` / `cmdi` / `ghost` / `complete` / `enrich` / `other`) and `ActivityOutcome` (`ok` / `error` / `cancelled`) enums and the `ActivityToolCall { name, ok }` row type. `ActivityRecorder { ctx: Arc<KernelPluginContext>, write_lock: Arc<Mutex<()>> }` exposes `append(entry)` (best-effort fire-and-forget — disk failures `tracing::warn` and return), `read_all() -> Vec<ActivityEntry>` (oldest first; corrupt lines are skipped, not fatal), and `clear()` (deletes the on-disk file). The `Mutex<()>` serializes the read-modify-write window so two AI calls finishing at the same instant don't lose entries; the `Arc<KernelPluginContext>` clone keeps recorder cheap for handler futures. Storage path is `<forge>/.forge/ai-activity.log` (`ACTIVITY_LOG_PATH` constant); each entry is one JSON object per line. `ACTIVITY_LOG_MAX_BYTES = 256 KiB` triggers `head_trim_bytes` head-truncation to keep the file ≈1k entries; `ACTIVITY_PROMPT_MAX_CHARS = 256` clamps prompt text on append (Unicode-safe, ellipsis-suffixed). `append` `delete_file`s before `write_file` to sidestep the `KernelPluginContext::write_file` validator's `tail = ""` bug that appends a trailing `/` when overwriting an existing file (EISDIR on `tokio::fs::write`); the recorder is the only writer of this file so a stale read between delete + write is impossible. After a successful write the recorder publishes `com.nexus.ai.activity_appended` with the entry payload so subscribers can update without polling. **IPC** ([`crates/nexus-ai/src/core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs) + [`ipc.rs`](../../crates/nexus-ai/src/ipc.rs)): two new handlers — `HANDLER_ACTIVITY_LIST = 18` (args [`AiActivityListArgs { limit?: u32 }`](../../crates/nexus-ai/src/ipc.rs); returns [`AiActivityListResult { entries: Vec<ActivityEntry> }`](../../crates/nexus-ai/src/ipc.rs) newest-first) and `HANDLER_ACTIVITY_CLEAR = 19` (`{ cleared: bool }`). Both short-circuit before the `ctx.ok_or_else` arm in `dispatch_async` so the shell pane can hydrate even if it polls before `wire_context` runs (returns empty / `cleared:false` when the recorder is `None`). `AiCorePlugin` gains `activity: Option<ActivityRecorder>` populated in `wire_context` next to the `ToolRegistry` seed so the recorder shares the same `Arc<KernelPluginContext>` as the rest of the plugin. **Recording hooks**: `handle_stream_chat` and `handle_stream_ask` capture pre-flight metadata (provider / model / prompt / surface / `Instant::now()`) up front and record an `ActivityEntry` on every exit path — provider unconfigured, embed unconfigured (ask only), retrieve failure, provider build failure, dispatch error, cancellation, success — so an error timeline is just as useful as a success timeline. `run_tool_dispatch_loop` now returns a typed `ToolDispatchOutcome { text, tool_calls: Vec<ActivityToolCall>, files: Vec<String> }` instead of bare `String`; tool calls are recorded with `ok = !is_error`, and any tool input carrying a `path` field (read_file / write_file / note_append / similar) contributes a deduped entry to `files`. `handle_stream_ask` populates `files` from the RAG sources that grounded the answer, deduped while preserving retrieval order. `handle_stream_chat` accepts a new optional `surface` field on `AiStreamChatArgs` (`chat` / `cmdi` / `ghost` / `complete` / `enrich` / `other`) so callers tag their events; default heuristic = `chat` (or `complete` when `mode=complete`). `resolve_surface(explicit, mode)` does the resolution; `last_user_prompt` extracts the most recent user message for the timeline regardless of provider's history shape. Errors flow through one shared `record_activity_error` so the entry shape is identical across error paths. **Shell pane** ([`shell/src/plugins/nexus/activityTimeline/`](../../shell/src/plugins/nexus/activityTimeline/)): new default-off `nexus.activityTimeline` plugin contributing a `paneMode` view + activity-bar item (priority 55, between AI chat at 50 and Processes at 60). `activityTimelineStore.ts` mirrors the on-disk JSONL inside a Zustand store with a `TIMELINE_ENTRIES_CAP = 1000` ring; `prepend(entry)` drops the oldest when at cap. `ActivityTimelineView.tsx` renders a header + toolbar (text filter, surface dropdown, clear button, count) + entry list, each entry showing `HH:MM:SS  surface  outcome-glyph  prompt  duration` on the primary line and `model / files (1 + N more) / tool ✓ tool ✗ / error` on the secondary line. The clear button calls `com.nexus.ai::activity_clear` via the kernel and empties the local store. `index.ts` hydrates the store from `com.nexus.ai::activity_list` on activate (and on every activity-bar route), then subscribes to `com.nexus.ai.activity_appended` so live activity prepends without polling. Lifecycle ties to `workspace:opened` / `workspace:closed` (subscribe / unsubscribe) and covers the boot race by hydrating + subscribing immediately if `api.kernel.available()` is already `true`. PluginRegistry sweeps the bus disposer on plugin unload.
