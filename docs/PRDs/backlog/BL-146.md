## Closure notes (2026-05-16)

- `nexus-remote::RemoteClient` exposes `wait_for_disconnect()` (a `Notify`-backed
  signal that fires when the inbound router exits, whether via EOF, read error,
  or explicit `shutdown`).
- `ReconnectingRuntime` now holds a `SubscriptionRegistry`
  (`HashMap<id, (filter, sink)>`) plus public `subscribe(id, filter, sink)` and
  `unsubscribe(id)` methods that mirror the underlying `RemoteClient` API but
  thread the registration through the wrapper.
- A per-client watchdog task is spawned every time a fresh `RemoteRuntime` is
  installed into the slot (initial connect via `ensure_connected` /
  `ensure_client`, every iteration of the `ipc_call` reconnect loop). The
  watchdog awaits `wait_for_disconnect`, walks the backoff schedule, builds a
  new client, and re-installs every registered subscription against it before
  emitting `ConnectionState::Connected`.
- Replay counts publish on a new `subscribe_replays()` broadcast channel; the
  Tauri bridge forwards them on a new `kernel:subscriptions-replayed` event so
  the shell can render "N subscriptions restored" toasts (frontend wire-up
  pending; channel + envelope are in place).
- Tauri bridge's `kernel_subscribe` Remote arm now routes through
  `runtime.subscribe(...)` instead of `ensure_client().subscribe(...)`.
  `kernel_unsubscribe` calls `runtime.unsubscribe(...)` so the registry stays
  consistent.
- Edge cases covered by `crates/nexus-bootstrap/tests/subscription_replay.rs`:
  1. Two-runtime fixture: subscribe â†’ drop transport â†’ publish on the second
     server's bus â†’ sink receives without manual re-subscribe.
  2. Subscribe issued during a Reconnecting window queues the entry; the next
     install replays it.
  3. Unsubscribe during Reconnecting removes the entry so it's not replayed.

## Original scope (preserved for context)

Today the BL-140 reconnect path drops every active remote subscription on the floor. When the SSH transport drops, the underlying `RemoteClient`'s router task shuts down â€” every `event_subscribe` installed against that client dies, and the new connection that gets rebuilt has no record of those subscriptions server-side. The Tauri bridge's forwarder tasks similarly terminate.

User-visible symptom: shell plugins that watch kernel events (activity timeline, editor file-change events, BL-074 CRDT ops bus, notifications inbox, etc.) silently stop receiving events after a remote reconnect and stay stopped until the plugin is re-activated. For local forges this is a non-issue (the kernel + bus are in-process).

Subscriptions are currently invisible to the reconnecting layer â€” they're installed directly on the underlying `RemoteClient` via `runtime.ensure_client().await?.subscribe(...)`. To replay, the reconnecting wrapper has to track every subscription that flowed through it.

**Scope:**

1. Add a `SubscriptionRegistry` on `ReconnectingRuntime` storing `(subscription_id, filter, mpsc::UnboundedSender<EventDelivery>)` triples. Expose a `subscribe(id, filter, sink)` method that records the triple AND issues the underlying client subscribe.
2. On successful reconnect (Connected transition after Reconnecting), iterate the registry and re-install every subscription against the new client.
3. Update the Tauri bridge's `kernel_subscribe` Remote arm to route through the new method instead of `ensure_client().subscribe`.
4. Emit a `com.nexus.kernel.remote_subscriptions_replayed` event (or surface via the existing `ConnectionState` channel as a `Connected{replayed: N}` variant) so frontend can tell the difference between "first connect" and "reconnected".

**Definition of done:**
- Two-runtime test fixture: subscribe to a CustomPrefix filter, force a transport drop, the next publish on the topic reaches the subscriber's sink without manual re-subscription.
- Edge case: subscribe issued during the Reconnecting window queues + applies on the next successful connect.
- Edge case: unsubscribe during Reconnecting removes from the registry so it doesn't get replayed.
- Tauri bridge's `kernel_subscribe` Remote forwarder survives a transport drop without the frontend re-subscribing.
