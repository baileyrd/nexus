# BL-042: AI-DIR Nexus-as-MCP-server

**Status:** closed — 2026-04-30

**Delivered:** AI-INTEGRATION-DIRECTIONS direction 7 lands as a stdio MCP **server** in [`crates/nexus-mcp/src/server.rs`](../../crates/nexus-mcp/src/server.rs), driven by `nexus mcp` ([`crates/nexus-cli/src/commands/mcp.rs`](../../crates/nexus-cli/src/commands/mcp.rs) `serve`) so external IDEs and Claude Desktop can drive a Nexus forge through the same kernel IPC boundary the CLI / TUI / shell already use. Distinct from BL-023 / BL-024 / BL-025 (those stay on the *client* / Host side). **Tool surface (15)**: `nexus_read_note` / `nexus_create_note` / `nexus_update_note` / `nexus_delete_note` / `nexus_list_notes` (CRUD over `com.nexus.storage::read_file` / `write_file` / `delete_file` / `query_files`); `nexus_search` (rebuild-then-query against `com.nexus.storage::search`); `nexus_backlinks` / `nexus_outgoing_links` / `nexus_graph_status` (knowledge-graph readers); `nexus_list_tags` / `nexus_list_tasks` / `nexus_toggle_task` (forge metadata); `nexus_ask` (RAG via `com.nexus.ai::ask` with a 120 s IPC timeout to absorb provider RTT); and the BL-042 closing skills pair `nexus_list_skills` / `nexus_render_skill` (call `com.nexus.skills::list` / `render` so authored prompt templates from `.forge/skills/` are reachable as named, parameter-substituted prompts from any MCP client — the `run_skill` slot called out in the BL-042 entry). **Resource surface**: forge notes are also enumerated under `mcp://nexus/notes/<vault-relative-path>` via `list_resources` / `read_resource` (BL-026 path) so clients with a Resources UI see the vault as a browsable tree alongside the tools; the listing root `mcp://nexus/notes` is intentionally non-readable. **Plumbing**: `NexusMcpServer::new(Arc<KernelPluginContext>)` plus three thin `*_call<T>` helpers — `storage_call` (30 s timeout against `com.nexus.storage`), `skills_call` (30 s against `com.nexus.skills`), and a direct 120 s `ipc_call` for `com.nexus.ai::ask`. `ServerInfo::with_instructions(…)` advertises notes + skills + the resource prefix to the connecting client so Claude Desktop's "what can this server do?" prompt is self-documenting. Every tool is capability-checked at the kernel because every call funnels through `context.ipc_call(...)` rather than touching engines directly — same trust posture as the CLI / TUI. **Trust boundary**: stdio only — `serve_stdio()` wraps the server in `rmcp::transport::io::stdio()` so the parent process (Claude Desktop / IDE) is the trust boundary; no TCP listener, no auth layer, no off-host reachability (pre-audited 2026-04-16, see the security follow-ups below). **Coverage**: `cargo test -p nexus-mcp` 51 passing (was 46; +5 net) — new `render_skill_input_defaults_values_to_empty_map`, `render_skill_input_round_trips_values_map`, `list_skills_output_serializes_count_and_skills`, `render_skill_output_serializes_id_name_body` plus the existing `parse_note_uri_*` / `build_note_resource_*` shape tests. `cargo clippy -p nexus-mcp --all-targets` clean; `bash scripts/check_ipc_drift.sh` clean (the new tools are plain `serde_json::Value` round-trips against existing `com.nexus.skills` handlers — no IPC-typed structs changed). **Out of scope**: WebSocket / HTTP transport (the trust-boundary follow-up flagged that an auth layer must land alongside any non-stdio transport); MCP **Prompts** surface (the natural pairing with skills, but the `nexus_render_skill` tool already lets clients invoke skills today — promoting them to first-class `prompts/list` is a follow-up if a client appears that prefers that channel); a daemonised long-running server (today `nexus mcp` is a one-shot stdio child of its launcher, which is what Claude Desktop / Cursor / Cline expect).
