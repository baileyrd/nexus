# BL-101: `granted_caps.json` encryption at rest

**Files**: `crates/nexus-plugins/src/{grants_crypto.rs (new),loader.rs,lib.rs}`, `crates/nexus-plugins/Cargo.toml`, `Cargo.toml` (workspace `chacha20poly1305` + `rand` deps)

`grants_crypto` is a new internal module that wraps the existing `granted_caps.json` writes/reads in a ChaCha20-Poly1305 AEAD blob. File format: 6-byte magic header `b"NXENC1"`, 12-byte random nonce, then `cipher.encrypt(nonce, plaintext)` (ciphertext + 16-byte tag) â€” the contents are intentionally not human-editable. Reads dispatch on the leading byte: `N` â†’ encrypted; `{` â†’ plaintext (legacy / `NEXUS_NO_KEYRING=1` fallback). Both `load_granted_high_risk_caps` and `write_grant` now route through the module and accept a `plugin_id` so each plugin's grants are scoped to a separate keyring entry.

**Key management.** A 32-byte master key per plugin, stored in the OS keyring under `nexus.plugin_grants:<plugin_id>` as 64 hex characters. The first encrypted write generates a fresh random key and stores it; subsequent reads/writes fetch it. Wiping a keyring entry effectively rotates the key â€” the next read fails authentication, the loader logs a `tracing::warn!`, and grants reset to deny-all (per the BL-101 DoD: forces re-consent).

**Failure modes.** Keyring unreachable, `NEXUS_NO_KEYRING=1`, or `OsRng` failure all degrade to plaintext writes with a `tracing::warn!` so the reduced security level is visible in logs. Decryption failure on an `NXENC1` blob always returns deny-all.

**Deviation from the original PRD spec.** PRD-02 Â§6.3 prescribed PBKDF2-SHA256 over `forge_path + "granted_caps_key"` with a companion `.granted_caps.salt`. The shipped design stores the AEAD key directly in the keyring (32 bytes from `OsRng`), skipping PBKDF2: PBKDF2 only buys security when the input password is low-entropy (a memorised passphrase, a path string), and a uniformly random keyring-stored 256-bit key already saturates the AEAD's security margin. Adding PBKDF2 + a stored salt would have introduced a second attack surface (the salt file) without raising the security floor. Documented here so the deviation is explicit.

**Tests.** Six unit tests in `grants_crypto` cover round-trip, magic-prefix detection, AEAD authentication failure on tampered ciphertext / wrong key, and a short-blob safety case. A new loader test (`load_grants_resets_on_unrecoverable_encrypted_blob`) confirms the deny-all + warning behaviour on a blob with valid magic but bogus ciphertext. The existing `write_grant_round_trips_and_sorts` test now runs in `NEXUS_NO_KEYRING=1` mode so the plaintext shape stays byte-stable for assertion.

**Deferred.** `.granted_caps.salt` companion file (folded into the magic+nonce blob); explicit log-line on the *first* successful encrypted write per plugin (the first revoke after this lands silently transitions the file format â€” operators see the format change in the file itself).
