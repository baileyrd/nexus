# BL-142: REPL / in-buffer evaluation

**Status:** Phases 1 + 2a + 2b closed; Phase 2b CM6 wiring verified via 14 EditorView-level integration tests on 2026-05-17 (real CM6 mount under happy-dom — `replIntegration.test.ts`). Two production bugs surfaced and fixed in the process (see "Phase 2b verification + production-bug fixes" below). Phase 3 (replKernels settings panel polish) remains optional.

## Phase 2b verification + production-bug fixes (closed 2026-05-17)

The 2026-05-16 closure flagged the gutter click / Shift-Enter keymap / inline-output widget as "needs a live Tauri window to verify." Mounting a real `EditorView` under happy-dom turned out to be enough — and the integration pass caught two latent bugs the pure-factor tests couldn't see:

- **`replGutter` watcher dispatched from inside a CM6 update cycle.** The `ViewPlugin.update(u)` hook called `view.dispatch(...)` synchronously when `u.docChanged` — CM6 throws `Calls to EditorView.update are not allowed while an update is in progress`. Fix wraps the refresh in `queueMicrotask` with a `destroyed` guard so the dispatch lands as a follow-up transaction. The constructor path already used this idiom; the update path now matches.
- **`replOutput` declared block decorations from a `ViewPlugin`.** `Decoration.widget({ block: true })` from `ViewPlugin.fromClass({ decorations: (v) => v.decorations })` throws `Block decorations may not be specified via plugins` at view construction — so the inline output widget never rendered in production. Fix converts the decoration set to a `StateField` (which CM6 accepts as the source of block decorations); a companion `ViewPlugin` subscribes to `useReplStore` and dispatches a `refreshReplOutputDecorations` effect so the field also rebuilds when `ensureSession` mints a fresh sessionId — which closes the related "widget never appears after the first Run click" gap.

New test surface (14 tests, all green):
- 3 `replGutterExt — EditorView integration` — seed-via-watcher, refresh on doc-change-adding-a-fence, clear when last fence is removed.
- 4 `replKeymapExt — EditorView integration` — Shift-Enter and Mod-Enter inside a REPL cell fire `onRun`; outside any cell fall through; cursor in second of two cells picks the right block. Drives real `KeyboardEvent` dispatch through `view.contentDOM` so the CM6 keymap chain executes end-to-end.
- 5 `replOutputExt — EditorView integration` — no widget when `resolveSessionId` returns null; widget DOM appears + identity is preserved across store updates; per-cell buffers stay independent; widget rebuilds on doc change adding a fence; widget unsubscribes cleanly on `view.destroy()` (no post-destroy appends reach the detached DOM).

`pnpm --filter nexus-shell test` 1584 → 1598; `pnpm --filter nexus-shell typecheck` + `lint` clean.

**What still needs a live Tauri window**: actual `python3` (or other kernel) subprocess + the `repl_start` → PTY → output-bus pipeline end-to-end. That path is covered by the Rust IPC tests in `crates/nexus-bootstrap/tests/terminal_repl_ipc.rs`; in-process the only remaining unknown is whether Tauri's IPC adds latency that pushes the < 500 ms Shift-Enter → output target out of range. Worth a manual smoke check next time the shell is being driven on a desktop, but no longer a closure blocker.

## Phase 1 closure notes (2026-05-16)

Backend REPL surface shipped. A caller can spawn a language kernel
through `com.nexus.terminal::repl_start`, write code to it with
`repl_eval`, and tear it down with `repl_stop` — reusing all the
existing terminal infrastructure (PTY, scrollback, output bus,
memory monitor). The shell-side editor REPL UI (cell block type,
CM6 decorations, `Shift-Enter` keybinding, `<ReplOutput />`
rendering, `replKernels` settings panel) is Phase 2.

**Shipped:**

- Four new IPC handlers on `com.nexus.terminal`:
  - `repl_start { lang, program, args, working_dir?, env? } → { id, lang }`
    — spawns the kernel via `ServerSpawnConfig::create_session`,
    registers the session in a new `repls: HashMap<SessionId, ReplInfo>`
    map, names the session `repl:<lang>` so existing session-listing
    surfaces can tell REPLs apart from regular shells.
  - `repl_eval { id, code }` — writes `code` (with auto-appended `\n`
    matching `send_input`'s contract) to the kernel's PTY stdin.
    Output streams asynchronously on the existing
    `com.nexus.terminal.output.<id>` event topic.
  - `repl_stop { id }` — closes the session via `close_session` and
    removes the REPL bookkeeping entry.
  - `repl_list {} → Vec<ReplInfo>` — sorted snapshot for shell
    discovery + tests.
- `repl_eval` + `repl_stop` reject calls against non-REPL sessions
  with a clear "not a registered REPL" error. Protects against a
  keybinding mis-fire dropping code into the user's shell.
- Bootstrap manifest registers the four commands (ids 26–29);
  `cap_matrix.toml` classifies `repl_start` under `process.spawn`
  (same posture as `create_session`) and `repl_eval` / `repl_stop`
  / `repl_list` alongside the existing post-spawn handlers.
- 6 IPC tests in
  `crates/nexus-bootstrap/tests/terminal_repl_ipc.rs` cover the
  start/list/stop lifecycle, concurrent sessions with distinct ids,
  Python `print()` output streaming end-to-end through `read_raw_since`,
  Python error output (`NameError`) on the same stream, non-REPL
  session rejection on both `repl_eval` and `repl_stop` (with
  on-the-air verification that the rejected `repl_stop` leaves the
  underlying session alive), and empty-list invariants. Python
  tests skip cleanly when `python3` isn't on `$PATH`.

## Original scope (preserved for context)

**Source**: Zed capability assessment — see [../research/zed-capability-assessment.md](../research/zed-capability-assessment.md). Filed 2026-05-15.
**Effort**: Medium. Builds on existing `nexus-terminal` PTY infrastructure.
**Crates**: `nexus-terminal` (new `repl_*` IPC handlers), `shell/src/plugins/nexus/editor/` (cell block type + output rendering).

Zed ships Jupyter-style REPL execution: code cells in an editor buffer run against a language kernel; output renders inline below the cell. Nexus's block-tree MDX editor and `nexus-terminal`'s PTY session management are the natural foundation.

**Scope:**

- New MDX block type `<Repl lang="python" />` (or fenced code block with `repl` meta): marks a cell for in-buffer execution.
- New IPC handlers in `nexus-terminal`: `repl.start(lang, kernel_command)` → `session_id`; `repl.eval(session_id, code)` → streams `stdout/stderr` lines; `repl.stop(session_id)`.
- CM6 decoration: "Run" button gutter marker on repl cells; `Shift-Enter` keybinding to eval the current cell.
- Output block rendered below the cell as a `<ReplOutput />` MDX component: text, ANSI-stripped, error-highlighted. Output is ephemeral (not persisted to the source file).
- Language kernels configured via `nexus.editor.replKernels`: `{ python: "python3 -i", node: "node --interactive", ... }`.
- Works in both document-mode (markdown + MDX) and code-mode tabs.

**Definition of done:**
- `repl.start` / `repl.eval` / `repl.stop` IPC handlers wired and tested against a Python3 subprocess
- `Shift-Enter` on a repl cell streams output into the inline output block within 500 ms
- ANSI codes stripped; error output red-highlighted
- Multiple concurrent repl sessions supported (one per kernel per tab)
- `nexus.editor.replKernels` setting controls which languages are enabled (default: `{}` — opt-in)
- 6 integration tests: start/eval/stop lifecycle, concurrent sessions, error output, Shift-Enter keymap, output cleared on re-eval, session cleaned up on tab close
