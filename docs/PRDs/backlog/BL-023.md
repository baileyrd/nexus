# BL-023: MCP WebSocket + HTTP+SSE transports

**Status:** closed — 2026-04-28

**Delivered:** PRD-14 §4.2 lands behind a transport-abstraction layer in `nexus-mcp`. Pre-BL-023 [`McpClient::connect`](../../crates/nexus-mcp/src/client.rs) only spawned a child process and ran the MCP handshake over its stdio (`TokioChildProcess`). `mcp.toml` now carries a per-entry [`McpTransport`](../../crates/nexus-mcp/src/config.rs) discriminator (`stdio` / `http` / `websocket`) keyed off a new `transport` field on [`McpServerSpec`](../../crates/nexus-mcp/src/config.rs); back-compat is preserved by `transport = "stdio"` being the serde default — every existing file that omits the field continues to dispatch through the original child-process path. New remote-only fields ride on the same struct (`url: Option<String>`, `auth_header: Option<String>`, `headers: BTreeMap<String, String>`) with `skip_serializing_if` so stdio-mode entries don't grow unused TOML keys on round-trip. Validation moved into [`McpHostConfig::validate`](../../crates/nexus-mcp/src/config.rs): stdio entries must have a non-empty `command`; http / websocket entries must have a non-empty `url`. **HTTP+SSE (Streamable HTTP)**: `McpClient::connect_http` constructs an [`rmcp::transport::StreamableHttpClientTransport`](https://docs.rs/rmcp/1.5.0/rmcp/transport/struct.StreamableHttpClientTransport.html) via `from_config(StreamableHttpClientTransportConfig::with_uri(...).custom_headers(...).auth_header(...))`. Header strings are validated up front (`http::HeaderName::try_from` / `HeaderValue::try_from`) so malformed entries surface as `McpClientError::Config` before any wire I/O — easier to diagnose than rmcp's typed-error stack on first request. The reqwest-backed default client is constructed inside rmcp via the new `transport-streamable-http-client-reqwest` feature; the call site deliberately does NOT name `reqwest::Client` because the workspace pins reqwest 0.12 for the AI provider crates while rmcp 1.5 ships with reqwest 0.13, and the two `Client` types are not interchangeable. Going through rmcp's `from_config` constructor avoids the version gap (an in-source comment documents the rationale so a future bump doesn't re-introduce a typed `reqwest::Client` reference). New rmcp features added to the workspace dep: `transport-streamable-http-client`, `transport-streamable-http-client-reqwest`, `reqwest-native-tls` (so `reqwest 0.13` picks up native-TLS for the system OpenSSL, dodging the rustls crypto-provider panic on `Client::new` that fired without an explicit TLS feature), and `client-side-sse`. **WebSocket (reserved)**: rmcp 1.5 ships only a stub at [`src/transport/ws.rs`](https://docs.rs/rmcp/1.5.0/src/rmcp/transport/ws.rs) whose top-of-file comment is "Maybe we don't really need a ws implementation?" — the MCP working group folded WebSocket into Streamable HTTP in the 2025-03-26 spec, deprecating the WS transport upstream. Honoring that direction, `McpTransport::Websocket` parses cleanly in `mcp.toml` (so a deployed file declaring it doesn't fail to load) but `connect()` returns a precise `McpClientError::Unsupported` whose message points the operator at `transport = "http"`. The variant stays in the schema so a future rmcp WS impl can be wired in without a breaking config change. New `McpClientError` variants: `Config { reason }` (validated-up-front spec-shape errors) and `Unsupported { reason }` (recognised-but-not-dispatchable transports). The existing `Spawn` / `Handshake` / `Service` cases are unchanged. **Refactor**: the handshake driver moved into a `run_handshake<F, E>(...)` helper so both transport branches race the rmcp `serve_client(...)` future against the existing `CONNECT_TIMEOUT` (15 s) uniformly; previously the timeout policy was inlined in the stdio branch. **Coverage**: `cargo test -p nexus-mcp` 38 passing (was 28; +10 net). New tests in [`config.rs`](../../crates/nexus-mcp/src/config.rs): `parses_http_transport` (the full remote shape), `http_transport_requires_url`, `stdio_transport_default_keeps_back_compat` (regression for the no-`transport` file format), `websocket_transport_parses_but_is_reserved`. New tests in [`client.rs`](../../crates/nexus-mcp/src/client.rs): `websocket_transport_returns_unsupported` (asserts the migration hint mentions both "WebSocket" and "http"), `http_transport_rejects_missing_url_with_config_error`, `http_transport_rejects_invalid_header_name`, `http_transport_handshake_times_out_against_dead_endpoint` (routes through `connect_http` against `http://127.0.0.1:1` and asserts non-success without panicking). `cargo clippy -p nexus-mcp --all-targets -- -D warnings` clean. **Out of scope**: connection pooling / reconnection policy (lives in `pool.rs` and is transport-agnostic — pre-BL-023 design intent already covers both transports because each `McpClient` is opaque to the pool).
