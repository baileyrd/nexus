# BL-025: MCP authentication

**Status:** closed — 2026-04-28

**Delivered:** PRD-14 §8's three auth flows land as a tagged enum on the BL-023 transport layer. New [`crates/nexus-mcp/src/auth.rs`](../../crates/nexus-mcp/src/auth.rs) module exposes `pub enum McpAuth { ApiKey { header, value }, Bearer { token }, OauthClientCredentials { token_url, client_id, client_secret, scope } }` with `#[serde(tag = "type", rename_all = "snake_case")]` so a `mcp.toml` block reads naturally:

```toml
[servers.alpha.auth]
type = "api_key"
header = "X-API-Key"
env = "ALPHA_API_KEY"

[servers.beta.auth]
type = "bearer"
token = "ey…"

[servers.gamma.auth]
type = "oauth_client_credentials"
token_url = "https://auth.example.com/oauth2/token"
client_id_env = "GAMMA_CLIENT_ID"
client_secret_env = "GAMMA_CLIENT_SECRET"
scope = "mcp:tools"
```

Each secret-bearing field uses an untagged `McpAuthSecret { Inline { value }, Env { env } }` (or its `client_id` / `client_secret` siblings) so `value = "…"` and `env = "…"` are mutually exclusive in the file — empty/unset env vars surface as a typed `AuthError::MissingEnv { name }` at resolve time. Future ADR-0009 keyring integration slots in alongside `Env` without a wire-format break (consumers of `ResolvedAuth` see only the final string regardless of source). [`auth::resolve(auth) -> Result<ResolvedAuth, AuthError>`](../../crates/nexus-mcp/src/auth.rs) is pure for `ApiKey` / `Bearer` (no I/O); the `OauthClientCredentials` path POSTs `grant_type=client_credentials` to the token endpoint via the workspace `reqwest 0.12` client (HTTP Basic auth on the credentials per RFC 6749 §2.3.1, `scope` in the form body, 30 s timeout — same order of magnitude as the rmcp `CONNECT_TIMEOUT`), parses the `TokenResponse`, validates `token_type` is `bearer` (case-insensitive), and returns the access token wrapped as `Bearer <token>`. Bearer's resolver is tolerant of either `"ey…"` or `"Bearer ey…"` so an operator migrating from a hand-rolled `auth_header = "Bearer …"` setup just changes the table header without re-stringing. `ApiKey` lands on `Authorization` if the `header` field is the canonical name (case-insensitive); otherwise on `extra_headers` so it ends up in rmcp's `custom_headers` map. **New `auth: Option<McpAuth>` field on [`McpServerSpec`](../../crates/nexus-mcp/src/config.rs)** with `skip_serializing_if = "Option::is_none"` so existing pre-BL-025 files round-trip untouched. **Wire-up in [`McpClient::connect_http`](../../crates/nexus-mcp/src/client.rs)**: when `spec.auth` is set the resolver runs first and any `MissingEnv` / `Oauth` / `OauthResponse` failure aborts connect with a new `McpClientError::Auth(#[from] AuthError)` variant — the operator gets a typed error pointing at the failing env var or token endpoint instead of the generic `Handshake` they used to see when the request landed without credentials. Resolved auth is merged into rmcp's `auth_header(...)` slot for `Authorization`-bearing flows and into `custom_headers(...)` for non-`Authorization` API keys; the legacy static `auth_header` field from BL-023 still works (back-compat with installs that pinned a token by hand) but a present `auth` block always wins on conflict — declarative beats legacy. **Why a hand-rolled flow rather than rmcp's `auth` feature**: rmcp 1.5 ships a full OAuth manager (`rmcp::transport::auth`) that covers PKCE / refresh tokens / DCR, but enabling it pulls in extra TLS-feature scaffolding the workspace doesn't otherwise need and exposes a stateful `OAuthState` surface that's overkill for the single-shot client-credentials flow PRD-14 §8 calls for. The ~50 lines in `auth.rs` are faster to audit and reuse the workspace `reqwest 0.12` already in the closure for the AI provider crates (rmcp's reqwest 0.13 is the rmcp transport's, not ours). Refresh-on-401 is deliberately not implemented — token is refetched on the next `connect()` (the rmcp transport's expired-session recovery covers MCP-level session expiration; OAuth token expiration is a separate concern that compounding both into one round-trip wouldn't simplify). **ADR-0009 interaction**: today `Env`-indirected secrets fall through to the env-var lookup; the same `McpAuthSecret` enum can grow a `Keyring { service, account }` variant when ADR-0009's keyring service lands without breaking deployed `mcp.toml` files. **Coverage**: `cargo test -p nexus-mcp` 47 passing (was 38; +9 net). New tests in [`auth.rs`](../../crates/nexus-mcp/src/auth.rs): `api_key_inline_authorization_lands_on_authorization_slot`, `api_key_custom_header_lands_on_extras`, `bearer_inline_prepends_scheme_when_missing`, `bearer_inline_preserves_caller_supplied_scheme`, `env_indirection_resolves_when_set`, `env_indirection_errors_when_unset`, `toml_parses_api_key_with_env_indirection`, `toml_parses_oauth_client_credentials`. New integration test in [`client.rs`](../../crates/nexus-mcp/src/client.rs): `http_transport_auth_missing_env_surfaces_as_auth_error` (proves the typed `Auth(MissingEnv)` error fires from `connect()` before any network I/O happens). `cargo clippy -p nexus-mcp --all-targets -- -D warnings` clean. **Out of scope**: PKCE / authorization-code flow / DCR / refresh tokens (rmcp's `auth` feature is the natural home if those are wanted later); keyring integration (waits on ADR-0009 enforcement).
