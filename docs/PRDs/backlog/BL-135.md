# BL-135: Notification router refactor

**Status:** closed — 2026-05-15

**See above for the close note.** Forge-local `<forge>/.forge/notifications.toml` schema + `Router` + `AiEvent` subscriber + producer migration + live-reload all shipped. Legacy `[notifications.*]` blocks remain as a fallback for forges that haven't authored `notifications.toml`.

**Source**: [BL-135.md](BACKLOG.md). Filed 2026-05-14; closed 2026-05-15.
**Files**: `crates/nexus-notifications/Cargo.toml` (`toml` + `notify` + `chrono` + `tokio` + dev-`tempfile` deps); `crates/nexus-notifications/src/lib.rs` (re-export `config::*` and `router::*`; `NOTIFICATIONS_CONFIG_RELPATH`); `crates/nexus-notifications/src/config.rs` (new — `NotificationsConfig`, `SourceConfig`, `ChannelsConfig`, `Severity`, `QuietHours`, `ResolvedSource`, `ConfigError`); `crates/nexus-notifications/src/router.rs` (new — `Router`, `Resolution`, `current_min_of_day`); `crates/nexus-notifications/src/core_plugin.rs` (`from_config` + `with_transports_and_config` constructors; `dispatch_routed`; AiEvent subscriber spawned from `on_start` + `wire_context`; live-reload via `notify` watcher; `SendArgs.channel` becomes optional and `source`/`severity` added; new `SendReply.channels` / `failures` / `routing` fields + `ChannelFailure`); `crates/nexus-bootstrap/src/lib.rs` (`load_notifications_config` — reads `<forge>/.forge/notifications.toml` with legacy `[notifications.*]` fallback that synthesises default `[sources.*]` so source-tagged producers still deliver on Desktop); `crates/nexus-workflow/src/core_plugin.rs` (`NotifyStepArgs` reshape — `channel` now optional, `source` + `severity` added; dispatcher defaults to `source = "workflow"`); `crates/nexus-cli/src/commands/agent.rs` (auto-notify emits `source = "agent"` instead of hardcoded `channel = "desktop"`); `crates/nexus-cli/src/commands/notify.rs` (`send` accepts optional `channel` or `source` + optional `severity`; `print_outcome` renders the BL-135 reply shape); `crates/nexus-cli/src/main.rs` (`NotifyCommand::Send` grows `--source` / `--severity`; `--channel` becomes optional and `conflicts_with = "source"`); regenerated `packages/nexus-extension-api/src/generated/ipc/{SendArgs,SendReply,ChannelFailure,Severity}.ts` + `crates/nexus-bootstrap/schemas/ipc/com_nexus_notifications__send_{args,reply}.json`.
**Related**: [BL-133](#bl-133-multi-channel-notification-output-thoth-port--2026-05-14--v1-scope-desktop--discord) (the dispatcher this refactors), [BL-134](BL-134.md) (provides the `com.nexus.ai.runtime.*` topic the new subscriber translates), [BL-136](BL-136.md) (downstream — consumes the same router output).

Before this change, every producer hardcoded which channel/transport it targets: the workflow `notify` step picked `channel = "desktop"` inline, the agent CLI's `--notify-after-secs` post path picked `"desktop"`, the BL-134 runtime's typed `AiEvent` topics had nowhere to land. This scales poorly once a fourth high-volume source shows up; the BL-133 follow-up note called it out. BL-135 pulls producer→transport routing out of producers and into a single forge-local config that the notifications plugin reads on `on_start` and reloads via a file watcher.

**What landed.**

- **`<forge>/.forge/notifications.toml`** — new schema parsed in `nexus-notifications`. Two top-level groups: `[sources.<name>]` blocks (`route = [...]`, `min_severity`, `quiet_hours`, `on = [...]`) and `[channels.<name>]` blocks (Discord webhook URL, Telegram bot token + chat id, SMTP host/port/from/to/etc). The legacy `[notifications.<channel>]` blocks in `config.toml` continue to be read when `notifications.toml` is absent so a pre-BL-135 forge keeps working untouched. Unknown channel names in `route = [...]` are warned + dropped at config-load time; quiet_hours malformed strings surface as `ConfigError::QuietHours` and the boot path falls back to an empty router.
- **`Router`** — `Arc<RwLock<RouterState>>` over a name → `ResolvedSource` map. `resolve(source, severity, min_of_day) -> Resolution` returns `Routed(Vec<Channel>)` / `Filtered` / `UnknownSource`. Filters: severity below `min_severity` → `Filtered`; current local minute-of-day inside `quiet_hours` (overnight ranges supported) → `Filtered`; empty route → `Filtered`. `swap_config` rebuilds the rule set in place under a write guard — concurrent in-flight dispatches release their read guard and pick up the new rules on next call. Live-reload reuses this so transports aren't recreated mid-flight.
- **`NotificationsCorePlugin::dispatch_send` reshape** — `SendArgs.channel` becomes optional; `source` and `severity` added. Override path: when `channel` is set, route directly to that transport bypassing the router (legacy callers and the `nexus notify send --channel discord` CLI verb keep working). Router path: when `source` is set, consult the router; fan out to every resolved channel; aggregate into `SendReply { delivered, channel, channels, failures, routing }`. Per-channel failures are reported in the `failures: Vec<ChannelFailure>` array without aborting the fan-out — the override path still surfaces a transport error as a top-level IPC error to preserve BL-133 semantics. Rejecting "neither `channel` nor `source` supplied" lives at the IPC entry point.
- **Built-in `AiEvent` subscriber** — `on_start` captures the `KernelPluginContext` from `wire_context`, subscribes to the `com.nexus.ai.runtime.` topic prefix, and spawns a tokio task that translates each `Finished` / `Failed` payload into a `Notification` (`source = "ai_runtime"`). The translation is suffix-driven so it works whether BL-134 emits typed `kind = "finished"` payloads (post-Phase-2) or just raw topic-prefix events today; token chunks / tool calls / round_* are silent on purpose. Republishes through the existing `com.nexus.notifications.delivered` topic so the shell toast pipeline picks routed events up without code change. Dormant when no runtime topic publishes — BL-135 doesn't depend on BL-134 Phase 2.
- **Live reload** — `on_start` spawns a `notify::recommended_watcher` on the config file's *parent* directory (watching the inode directly is unreliable with atomic-rename editors) and filters events by filename. On Create / Modify / Remove of `notifications.toml`, re-parses + calls `Router::swap_config`. Transport credentials (Discord/Telegram/Email) are *not* hot-swapped — a half-applied SMTP relay change would race with in-flight `send`s — so the watcher only updates routing rules. A unit test (`reload_config_from_disk_swaps_router_rules`) covers the parse + swap path directly without spinning up the OS watcher.
- **Producer migration** — `workflow.notify` step now accepts either an explicit `channel = "..."` (override) or a `source = "..."` tag (router). A bare `notify` step with neither defaults to `source = "workflow"`. CLI agent auto-notify drops its hardcoded `"desktop"` for `source = "agent"`. CLI `nexus notify send` grows `--source` and `--severity` flags; `--channel` becomes optional and `conflicts_with = "source"`. A bare `nexus notify send "msg"` defaults to `source = "cli"` so the canonical `[sources.cli]` block in `notifications.toml` is the single configuration knob.
- **Bootstrap loader** — `load_notifications_config(forge_root)` prefers `<forge>/.forge/notifications.toml`. On miss it synthesises a `NotificationsConfig` from the legacy `[notifications.discord/.telegram/.email]` blocks AND injects default `[sources.workflow/.agent/.cli/.ai_runtime] route = ["desktop"]` rules so the renamed producers keep delivering to Desktop without any new config file. Authors opt into Discord/Telegram/email routing per-source by creating `notifications.toml`.

**IPC shape changes** (drift-checked):
- `SendArgs`: `channel` → `Option<Channel>`; new optional `source: Option<String>` + `severity: Option<Severity>`. Required fields shrink from `["channel", "message"]` to `["message"]`.
- `SendReply`: `channel` → `Option<Channel>`; new fields `channels: Vec<Channel>`, `failures: Vec<ChannelFailure>`, `routing: String`. Required field drops back to `["delivered"]`.
- New types: `Severity` (`debug`/`info`/`warn`/`error`), `ChannelFailure` (`channel` + `error`). Both ts-rs + schemars exported.

**Tests** (`cargo test -p nexus-notifications` 61/61, `-p nexus-workflow` 196/196, `-p nexus-cli` 189/189):
- `config` module: parses empty + full schema; rejects unknown top-level fields; `resolve_sources` drops unknown channel names with a warning + parses quiet_hours; quiet_hours overnight windows (22:00–08:00) match every minute they should; severity ordering matches the filter intent.
- `router` module: unknown source → `UnknownSource`; configured routes fan out; severity filters; quiet_hours filters; empty routes filter; `swap_config` swaps rules atomically; `source_names` reflects the loaded config.
- `core_plugin` BL-135 tests: override-path delivery + reply shape; missing `channel` / `source` rejected; router-path fans out; per-channel failures reported without aborting; explicit `channel` wins over `source` when both supplied; unknown source reports correct `routing` label; severity filter from the IPC layer; AiEvent translator produces notifications for `finished` / `failed` and stays silent for `token_chunk`.
- Live-reload: write `notifications.toml`, build plugin, call `reload_config_from_disk`, observe the router's new route.
- Workflow `NotifyStepArgs`: parses both explicit `channel` and `source`-only shapes; empty `channel = "   "` collapses to `None` so the dispatcher routes by source instead of erroring; `accepts_unknown_channel_at_parse_time` regression-pins the deliberate-ish-validation contract.
- CLI `notify::validate_severity`: accepts every level + rejects unknowns with a friendly message.

**Definition of done coverage**: ✅ `<forge>/.forge/notifications.toml` schema parsed in `nexus-notifications` (`[sources.<name>]` + `[channels.<name>]`); ✅ built-in router dispatches to configured transports per source; ✅ built-in `AiEvent` subscriber translates `com.nexus.ai.runtime.*` topics into notifications (dormant until BL-134 Phase 2 lights up the runtime); ✅ existing producers drop hardcoded channel selection and emit a `source` tag (workflow `notify`, CLI auto-notify, CLI `nexus notify send`); ✅ legacy explicit-channel callers keep working (override path); ✅ live-reload verified by `reload_config_from_disk_swaps_router_rules` unit test; ✅ `scripts/check_ipc_drift.sh` clean across `SendArgs` / `SendReply` / new `ChannelFailure` + `Severity` types.

**Out-of-scope / non-goals** (matches the BL-135 entry):
- No new transports (Slack / Matrix / ntfy / web-push). Deferred until someone asks.
- No persistent inbox — that's [BL-136](BL-136.md).
- File watcher only reloads routing rules, not transport credentials — changing SMTP host requires a restart. Documented in the `reload_config_from_disk` doc comment.
