### BL-010: `nexus ai chat` interactive REPL ✅ (2026-04-28)

**Delivered:** PRD-05 §3.5.2 / §10.2's multi-turn chat REPL is now wired up as a CLI consumer of the shared `com.nexus.ai::stream_chat` engine. New [`AiCommand::Chat`](../../crates/nexus-cli/src/main.rs) variant takes `--context <FILE>` / `--model <NAME>` / `--session <ID>` / `--system <PROMPT>` flags. [`commands::ai::chat`](../../crates/nexus-cli/src/commands/ai.rs) drives a [`rustyline = "14"`](../../Cargo.toml) line editor (added as a workspace dependency), dispatching each user turn through `runtime.context.ipc_call("com.nexus.ai", "stream_chat", { mode: "chat", ...messages, session_id })` while a parallel `tokio::spawn` task drains `EventFilter::CustomPrefix("com.nexus.ai.stream_")` and prints chunks to stdout. The bus contract is exercised through one shared `stream_and_collect` pump that BL-011 also uses — same drain + same session-id correlation, so the two surfaces produce byte-identical wire events. Slash commands cover `/help` / `/clear` / `/save <FILE>` / `/model <NAME>` / `/context <FILE>` / `/system <PROMPT>` / `/quit`. `/save` writes a Markdown transcript; `/model` round-trips through `config` + `set_config` (the AI plugin's `set_config` keys on the full provider record, so a "set just the model" is a read-modify-write). Exit codes match shell convention: `0` on EOF / `/quit`, `1` on any IPC error, `130` on Ctrl-C at the prompt (mapped from `ReadlineError::Interrupted` so wrapping shells see the canonical interrupted code). Coverage: `cargo test -p nexus-cli` 45 passing including 8 new tests in [`commands::ai::tests`](../../crates/nexus-cli/src/commands/ai.rs) covering `build_chat_args` round-trip + multi-byte-safe `slice_prompt` (shared with BL-011). Shipped commit `2a5b1fd`.

### BL-031: Bases cell / row clipboard ✅ (2026-04-28)

**Delivered:** Mod-C / Mod-X / Mod-V on a focused `.bases` leaf now copy a rectangular cell range (anchor + shift-click), a single selected row, or paste either of those — gated on `when: 'bases.focused && !bases.editing'` so the chords don't shadow `CellEditor` input. Each paste enrols **one** `HistoryEntry` so undo/redo round-trips the bulk operation, satisfying the BL-030 pairing requirement. New [`shell/src/plugins/nexus/bases/clipboard.ts`](../../shell/src/plugins/nexus/bases/clipboard.ts) owns the pure helpers (TSV serialization, JSON payload format, parser, per-cell coercion via `fieldTypes.ts`); new [`tableClipboard.ts`](../../shell/src/plugins/nexus/bases/tableClipboard.ts) is a registry-of-one (mirror of `activeBases.ts`) so the plugin's `cut/copy/paste` commands route through whichever table currently owns focus. Two typed clipboard MIME payloads ride alongside `text/plain` TSV: `application/x-nexus-bases-cells` for rectangular ranges and `application/x-nexus-bases-rows` for whole records — external apps still see the TSV grid, while cross-base paste preserves typing. Three paste modes land per the BL-031 plan: **range** (rectangle → tile/clip when source != range size; single-cell selection treats source as anchor + extent), **single-row** (record selected → shallow-merge by field name), and **no-selection** (typed `rows` payload or TSV-with-header → append new records via `base_record_create`). Cross-base coercion keys on `coerceValue(destKind, value, sourceKind)` — same-type values pass through unflagged, otherwise the per-cell coercion attempt either succeeds (numbers strip currency / percent symbols, text joins multi-select arrays with `, `, etc.) or drops to `null` and trips `coercedCount`. When `coercedCount > 0` the table emits a `bases:paste-coerced` notification via `api.notifications.show`. Read-only column types (`formula` / `rollup` / `lookup` / `uuid` / `relation`) are skipped at both copy and paste edges via `isPasteable`. Cell-range selection lives on a new `cellSelection: CellRange | null` field in `BasesTabState` — disjoint from the existing `selectedRecordId` (setting one clears the other) so the two selection modes don't shadow each other in the keyboard handlers. New `bases.editing` context-key is published from `BasesTable` on every `editing` state change so the `when:` clause resolves correctly. Coverage: `pnpm --filter nexus-shell test` 444 passing including 14 new BL-031 tests in [`basesStore.test.ts`](../../shell/src/plugins/nexus/bases/basesStore.test.ts) covering `normalizeRange` flip, `rangeContains` edges, `cellToTsv` / `cellsToTsv` / `rowsToTsv` / `parseTsv` round-trips, six `coerceValue` scenarios, `isPasteable`, and the disjoint-selection-modes invariant. Shipped commit `c30a237`.

### BL-019: AI local embeddings backend ✅ (2026-04-29)

**Delivered:** PRD-12 §9.1's local-embeddings escape hatch lands as a new [`crates/nexus-ai/src/local_embedding.rs`](../../crates/nexus-ai/src/local_embedding.rs) module gated behind a new `local-embeddings` Cargo feature, per ADR 0018 ([fastembed-rs vs candle vs sqlite-lembed](../adr/0018-embedding-backend.md)). Pre-BL-019 `nexus-ai`'s embedding path was strictly remote (`OpenAiProvider` / `OllamaProvider`), so a fully-offline forge could not produce vectors. **Backend** ([`local_embedding.rs`](../../crates/nexus-ai/src/local_embedding.rs)): `LocalEmbedding { inner: Mutex<TextEmbedding>, dim, cache: DashMap<u64, Vec<f32>>, cache_max_entries }` wraps fastembed-rs's `TextEmbedding`. The Mutex around `inner` is forced by fastembed v5's `&mut self` signature on `embed` (the underlying ORT session holds non-Sync state during a forward pass); the `EmbeddingProvider` trait's `&self` shape is preserved so callers don't change. **Trait + cache** (sub-task 2): `embed_with_cache(cache, cache_max, texts, fetch_fn)` is factored as a free function so the cache logic — hit / partial-miss / mismatch / over-budget eviction / disabled — can be exercised with a closure-stub `fetch_fn` that doesn't pull the BGE weights from `HuggingFace`. Cache keys are `xxhash3_64`; eviction is a coarse "drop everything when over budget" sweep (sub-LRU sophistication isn't worth the bookkeeping for a re-index that pays for itself within one run). **RAG wire-up** (sub-task 3): [`crates/nexus-ai/src/core_plugin.rs::build_embedding_provider`](../../crates/nexus-ai/src/core_plugin.rs) learns `provider == "local"` and routes to `build_local_embedding_provider(cfg)`, which is `cfg(feature = "local-embeddings")`-gated; the `not(feature)` branch returns a clear actionable error string instead of `unwrap`-ing or panicking, so a binary built without the feature is honest about it. `AiConfig` gains an optional `local_embedding_model` field consulted only when `provider == "local"` (default `bge-small-en-v1.5-int8` per ADR 0018; users override via the new `NEXUS_LOCAL_EMBEDDING_MODEL` env var). New `detect_local_embedding()` reads `NEXUS_LOCAL_EMBEDDINGS=1` (truthy: `1`/`true`/`yes`/`on`) and is preferred over remote providers in `detect_embedding_provider()` so opt-in users get the offline path automatically. The existing `vectorstore.rs` ChunkMatch shape is unchanged — local just slots into the same trait. **Batch indexer hook** (sub-task 4): public `LocalEmbedding::embed_batch(&self, &[String])` is the throughput-optimized path BL-041 will consume — batches over `BATCH_CACHE_BYPASS_THRESHOLD = 1_000` skip the per-key `DashMap` touch entirely and go straight to fastembed's bulk embed; smaller batches still pay the cache. The `EmbeddingProvider::embed` trait method delegates to `embed_batch` so callers get the same routing without knowing the threshold. **Model registry**: `map_model(name)` resolves caller-supplied identifiers (`bge-small-en-v1.5-int8` / `bge-small-en-v1.5` / `bge-base-en-v1.5` / `bge-large-en-v1.5` / `mxbai-embed-large-v1` / `nomic-embed-text-v1.5` / `all-minilm-l6-v2` and short aliases) to `fastembed::EmbeddingModel`. Empty + the canonical default both map to `BGESmallENV15Q`. Unknown ids return `AiError::Provider` so callers can fall back to remote without panicking. `model_dimension(model)` returns 384 / 768 / 1024 per family. **Cargo wiring** ([`crates/nexus-ai/Cargo.toml`](../../crates/nexus-ai/Cargo.toml)): `fastembed = { version = "5", optional = true, default-features = false, features = ["ort-load-dynamic", "hf-hub-rustls-tls"] }` + `dashmap = { version = "6", optional = true }` + `xxhash-rust = { version = "0.8", features = ["xxh3"], optional = true }`. Feature alias: `local-embeddings = ["dep:fastembed", "dep:dashmap", "dep:xxhash-rust"]`. The `ort-load-dynamic` choice (vs the ADR's hypothetical `ort-download-binaries` baseline) means cargo build doesn't reach for a network prebuilt-binary download — works in CI / sandboxed builds / air-gapped — and downstream deployments install `libonnxruntime` once via the system package manager (Debian: `libonnxruntime`; macOS: `brew install onnxruntime`) or set `ORT_DYLIB_PATH`. **Coverage**: `cargo test -p nexus-ai` 94 passing (default-feature build, unchanged from pre-BL-019 baseline) — the feature flag is correctly off in default builds. `cargo test -p nexus-ai --features local-embeddings` 102 passing (+8 net): `cache_hit_skips_fetch_call` / `partial_cache_hit_only_fetches_misses` / `cache_clears_when_over_budget` / `zero_capacity_disables_cache` / `empty_input_returns_empty_without_fetch` / `map_model_known_aliases` / `map_model_unknown_returns_provider_error` / `fetch_count_mismatch_is_provider_error`, plus one `#[ignore = "downloads ~33MB from HuggingFace on first run"]` round-trip test (`local_embedding_round_trip` — run with `cargo test -p nexus-ai --features local-embeddings -- --ignored` against a network-reachable + ORT-installed environment). `cargo clippy -p nexus-ai --all-targets -- -D warnings` clean both with and without the feature. `bash scripts/check_ipc_drift.sh` clean — `AiConfig.local_embedding_model` is internal only, not on the IPC boundary. **Out of scope**: pre-bundled BGE weights in the installer (still a first-call download from `HuggingFace`'s mirror; future installer track can pre-seed `~/.cache/fastembed/`); multilingual default (BGE-small-en is English; multilingual users set `local_embedding_model = "bge-m3"` once that lands in fastembed's enum); ADR 0008 addendum (parent task — separate doc PR). Shipped commits `5124c1b` (feature flag + skeleton), `17de317` (LocalEmbedding impl + cache + 8 unit tests), `b251814` (`provider = "local"` wire-up + config field + `detect_local_embedding`).

### BL-040: AI-DIR semantic search ✅ (2026-04-29)

**Delivered:** AI-INTEGRATION-DIRECTIONS direction 3 lands as a new `com.nexus.ai::semantic_search` IPC handler plus a default-off shell palette plugin that fans out to it alongside the existing keyword search. **Backend** ([`crates/nexus-ai/src/core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs)): new `HANDLER_SEMANTIC_SEARCH: u32 = 13` on `com.nexus.ai`, args `{ query: String, limit?: usize (default 10) }`, returns `{ matches: Vec<ChunkMatch> }`. The handler mirrors `handle_ask`'s embedder-build path (`build_embedding_provider(&embed_cfg)`) but skips the chat half entirely — the embedded query vector is forwarded to `com.nexus.storage::vector_query` (handler 18) via the existing `vectorstore::search` IPC client. The merged list is sorted descending, capped at 30, and the top hit is opened via the existing `files:open` event the keyword search sidebar already uses; an `api.notifications.show` confirms the open or reports zero results. The co-located `*.test.ts` is re-imported from [`shell/tests/semantic-search-merge.test.ts`](../../shell/tests/semantic-search-merge.test.ts) so the workspace test runner (which globs `tests/*.test.ts`) picks it up. ### BL-041: AI-DIR background indexing daemon ✅ (2026-04-29)

**Delivered:** AI-INTEGRATION-DIRECTIONS direction 8 lands as a new [`crates/nexus-ai/src/indexing_daemon.rs`](../../crates/nexus-ai/src/indexing_daemon.rs) module that subscribes to the existing storage file-watcher, debounces bursts of saves, and re-runs `rag::index_file` (the BL-019 throughput-optimised path that bypasses the cache for >1k chunks per batch) without the user touching `nexus ai index_file`. **Daemon** ([`indexing_daemon.rs`](../../crates/nexus-ai/src/indexing_daemon.rs)): `IndexingDaemon::start(ctx, status, embedder_factory)` spawns a dedicated `nexus-ai-indexing-daemon` thread that owns its own current-thread tokio runtime — sidesteps any assumption about the frontend's runtime topology and keeps `ctx.ipc_call` (async) reachable without blocking the kernel scheduler. Pure-logic `Debouncer { touched: HashSet, deleted: HashSet, last_event: Option<Instant>, debounce: Duration, max_batch: usize }` collapses rapid saves into one re-index per path per flush window (default `DEFAULT_DEBOUNCE = 2 s` / `DEFAULT_MAX_BATCH = 32`); time is injected so unit tests are deterministic. Delete supersedes touch (`a.md` deleted then touched cancels the delete; touched then deleted fires a delete). `event_to_msg` translates `com.nexus.storage.file_{created,modified,deleted,renamed}` custom events into `DaemonMsg::{Touched, Deleted}` and filters non-markdown extensions defensively (the storage watcher already does this; double-check guards future relaxations). **Worker loop**: 200 ms tick that drains both bus events (via `EventFilter::CustomPrefix("com.nexus.storage.file_")`) and a direct `mpsc::UnboundedReceiver<DaemonMsg>` (used by tests / future `index_trigger` handler), updates `pending_files` under one brief write lock, then calls `Debouncer::maybe_flush(Instant::now())`. On flush: deletions route through `com.nexus.storage::vector_delete_by_file` (handler 19); touches fetch blocks via `com.nexus.storage::query_blocks` (handler 21), `decode_blocks` re-shapes the JSON return into the `(u64, String, String, Option<i32>)` `BlockTuple` shape `rag::index_file` consumes (accepts both the raw array and `{ blocks: [...] }` wrapper for forward-compat), then `rag::index_file(ctx, embedder.as_ref(), &path, &blocks)` runs. Per-IPC timeout `IPC_TIMEOUT = 30 s`. Errors are captured into `IndexStatus.last_error` but never abort the daemon — the next batch retries; `last_error` is cleared on the next successful flush. **Status state**: `IndexStatus { indexed_files: u64, pending_files: u64, total_seen: u64, last_error: Option<String>, running: bool }` lives behind `SharedStatus = Arc<RwLock<IndexStatus>>`; cheap to clone, briefly write-locked on each event arrival and the snapshot read. **IPC handler** ([`crates/nexus-ai/src/core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs)): new `HANDLER_INDEX_STATUS: u32 = 14` on `com.nexus.ai`, args `{}`, returns `IndexStatus` verbatim. Dispatches in both sync and async paths so the shell's standard `kernel_invoke` async route works alongside CLI sync probes. **Plugin lifecycle**: `AiCorePlugin` gains `index_status: SharedStatus` (allocated unconditionally in `new()` so a fast `index_status` read returns a meaningful "never ran" view) and `indexing_daemon: Option<IndexingDaemon>` (spawned in `wire_context`, *not* `on_start`, because `on_start` runs before context wiring and the daemon needs `ctx.subscribe` + `ctx.ipc_call`). The embedder is supplied through an `EmbedderFactory = Arc<dyn Fn() -> Option<Box<dyn EmbeddingProvider>> + Send + Sync + 'static>` closure that captures the live `embed_config` handle so the daemon picks up `set_config` updates without a restart. `on_stop` signals shutdown via `Arc<AtomicBool>` and joins the worker. **Manifest** ([`crates/nexus-bootstrap/src/lib.rs`](../../crates/nexus-bootstrap/src/lib.rs)): `("index_status", HANDLER_INDEX_STATUS)` appended to the `com.nexus.ai` core manifest; `LifecycleFlags { on_init: true, on_stop: true, ..NONE }` so the kernel calls `on_stop` on graceful shutdown. **Shell badge** ([`shell/src/plugins/nexus/statusBar/IndexingStatus.tsx`](../../shell/src/plugins/nexus/statusBar/IndexingStatus.tsx)): polls `com.nexus.ai::index_status` every 2 s, renders a compact `dot + label` (green = idle, blue + glow = pending, red = error) with the indexed-files count or pending count or last-error message; latches the last-good snapshot so a transient kernel hiccup doesn't blank the badge; renders `null` when the daemon has never run (no AI / no embedder configured) so users don't see a permanently dead grey dot. The `statusBar/index.{ts → tsx}` rename is necessary to take JSX directly in the registration; registered with `priority: 20` so it slots after `FileStats`. **Coverage**: `cargo test -p nexus-ai` 118 passing (was 99 after BL-040; +19 net). New tests: 7 pure-logic `Debouncer` cases (`empty_queue_never_flushes`, `single_push_waits_for_debounce_window`, `batch_cap_forces_flush_before_debounce`, `duplicate_touches_dedupe`, `delete_supersedes_prior_touch`, `touch_after_delete_cancels_delete`, `pending_count_matches_drained_batch`); 4 `event_to_msg` translations (`storage_file_modified` → `Touched`, `storage_file_deleted` → `Deleted`, non-markdown skipped, unrelated events skipped); 3 `decode_blocks` shape tests (raw array / `{blocks}` wrapper / non-array rejection); 2 status-snapshot tests (default-when-unmodified, reflects-writes); 1 integration-style daemon test (`daemon_observes_storage_event_and_increments_total_seen`) that boots a real `KernelPluginContext`, spawns the daemon with a no-op embedder factory, publishes a `com.nexus.storage.file_modified` event onto the bus after asserting `bus.subscriber_count() >= 1`, and asserts `total_seen` advances + `running` flips on/off across `start` / `stop`; 2 dispatch-path tests on `HANDLER_INDEX_STATUS` (default snapshot returned when daemon unstarted; wire-id pin at 14). No live fastembed instantiation per the BL-040 / BL-019 test discipline. `cargo clippy -p nexus-ai --all-targets -- -D warnings` clean (required two type aliases — `EmbedderFactory` and `BlockTuple` — to satisfy `clippy::type_complexity`). `cargo test -p nexus-bootstrap` 11 passing; `cargo clippy -p nexus-bootstrap --all-targets -- -D warnings` clean. `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` no new warnings; `pnpm --filter nexus-shell test` 470 passing + 1 skipped (was 471 / 1; the badge is wired through `views.register` and exercised by the shell at runtime — no unit tests added since the polling loop is one `useEffect` and the rendering is mostly CSS). `bash scripts/check_ipc_drift.sh` not required — args/return stay inline `serde_json::Value` per the BL-041 task allowance. **Out of scope**: explicit `index_trigger` handler for "reindex now" (the design slot exists at handler id 15 but isn't wired today; the daemon already covers the watcher-driven path and the user can still call `com.nexus.ai::index_file` per-file as before — a forge-walking trigger is a small follow-up); rename-event splitting (today `file_renamed` is treated as touch-of-`to`, with a `TODO(BL-041 follow-up)` comment in `event_to_msg`; the next `file_modified` on the old path will pick up the tombstone via `vector_delete_by_file`); CLI badge surface (`nexus status` already prints AI status via `HANDLER_STATUS` — adding `index_status` to its block is a future line-item). ### BL-039: AI-DIR auto-link suggestions ✅ (2026-04-29)

**Delivered:** AI-INTEGRATION-DIRECTIONS direction 2 lands as a default-off shell-side plugin pair: a CodeMirror extension at [`shell/src/plugins/nexus/editor/cm/linkSuggest.ts`](../../shell/src/plugins/nexus/editor/cm/linkSuggest.ts) that surfaces a `[[wiki-link]]` ghost suggestion at the caret when the typed phrase semantically matches an existing forge note, plus a thin activation plugin at [`shell/src/plugins/nexus/linkSuggest/index.ts`](../../shell/src/plugins/nexus/linkSuggest/index.ts) that owns the configuration schema. **Loop**: `ViewPlugin` debounces 600 ms on each user keystroke (cancel-on-edit semantics mirror BL-034 — the `inFlightRequestId` counter is bumped on every doc change so any late resolution falls stale and is discarded). ## Suspected issues — not fully investigated

- [x] **Subscriptions persistence integrity.** `load_disabled_subscriptions` in `nexus-plugins/src/loader.rs` no longer silently swallows parse errors. On serde failure it renames the corrupt file to `subscriptions.json.corrupt-<unix-ts>` and emits an `audit = true` `tracing::warn!` naming the plugin directory, the error, and the backup path. The empty set is still returned so the plugin loads (the alternative — blocking load — is worse than briefly re-enabling subscriptions while operators investigate), but operators can now see the corruption instead of having it hidden.
- [x] **Fuel replenishment per call vs. per lifetime.** Investigated — confirmed bug. `WasmSandbox::new` calls `set_fuel` once at construction and never replenishes. A long-lived plugin with `fuel = 1_000_000` accumulates instruction usage across dispatches and eventually returns `OutOfFuel` on every call. **Fix:** `WasmSandbox` now stores `fuel_per_call` and resets it at the top of every `dispatch` via `store.set_fuel(self.fuel_per_call)`. Fuel is now semantically a per-call budget, which matches the "each handler invocation is bounded" model the manifest implies.
- [x] **Frontend XSS via unescaped plugin contribution strings.** Audited — `grep -r 'dangerouslySetInnerHTML\|innerHTML =' app/src` returns zero matches, so manifest-declared strings (`tooltip`, `label`, `title`, `description`) are all rendered as JSX text nodes, which React auto-escapes. The one plugin-controlled URL path — `WebviewPanelConfig.htmlUrl` injected as an iframe `src` — is now scheme-filtered in `makeWebviewComponent`: a new `ALLOWED_WEBVIEW_SCHEMES` set allows `http/https/tauri/blob/file` and rejects anything else (including `javascript:` / `data:` / `vbscript:`) with a visible error panel instead of silently loading. The scheme check + React's default escaping closes the plausible stored-XSS vectors without needing the full strict-CSP work (UI F-5.1.2, which is still blocker-tracked).
- [x] **MCP server trust boundary.** Audited — `nexus-mcp` exposes only an **stdio transport** (`NexusMcpServer::serve_stdio` in `crates/nexus-mcp/src/server.rs`, driven by `nexus mcp` CLI subcommand). There is no TCP listener, no network binding, no authentication layer. The trust boundary is the parent process that spawns `nexus mcp` — typically Claude Desktop or an IDE — which is the same trust boundary as the user running `nexus` on their own machine. No off-host reachability, no bypass of the plugin capability model. **Future risk:** if a WebSocket or HTTP transport is ever added for remote MCP, an auth layer must land alongside it. Documented expectation here so the decision is visible.

- [x] **Enable a strict Tauri CSP (UI F-5.1.2).** `crates/nexus-app/tauri.conf.json` `security.csp` is now a directive-keyed object: `default-src 'self'`, `script-src 'self'` (no `'unsafe-eval'`), `style-src 'self' 'unsafe-inline'` (required for CM6 + React inline styles), `connect-src 'self' ipc: http://ipc.localhost https:`, `img-src 'self' data: blob: https: asset: http://asset.localhost`, `font-src 'self' data: https:`, `frame-src 'self' blob: https: tauri:` (webview panels), plus `object-src 'none'`, `base-uri 'self'`, `form-action 'self'`. Vite HMR's `'unsafe-eval'` + `ws:` + `http://localhost:5173` live in the sibling `devCsp` block so production remains eval-free. A grep of `app/src` confirmed no `dangerouslySetInnerHTML`, `eval(`, or `new Function(` call sites, so the strict production policy should not break any existing plugin surface — React inline `style={{…}}` uses CSSOM writes (not CSP-gated) and the one CSP-relevant inline usage is CM6's dynamic `<style>` injection, which `'unsafe-inline'` on `style-src` covers. Full plugin-panel regression requires running `tauri dev` on a display, which this environment cannot do — calling out here so the next interactive run exercises the CSP.
- [x] **Drop `allow-same-origin` from webview-panel iframes (UI F-5.1.1).** `makeWebviewComponent` in `app/src/contributions/registry.ts` now emits `sandbox="allow-scripts"` only (with an optional `allow-popups`), forcing every plugin-supplied panel into a null origin so it cannot read shell cookies, `localStorage`, `indexedDB`, or reach `window.top`. Doc comments on `WebviewPanelConfig` (registry.ts) and `ctx.ui.registerWebviewPanel` (nexusContext.ts) updated to record the invariant and point panel authors at the plugin IPC surface for persistence. Serving panels from a dedicated synthetic `tauri-plugin://<id>/...` scheme is a future refinement but is not required to close the escape — the sandbox-token change alone removes the escalation path regardless of which origin the iframe `src` resolves to.

- [x] **Keybinding-conflict UI in Settings (UI F-4.1.1).** New `KeybindingConflicts` panel at the top of the Hotkeys settings tab scans the live palette-command list for chords claimed by more than one command and renders a yellow warning banner grouping every colliding entry under its chord. The first-registered command is tagged `"active"`; every other entry shows its command id so the user can scroll down to the row and rebind through the existing `KeyCaptureInput` (which already persists via `keybindings.set_keybinding_override`). The dispatcher's silent first-wins policy is unchanged — this surface just makes the normally-hidden conflict visible.

Extending the primitive set is additive — no breaking change required to add `text-input`, `divider`, `section`, etc. when plugins demand them.
- [x] **Per-plugin error boundary in the frontend (UI F-7.2.1).** New `runLifecycle` helper in `scriptRuntime.ts` wraps every `onInit` / `onStart` / `onStop` in a try/catch that records the failure on a new `usePluginStatusStore` (Zustand) keyed by plugin id; the throw is swallowed so plugins loaded later still register. The contribution bridge (`contributions/plugins.ts`) now similarly wraps per-entry registrations in `syncCommands` / `syncPanels` so a single broken contribution cannot abort the rest of the snapshot. Failures surface on the new Running Extensions settings tab (F-10.1.1) with the most recent error message.
- [x] **UI-thread time budget on plugin command dispatches (UI F-8.2.1).** Plugin-originated command dispatches now route through `runWithTelemetry` in `contributions/plugins.ts`, which races the call against a `setTimeout(…, SLOW_COMMAND_CANCEL_MS = 2_000)` macrotask and records every call's elapsed time on the plugin-status store. Dispatches over `SLOW_COMMAND_WARN_MS = 250` flip the plugin's level to `"slow"`; ones that cross the 2 s budget record a failure and return `undefined` to the caller while the in-flight promise settles silently in the background. - [x] **Menu-item ordering groups — built-in vs plugin separators (UI F-4.3.1).** `menuItemsSnapshotFn` in `contributions/registry.ts` partitions items per top-level menu into built-ins (no `plugin:` id prefix) and plugin items, sorts each group by `order`, adds `(pluginId, id)` as a stable tiebreaker on plugin items, and forces `separatorBefore: true` on the first plugin item per menu when built-ins are present. `registerMenuItem` clamps any plugin-supplied `order < 0` to `0` (with a warning) since negatives are reserved for built-ins — a plugin can no longer sort itself above the "Save" row.
- [x] **Expose `ctx.workspace` / `ctx.editor.active` APIs (UI F-6.1.1).** Added `ctx.workspace.{root,name}()` and `ctx.editorActive.{relpath,content,isDirty,open}()` on `NexusPluginContext`, wrapping the existing `useForgeStore` / `useOpenFileStore` getters. - [x] **Multi-root workspace posture decision (UI F-6.3.1).** **Decision: single-root for the 1.x line; multi-root is a 2.x API break.** Rationale: (a) Nexus' core unit is a "forge" (one knowledge base) rather than a collection of unrelated code trees, so the VS Code multi-root ergonomics matter less than Obsidian's single-vault model; (b) the contribution APIs that currently assume a single root — tree providers, file handlers, snippet file-type filters, URI handlers — would all need a `forgeRoot` parameter if multi-root landed, which is a breaking change we're not ready to absorb before 1.0; (c) switching forges (already implemented via `useForgeStore.open`) covers the "open two projects" workflow that drives most multi-root requests. When a 2.x cycle opens, revisit with a dedicated PRD that specifies which APIs need per-root dispatch vs. aggregate views.

- [x] **`performance.measure` around plugin lifecycle (UI F-10.3.1).** `scriptRuntime::runLifecycle` now brackets each `onInit` / `onStart` / `onStop` call with `performance.mark` + `performance.measure` (labels `plugin:<id>:<hook>`), so cold-start costs show up in DevTools' Performance timeline. ### Suspected issues — UI audit §6 spike candidates

- [x] **SI-2 — URI handler dispatch order.** Adopted VS Code's first-match-wins semantics. `contributions.registerUriHandler` now rejects a second registration for a scheme already claimed (warning + no-op disposable), so two plugins cannot silently fight over `nexus://…`. `dispatchUri` breaks after the first matching handler to make the invariant explicit at dispatch time too.
- [x] **SI-3 — Settings modal `clearByPlugin` on uninstall.** Investigated. **Finding: no `plugin disable` Tauri command exists yet** — the only code path that deactivates a plugin's contributions is the hot-reload event (`plugins:reloaded`), and `registerPluginContributions.syncAll` already does the right thing: it disposes every entry in `activeDisposables` before re-registering the fresh snapshot. When a disable / uninstall command is added in the future, it MUST emit `plugins:reloaded` (or a new `plugins:changed` event that funnels into `syncAll`) so the disposable chain runs. This requirement is now captured here and will be enforced by code review on the disable PR.
- [x] **SI-4 — Tree data provider caching across forge switch.** `GenericTreePanel` now subscribes to `useForgeStore(s => s.info?.root)` and includes the root in its effect deps, so switching forges re-runs `provider.getChildren(null)` and rebuilds the tree against the new root. The provider object identity stays stable (plugin-owned), but the panel's local state resets — no stale children render after a forge switch.
- [x] **SI-5 — `queueMicrotask` in CommandPalette.** Swapped `queueMicrotask` for `setTimeout(…, 0)` in `CommandPalette.commit` so dispatch is a macrotask rather than a microtask. React fully commits the palette-close render before the command runs, eliminating the one-frame overlap risk when the dispatched command opens another modal (`workspace.settings`, settings-tab openers, etc.).
- [x] **SI-7 — Snippet trigger collisions.** `contributions.registerSnippet` now scans existing snippets for a matching `trigger` (ignoring same-id re-registrations, which are already handled upstream). On collision the first registration wins; the duplicate is rejected with a console warning that names both plugin ids and the trigger, and a no-op disposable is returned. Mirrors the keybinding policy.

## Decisions — PRD-04 audit (2026-04-17)

- [x] **Service registry — implemented as the IPC command registry.** PRD-04 §8.2's "service registry for plugin-to-plugin discovery" is already satisfied by the existing IPC plumbing: every plugin's `[[registrations.ipc_command]]` entries are aggregated in `PluginLoader::loaded` and `nexus-app` exposes them to the frontend via `list_plugins` + manifest. Discovery is "scan `ui_contributions` / manifest for the target plugin and command id"; invocation is `CompositeIpcDispatcher` (which transparently routes across community → core). A typed `register_service` / `get_service<T>` layer on top would duplicate the IPC surface without adding capability or sandbox value. Decision: do **not** add a separate service registry; revisit only if a concrete plugin needs typed service contracts beyond JSON IPC.
- [x] **Plugin dependency resolution (PRD-04 §12).** `[dependencies]` map is now parsed into `PluginDependency` and validated at load time: missing deps or semver-range mismatches are rejected with `PluginError::DependencyUnsatisfied`. Topological load ordering is the caller's responsibility (bootstrap orders core plugins; community load order is discovery order today).
- [x] **Watchdog: consecutive-timeout quarantine (PRD-04 §13).** `PluginLoader` tracks a per-plugin consecutive-`ExecutionTimeout` counter; hitting `PluginManagerConfig::max_timeout_streak` (default 3) quarantines the plugin — every subsequent dispatch returns `PluginError::Quarantined` until the plugin is reloaded (hot-reload clears state automatically) or the user runs `nexus plugin reset <id>`. The persistent crash counter is bumped at quarantine time so the state survives restart.
- [x] **Extract `nexus-plugin-api` crate (F-2.1.1, F-1.3.1) — partial close, 2026-04-22.** New `crates/nexus-plugin-api` contains `Capability`, `CapabilitySet`, `CapabilityParseError`, `NexusEvent`, `EventFilter`, `EventMetadata`, `PublishedEvent`, `StopReason`, `IpcDispatcher`, `IpcFuture`, `LogLevel`, `TrustLevel`, `PluginInfo`, `PluginStatus`, `IpcError`, `BusError`, `CapabilityError`, plus `PLUGIN_API_VERSION: u32 = 1`. `nexus-kernel`'s capability/event/ipc/log/plugin modules are now shim re-exports from `nexus-plugin-api`. Plugin authors depend only on the new crate — the original audit goal ("any kernel refactor is a silent plugin-compat break") is satisfied for community plugins. `nexus-plugins/Cargo.toml` retains a path dep on `nexus-kernel` (for `EventBus`, `EventSubscription`, `KernelPluginContext`, `KvStore` — all kernel-internal infrastructure used by the loader/sandbox; never linked into community plugin .wasm). `CorePlugin` was not moved — its `wire_context(Arc<KernelPluginContext>)` hook ties it to kernel internals; staying in `nexus-plugins` is correct. Decision (2026-04-22): do **not** pursue host-side severance via dyn-trait-ifying EventBus/KvStore/PluginContext or moving the loader into `nexus-kernel` — the costs (dyn dispatch on hot paths, or inverted layering) exceed the benefit when the only external contract that matters is already insulated. Follow-up "CI guard: `nexus-plugin-api` stays kernel-free" tracked under 🟠 Orange in the open backlog as the cheap discipline that keeps this from regressing.
- [x] **CI guard: `nexus-plugin-api` stays kernel-free — follow-up to F-2.1.1, 2026-04-22.** Two `#[test]`s in `crates/nexus-plugin-api/tests/kernel_free_guard.rs`: one parses the crate's own `Cargo.toml` and asserts no forbidden workspace crate appears under `[dependencies]`, `[build-dependencies]`, or `[target.*.dependencies]`; the other walks `src/**.rs` and rejects stray `use nexus_kernel::…`-style mentions that would survive a dep-removal mid-refactor. Forbidden list: `nexus-kernel`, `nexus-plugins`, plus every other workspace crate that pulls them in transitively. Failure message points at this archive entry for rationale. Cargo's own resolver already rejects the simplest break (cyclic dep) for any current workspace member; the guard earns its keep against external crates re-exporting kernel types, future restructures that break the cycle, and stray imports added mid-refactor.
