## New Features (not addressed in any PRD)

### BL-002: Typed Property Index ✅ (2026-04-18)

**Delivered:** [`crates/nexus-storage/src/search_scope.rs`](../../crates/nexus-storage/src/search_scope.rs) `prop:` scope now consumes the typed columns (`value_num`, `value_date`, `value_bool`) populated by migration 003. `ScopeFilter::Property` gained a `PropertyOp` discriminator (`Contains` / `NumCmp(CmpOp, f64)` / `DateCmp(CmpOp, i64)` / `BoolEq(bool)`) with operator parsing inside `parse_prop_body`. Supported surface:

- `prop:priority>3`, `prop:score<=7.5`, `prop:priority=3` — numeric comparison against `value_num`.
- `prop:due<2026-01-01`, `prop:due>=2026-03-01` — YYYY-MM-DD dates parsed to unix seconds and compared against `value_date`.
- `prop:draft=true` / `prop:draft=FALSE` — boolean equality against `value_bool`.
- `prop:status:done` — legacy substring match against the raw JSON `value` column (unchanged).

Typed operators dispatch to typed columns only (`… IS NOT NULL AND value_num > ?`), so a non-numeric property never accidentally matches a numeric query. Malformed tokens (unparseable numbers/dates on `>`/`<` operators, empty keys/values) fall through to plain text. Coverage: 11 new unit tests in `search_scope.rs` — 6 parser cases (num_gt, num_lte, date_lt, bool_eq, eq_numeric, contains, unparseable-fallback) + 5 filter cases (num_gt match/exclude, date_lt match, bool_eq match/exclude, contains legacy). No PRD surface changed; strictly a `nexus-storage` library improvement consumed through the existing `com.nexus.storage::search` IPC handler.

### BL-003: Search Scoping Operators ✅ (2026-04-18)

**Delivered:** [`crates/nexus-storage/src/search_scope.rs`](../../crates/nexus-storage/src/search_scope.rs) parses four scope prefixes and post-filters Tantivy hits:

- `tag:NAME` — post-filter via SQLite `tags` table.
- `path:PREFIX` — post-filter on file-path prefix.
- `prop:KEY:VALUE` — post-filter via SQLite `properties` table (`value LIKE %VALUE%`).
- `type:BLOCK_TYPE` — post-filter on the `block_type` column stored on each `SearchResult` (no DB roundtrip).

Wired into `search()` in [`lib.rs:903`](../../crates/nexus-storage/src/lib.rs): `parse_scoped_query` splits the query into plain text + filters, Tantivy runs on the plain text, `filter_results` post-filters by SQLite lookups + block_type comparison. Coverage: 14 unit tests in `search_scope.rs` covering every prefix + malformed inputs + multi-prefix combinations, plus 4 end-to-end assertions in [`tests/prd-03-smoke.rs`](../../crates/nexus-storage/tests/prd-03-smoke.rs).

## Architecture review (2026-04-16) — microkernel adherence

- [x] **`nexus-app/src/forge.rs` routes through the kernel for file-tree ops.** All seven commands (`read_forge_file`, `write_forge_file`, `create_forge_file`, `create_forge_dir`, `rename_forge_entry`, `delete_forge_entry`, `list_forge_dir`) now call `ipc_call("com.nexus.storage", …)` via `editor::KernelRuntime`. New storage handlers `list_dir` / `create_file` / `create_dir` / `rename_entry` / `delete_entry` added (ids 27–31) on `StorageEngine` with `resolve_within` / `resolve_target` path-confinement. Writes go through the plugin's `atomic_write` (temp + fsync + rename). Coverage: `crates/nexus-bootstrap/tests/forge_ipc.rs` (7 tests).
- [x] **`nexus-editor::core_plugin` routes FS I/O through `com.nexus.storage`.** Adopted the existing `CorePlugin::wire_context` + `dispatch_async` pattern that `com.nexus.ai` already uses (no `CorePlugin::dispatch` signature change needed — punch list's option (a) assumed a trait change that turned out to be unnecessary). Editor captures its `KernelPluginContext` via `wire_context`; async `open`/`save` handlers `ipc_call("com.nexus.storage", "read_file"/"write_file", …)` so reads/writes go through storage's atomic-write + index-update + capability-check envelope. Sync dispatch kept as a unit-test-only fallback with the fsync gap fixed (`sync_all` on temp file + best-effort parent-dir fsync). Coverage: new `save_routes_through_storage_and_updates_index` test in `crates/nexus-bootstrap/tests/editor_ipc.rs` proves save updates the storage SQLite index (would stay stale if save still used `std::fs`).
- [x] **Collapsed bases SQL into `nexus-storage`; `nexus-database` is now a pure library.** `schema.rs` / `query.rs` / `relation.rs` moved from `crates/nexus-database/src/` to `crates/nexus-storage/src/bases/` — same SQL, same tests, now living alongside the only `rusqlite` owner. `nexus-database` keeps types/validation/formulas/CSV (no rusqlite dep) and is reclassified as a library crate; its no-op `DatabaseCorePlugin` (empty dispatch, never called) is deleted along with the bootstrap registration. Added `("nexus-database", "rusqlite")` to `dep_invariants::FORBIDDEN`. ARCHITECTURE.md §3/§4.2/§6.1/§7 updated to show storage as the single SQLite owner and nexus-database as a leaf library.
- [x] **`nexus-cli` links `nexus-database` directly for CSV/formulas.** Reintroduced `com.nexus.database` as a pure-logic core plugin in `crates/nexus-database/src/core_plugin.rs` exposing three handlers (`csv_import` / `csv_export` / `formula_eval`) — no SQLite, just a thin IPC wrapper around the existing CSV import/export and formula-evaluation helpers. Registered it in `nexus-bootstrap/src/lib.rs` alongside the other core plugins, added typed client helpers in `nexus-bootstrap/src/database.rs`, rewired `nexus-cli/src/commands/bases.rs` to call them via `ipc_call("com.nexus.database", …)`, dropped the direct `nexus-database` + `csv` deps from `nexus-cli/Cargo.toml`, and extended `dep_invariants::FORBIDDEN` with `("nexus-cli", "nexus-database")` + `("nexus-tui", "nexus-database")` so this can't regress. ARCHITECTURE.md §4.2 / §6.1 updated to reflect the new IPC surface. Coverage: `crates/nexus-bootstrap/tests/database_ipc.rs` (4 tests) plus the existing unit tests on the plugin itself.
- [x] **De-dup `resolve_within` path-confinement logic.** New `nexus_types::paths::resolve_within` is the single owner of forge-relative path-component validation. `nexus-storage/src/lib.rs` and `nexus-editor/src/core_plugin.rs` now delegate to it (storage maps the shared `PathError` → `StorageError::PermissionDenied`; editor adds its own `canonicalize` pass for the sync unit-test fallback). `nexus-app/src/forge.rs`'s prior copy was already gone post-storage-IPC migration, so no change there. `KernelPluginContext::confine_path` is left alone — different signature (`Path` + absolute paths + always canonicalize), not a relpath validator. Coverage: 5 new unit tests in `nexus-types::paths` covering empty/normal/parent/current-dir/absolute cases; all existing storage + editor + bootstrap IPC tests still pass.
- [x] **Dual FS watchers on the forge root.** `nexus-app/forge.rs` starts a `notify_debouncer_mini` watcher (`lib.rs` setup block) to push `forge:fs-changed` to the frontend. `StorageEngine::open_internal` also starts its own `Watcher` on the same root. Two independent OS-level watches are running simultaneously. The Tauri-side watcher is the wrong layer: `StorageCorePlugin` should emit a `NexusEvent::Custom { type_id: "storage.file.changed" }` on its internal watcher events, and the Tauri event forwarder should relay that to the frontend. Removing the `notify_debouncer_mini` dep from `nexus-app` would also guarantee the frontend sees change notifications only *after* the index has been updated (current order is undefined). **Resolved:** `notify_debouncer_mini` removed from `nexus-app`. `nexus-app/src/lib.rs` now spawns `start_storage_event_forwarder()` which subscribes to the kernel bus and forwards storage plugin events to the frontend as Tauri events. The storage plugin's internal `Watcher` is the sole OS-level observer; the frontend receives `forge:fs-changed` only after the index has been updated.
- [x] **Theme engine is now a core plugin.** New `ThemeCorePlugin` in `crates/nexus-theme/src/core_plugin.rs` owns the `ThemeEngine` state behind a `Mutex` and registers as `com.nexus.theme` with 11 IPC handlers (`get_available_themes` / `apply_theme` / `compute_variables` / `get_available_snippets` / `toggle_snippet` / `reorder_snippets` / `get_theme_config` / `set_mode` / `apply_config` / `set_plugin_overrides` / `reload`). Every mutating handler publishes a `com.nexus.theme.changed` event to the kernel bus with the updated `ThemeConfig` as payload, so plugins can subscribe via `EventFilter::CustomPrefix("com.nexus.theme.")` and react to theme-driven changes. `nexus-app/src/commands.rs` now dispatches via `ipc_call("com.nexus.theme", …)` on the kernel runtime instead of a shell-owned `EngineState`. `nexus-app/src/lib.rs` spawns a `start_theme_event_forwarder` background thread that relays bus events to the frontend as `theme:changed` Tauri events; `app/src/App.tsx` listens for those and calls a new `syncFromEngine` action on the theme store to reflect plugin-driven mutations in the UI. Coverage: 6 new unit tests in `core_plugin.rs` + 4 integration tests in `crates/nexus-bootstrap/tests/theme_ipc.rs`.
- [x] **Community plugins can now reach core plugin IPC.** Community plugins (in `PluginManager`) and core plugins (in bootstrap's `SharedPluginLoader`) stay in separate registries by design — different trust levels, different backends — but the dispatcher injected into community WASM plugins is now a new `CompositeIpcDispatcher` (`crates/nexus-plugins/src/composite.rs`) that tries the community manager first and falls through to the bootstrap loader on `IpcError::PluginNotFound`. Only `PluginNotFound` triggers fall-through; `CommandNotFound` propagates as-is so a community plugin can't shadow a core command. `PluginState` gains a write-once `core_loader: FallbackCell` populated in the Tauri `setup` closure after `build_cli_runtime` returns, which means community WASM plugins can now `host::invoke_command("com.nexus.storage", "read_file", …)` from inside their sandbox. Coverage: 5 unit tests in `composite.rs` (primary-short-circuit, fall-through, command-not-found passthrough, empty-fallback, late-install) plus 5 integration tests in `crates/nexus-bootstrap/tests/community_to_core_ipc.rs` that use an `EmptyCommunityDispatcher` as a stand-in for the real WASM community dispatcher and prove theme/storage core plugins are reachable.

- [x] **PRD-08 §4 CM6 architecture — spec needs updating to match implementation.** PRD-08 §4 specifies CM6 as the live document model with Rust providing decorations and a `BlockPositionMap`. The implementation has de facto chosen a different (pragmatic) approach: CM6 owns character-level text state in the frontend; `EditorSurface.tsx` fires `editor_sync_content` via an 800 ms debounce on `update.docChanged`, which updates the in-memory Rust block tree without disk I/O. However, PRD-08 §4 has not been formally amended to reflect this architecture, and the `BlockPositionMap` / decoration-provider model described in the PRD is not what was built. **Remaining work:** update PRD-08 §4 to document the CM6-owns-text / debounced-sync model as the canonical approach, retire the `BlockPositionMap` spec, and confirm the 800 ms debounce is acceptable for the AI-inline-edit and collaborative-editing layers.

### PRD gap — no owner for plugin-contributed tab surfaces

## Suspected issues — not fully investigated

- [x] **Subscriptions persistence integrity.** `load_disabled_subscriptions` in `nexus-plugins/src/loader.rs` no longer silently swallows parse errors. On serde failure it renames the corrupt file to `subscriptions.json.corrupt-<unix-ts>` and emits an `audit = true` `tracing::warn!` naming the plugin directory, the error, and the backup path. The empty set is still returned so the plugin loads (the alternative — blocking load — is worse than briefly re-enabling subscriptions while operators investigate), but operators can now see the corruption instead of having it hidden.
- [x] **Fuel replenishment per call vs. per lifetime.** Investigated — confirmed bug. `WasmSandbox::new` calls `set_fuel` once at construction and never replenishes. A long-lived plugin with `fuel = 1_000_000` accumulates instruction usage across dispatches and eventually returns `OutOfFuel` on every call. **Fix:** `WasmSandbox` now stores `fuel_per_call` and resets it at the top of every `dispatch` via `store.set_fuel(self.fuel_per_call)`. Fuel is now semantically a per-call budget, which matches the "each handler invocation is bounded" model the manifest implies.
- [x] **Frontend XSS via unescaped plugin contribution strings.** Audited — `grep -r 'dangerouslySetInnerHTML\|innerHTML =' app/src` returns zero matches, so manifest-declared strings (`tooltip`, `label`, `title`, `description`) are all rendered as JSX text nodes, which React auto-escapes. The one plugin-controlled URL path — `WebviewPanelConfig.htmlUrl` injected as an iframe `src` — is now scheme-filtered in `makeWebviewComponent`: a new `ALLOWED_WEBVIEW_SCHEMES` set allows `http/https/tauri/blob/file` and rejects anything else (including `javascript:` / `data:` / `vbscript:`) with a visible error panel instead of silently loading. The scheme check + React's default escaping closes the plausible stored-XSS vectors without needing the full strict-CSP work (UI F-5.1.2, which is still blocker-tracked).
- [x] **MCP server trust boundary.** Audited — `nexus-mcp` exposes only an **stdio transport** (`NexusMcpServer::serve_stdio` in `crates/nexus-mcp/src/server.rs`, driven by `nexus mcp` CLI subcommand). There is no TCP listener, no network binding, no authentication layer. The trust boundary is the parent process that spawns `nexus mcp` — typically Claude Desktop or an IDE — which is the same trust boundary as the user running `nexus` on their own machine. No off-host reachability, no bypass of the plugin capability model. **Future risk:** if a WebSocket or HTTP transport is ever added for remote MCP, an auth layer must land alongside it. Documented expectation here so the decision is visible.

The bare-function `Disposable` alias stays for direct-call plugins.
- [x] **Explicit `runtime` field in manifest `[plugin]` (UI F-3.3.1).** New `PluginRuntime` enum (`Native` / `Wasm` / `Script`) exported from `nexus-plugins`. `[plugin] runtime = "…"` is parsed from the manifest when present and must agree with the declared `[wasm]` / `[script]` section — mismatches are rejected with `ManifestInvalid`. For backwards compatibility the field is optional: when absent, the loader infers the tier from the section present so pre-F-3.3.1 manifests keep loading unchanged. Coverage: 5 new parse tests — inferred-from-wasm / inferred-from-script / explicit-runtime-matches-section / mismatch-rejected / unknown-value-rejected, plus an updated "core without wasm" test asserting `runtime == Native`.
- [x] **Implement `when`-clause evaluator or strip the field (UI F-4.1.2).** Stripped. `EditorKeybinding.when` removed from `contributions/registry.ts`; documented in `DEPRECATED.md` as a pre-1.0 removal. Plugins that need conditional activation should branch inside the dispatched command, or wait for a future when-clause evaluator PRD. Keeping the field in the API shape actively misled plugin authors (it was never parsed), so removal is the less-harmful option.
- [x] **Menu-item ordering groups — built-in vs plugin separators (UI F-4.3.1).** `menuItemsSnapshotFn` in `contributions/registry.ts` partitions items per top-level menu into built-ins (no `plugin:` id prefix) and plugin items, sorts each group by `order`, adds `(pluginId, id)` as a stable tiebreaker on plugin items, and forces `separatorBefore: true` on the first plugin item per menu when built-ins are present. `registerMenuItem` clamps any plugin-supplied `order < 0` to `0` (with a warning) since negatives are reserved for built-ins — a plugin can no longer sort itself above the "Save" row.
- [x] **Expose `ctx.workspace` / `ctx.editor.active` APIs (UI F-6.1.1).** Added `ctx.workspace.{root,name}()` and `ctx.editorActive.{relpath,content,isDirty,open}()` on `NexusPluginContext`, wrapping the existing `useForgeStore` / `useOpenFileStore` getters. Read-only for now — write paths still flow through `ctx.ipc.call("com.nexus.storage", …)` with its existing capability checks, and a future capability-gated `ctx.editorActive.applyTransaction` is gated on the iframe sandbox (UI F-8.1.1). Raw `invoke("editor_apply_transaction", …)` from script plugins is now formally discouraged; the new APIs are the blessed surface.
- [x] **Multi-root workspace posture decision (UI F-6.3.1).** **Decision: single-root for the 1.x line; multi-root is a 2.x API break.** Rationale: (a) Nexus' core unit is a "forge" (one knowledge base) rather than a collection of unrelated code trees, so the VS Code multi-root ergonomics matter less than Obsidian's single-vault model; (b) the contribution APIs that currently assume a single root — tree providers, file handlers, snippet file-type filters, URI handlers — would all need a `forgeRoot` parameter if multi-root landed, which is a breaking change we're not ready to absorb before 1.0; (c) switching forges (already implemented via `useForgeStore.open`) covers the "open two projects" workflow that drives most multi-root requests. When a 2.x cycle opens, revisit with a dedicated PRD that specifies which APIs need per-root dispatch vs. aggregate views.
- [x] **Call `onStop` on window close (UI F-7.3.1).** New `stopScriptPlugin(id)` / `stopAllScriptPlugins(budgetMs = 100)` helpers in `app/src/plugins/scriptRuntime.ts` run each loaded plugin's `onStop(ctx)` and flush its `ctx.disposables`, racing each call against a per-plugin budget so a blocked plugin cannot delay window close. `App.tsx` wires `window.addEventListener("beforeunload", …)` to invoke `stopAllScriptPlugins()`, giving every script plugin a best-effort chance to flush debounced saves or file handles before the WebView tears down.
- [x] **`performance.measure` around plugin lifecycle (UI F-10.3.1).** `scriptRuntime::runLifecycle` now brackets each `onInit` / `onStart` / `onStop` call with `performance.mark` + `performance.measure` (labels `plugin:<id>:<hook>`), so cold-start costs show up in DevTools' Performance timeline. The same helper stores the elapsed ms on `usePluginStatusStore.noteLifecycle`, which the Running Extensions tab renders as a "cold start: onInit 23ms · onStart 8ms" line per plugin — the F-3.2.1 scaling problem is now visible without spelunking the DevTools timeline.

### Suspected issues — UI audit §6 spike candidates

- [x] **SI-2 — URI handler dispatch order.** Adopted VS Code's first-match-wins semantics. `contributions.registerUriHandler` now rejects a second registration for a scheme already claimed (warning + no-op disposable), so two plugins cannot silently fight over `nexus://…`. `dispatchUri` breaks after the first matching handler to make the invariant explicit at dispatch time too.
- [x] **SI-3 — Settings modal `clearByPlugin` on uninstall.** Investigated. **Finding: no `plugin disable` Tauri command exists yet** — the only code path that deactivates a plugin's contributions is the hot-reload event (`plugins:reloaded`), and `registerPluginContributions.syncAll` already does the right thing: it disposes every entry in `activeDisposables` before re-registering the fresh snapshot. When a disable / uninstall command is added in the future, it MUST emit `plugins:reloaded` (or a new `plugins:changed` event that funnels into `syncAll`) so the disposable chain runs. This requirement is now captured here and will be enforced by code review on the disable PR.
- [x] **SI-4 — Tree data provider caching across forge switch.** `GenericTreePanel` now subscribes to `useForgeStore(s => s.info?.root)` and includes the root in its effect deps, so switching forges re-runs `provider.getChildren(null)` and rebuilds the tree against the new root. The provider object identity stays stable (plugin-owned), but the panel's local state resets — no stale children render after a forge switch.
- [x] **SI-5 — `queueMicrotask` in CommandPalette.** Swapped `queueMicrotask` for `setTimeout(…, 0)` in `CommandPalette.commit` so dispatch is a macrotask rather than a microtask. React fully commits the palette-close render before the command runs, eliminating the one-frame overlap risk when the dispatched command opens another modal (`workspace.settings`, settings-tab openers, etc.).
- [x] **SI-7 — Snippet trigger collisions.** `contributions.registerSnippet` now scans existing snippets for a matching `trigger` (ignoring same-id re-registrations, which are already handled upstream). On collision the first registration wins; the duplicate is rejected with a console warning that names both plugin ids and the trigger, and a no-op disposable is returned. Mirrors the keybinding policy.

## Decisions — PRD-04 audit (2026-04-17)

- [x] **Service registry — implemented as the IPC command registry.** PRD-04 §8.2's "service registry for plugin-to-plugin discovery" is already satisfied by the existing IPC plumbing: every plugin's `[[registrations.ipc_command]]` entries are aggregated in `PluginLoader::loaded` and `nexus-app` exposes them to the frontend via `list_plugins` + manifest. Discovery is "scan `ui_contributions` / manifest for the target plugin and command id"; invocation is `CompositeIpcDispatcher` (which transparently routes across community → core). A typed `register_service` / `get_service<T>` layer on top would duplicate the IPC surface without adding capability or sandbox value. Decision: do **not** add a separate service registry; revisit only if a concrete plugin needs typed service contracts beyond JSON IPC.
- [x] **Plugin dependency resolution (PRD-04 §12).** `[dependencies]` map is now parsed into `PluginDependency` and validated at load time: missing deps or semver-range mismatches are rejected with `PluginError::DependencyUnsatisfied`. Topological load ordering is the caller's responsibility (bootstrap orders core plugins; community load order is discovery order today).
- [x] **Watchdog: consecutive-timeout quarantine (PRD-04 §13).** `PluginLoader` tracks a per-plugin consecutive-`ExecutionTimeout` counter; hitting `PluginManagerConfig::max_timeout_streak` (default 3) quarantines the plugin — every subsequent dispatch returns `PluginError::Quarantined` until the plugin is reloaded (hot-reload clears state automatically) or the user runs `nexus plugin reset <id>`. The persistent crash counter is bumped at quarantine time so the state survives restart.
