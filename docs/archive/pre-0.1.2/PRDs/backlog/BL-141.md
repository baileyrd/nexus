# BL-141: Multibuffer / multi-excerpt view

**Status:** Fully closed. Phases 1 + 2 (Approach A + Approach B) + 3 (find-refs + rename preview + diagnostics panel + external-edit sync + content-anchored relocation) all shipped 2026-05-17.

## Phase 2 — Approach B (2026-05-17)

Per-keystroke editing on multibuffer excerpts now lands directly,
external-edit sync via `refresh_excerpts` keeps snapshots fresh
while another tab edits the source, and post-save bookkeeping
keeps each Excerpt's stored `(line_start, line_end)` coherent with
its current content. The originally-sketched approach
(per-character dispatch to the source `Session`) was replaced with
a cheaper composition: text ops mutate excerpt content directly,
source-side refresh fires off the existing `changed` bus event,
and a save-time reflow updates excerpt anchors. Same user value
without the source-block byte-offset resolution complexity.

**Step 1 — Position-mapping primitive** (`crates/nexus-editor/src/excerpt_map.rs`)

`LineIndex` + `excerpt_to_source` / `source_to_excerpt` pure
converters handle LF / CRLF terminators, multi-line excerpts,
out-of-range positions, and the line-count-mismatch case where
the snapshot has gone stale. 19 unit tests cover the matrix;
round-trip property exercises every byte offset in a multi-line
excerpt. Reserved for a future call site that needs source-byte
↔ excerpt-byte translation — not currently consumed by steps 2+
(`#[allow(dead_code)]`).

**Step 2 — Text ops on excerpts** (`crates/nexus-editor/src/core_plugin.rs`)

The synthetic-session gate in `apply_transaction` now accepts
`InsertText` / `DeleteText` on Excerpt blocks (was
`UpdateBlockContent`-only). Structural ops
(`InsertBlock` / `DeleteBlock` / `ReparentBlock` /
`UpdateAnnotations`) stay rejected — they'd corrupt the
synthetic tree's Excerpt-only invariant. Per-keystroke editing
now works inside multibuffer tabs the same as regular tabs;
save still uses `splice_excerpts` to round-trip changes into the
source file. Test 5 in `editor_excerpts_ipc` split into 5a (text
ops accepted) + 5b (structural ops rejected).

**Step 3 — `refresh_excerpts` IPC** (handler 15)

New `com.nexus.editor::refresh_excerpts { relpath }` re-reads
every source file behind a synthetic session's Excerpt blocks
and replaces each block's content snapshot in place. Block ids
stay stable so cursor anchors survive. Pure helper
`excerpt_sources(tree)` returns the unique source relpaths in
first-appearance order — used by the shell-side subscriber.
Cap matrix entry classifies as `unrestricted` (source reads
route through `com.nexus.storage::read_file`).

**Step 3b — Shell subscriber** (`shell/src/plugins/nexus/multibufferSync/`)

New `nexus.multibufferSync` plugin watches `files:open` events
for `multibuffer://<uuid>` relpaths, fetches the snapshot once,
and registers the unique source files. A single subscriber on
`com.nexus.editor.changed.` walks the registry and calls
`editor.refresh_excerpts` for every multibuffer covering the
changed file. Refresh-loop guard: skip events whose suffix is
itself a `multibuffer://` relpath. Closed-multibuffer detection:
`refresh_excerpts` errors → prune the registry entry. New
`EditorKernelClient.refreshExcerpts(relpath)` exposes the typed
IPC.

**Step 4A — Save-time range reflow**

After a synthetic session's save splices changed content into
the source, `apply_reflow_after_save` walks the session's
Excerpt blocks (grouped by source) and updates each
`(line_start, line_end)` to reflect the post-splice positions.
Pure helper `reflow_excerpt_ranges_for_source(&[(start, end,
new_count)])` does the math (sort by line_start ascending,
accumulate running delta). Closes the interaction bug between
step 2 (typing grows excerpt content) and step 3 (refresh
re-slices using stored line_end) — a save now keeps the
invariant `slice_lines(source, start, end) == block.content`
true, so the next refresh is a no-op for unchanged blocks.

**Step 4B — Content-anchored relocation**

When another tab inserts / deletes lines *before* the multibuffer's
excerpt range, the stored `(line_start, line_end)` would otherwise
point at the wrong lines after refresh. Pure helper
`relocate_excerpt_by_content(source, needle)` searches the new
source for the snapshot text as a contiguous line sequence.
**Safety:** returns `None` unless exactly one match exists —
ambiguous matches refuse to guess, falling back to the baseline
slice-and-overwrite. The user sees the same lines at their new
line numbers; cursor anchors survive.

`handle_refresh_excerpts` calls the helper before overwriting
snapshot content: if the slice at the stored range already matches
the snapshot, skip (no edit landed). If it differs and the
snapshot relocates uniquely, update the anchors and keep the
snapshot. Otherwise fall back to overwriting with the new slice.

Known limit: if the user has been typing into the excerpt
(snapshot diverged from any source content) AND the source has
been edited externally, the search will fail and the fallback
overwrites the user's typing. Conflict resolution between
in-flight typing and external source edits is out of scope —
the user's save would otherwise lose to the external edit either
way; surfacing it as a merge UI is a future concern.

Tests: 7 pure-helper tests (unique match at top, after prepend,
ambiguous → None, missing → None, empty → None, single-line
unique, longer-than-source → None) + 2 IPC integration tests
(relocate-on-prepend, fall-back-on-ambiguous).

**Tests:** 19 (excerpt_map) + 6 (reflow_tests) + 7
(relocate_tests) + 3 (excerpt_sources_tests) crate-internal + 4
(refresh_excerpts) + 1 (save-reflow) + 2 (relocate) IPC
integration + 10 (multibuffer-sync) shell pure-helper tests.
`cargo test -p nexus-editor --lib` 285 passing (was 250
baseline); `cargo test -p nexus-bootstrap --test
editor_excerpts_ipc` 19 passing; `pnpm --filter nexus-shell
test` 1685 / 1686 passing.

**NOT verified — needs a live Tauri window:**

## Phase 2 closure notes — Approach A (2026-05-16)

Read-write multibuffer shipped via the **"opaque-text MVP"** route
spelled out in the original BL split. A user can edit each Excerpt
block's snapshot as opaque text via `UpdateBlockContent`, and a
subsequent `save` splices the new content back into the source
file's line range. The lossier-but-simpler route delivers the
core UX win (editable multibuffer) without the per-character
position-mapping work Approach B requires.

**Shipped:**

- **`apply_transaction` accepts `UpdateBlockContent` on Excerpt
  blocks** in synthetic sessions. Other op types (`InsertText` /
  `DeleteText` / `InsertBlock` / `DeleteBlock` / `ReparentBlock` /
  `UpdateAnnotations`) and content edits on non-Excerpt blocks
  stay rejected with a clear "multibuffer; only UpdateBlockContent
  ops on Excerpt blocks" error referencing BL-141 Phase 2
  Approach A.
- **`save` for synthetic sessions** walks every Excerpt block,
  groups by source `relpath`, and applies splices per source via
  the new pure `splice_excerpts(old_content, splices)` helper.
  Reverse-line-order processing inside the helper keeps later
  ranges valid even when earlier splices shrink or grow line
  counts. Both the sync (`fs::read_to_string` + `atomic_write`)
  and async (`com.nexus.storage::read_file` /
  `write_file`) save paths handle synthetic sessions.
- **Defensive splice guards**: out-of-range start (excerpt past
  source EOF — happens when an external edit truncated the file
  since the multibuffer was opened) skips that splice rather than
  panicking; trailing-newline preservation matches the canonical
  `MarkdownSerializer::serialize` convention.
- **Tests**: 7 new `splice_excerpts` unit tests cover single +
  multi-range, growing + shrinking replacements, reverse-order
  processing, EOF-clamp, trailing-newline behaviour. 3 new IPC
  integration tests cover the edit→save round-trip, multi-excerpt
  same-file with line-shift handling, and the no-op save case.
  The Phase 1 read-only test was rewritten for Phase 2 semantics
  (`InsertText` still rejected; source bytes still verified
  untouched).

**Known limitations (deferred):**

- **Excerpt line ranges go stale after save.** The multibuffer's
  in-memory `line_start`/`line_end` fields don't update when a
  splice grows or shrinks the source. Reopening the multibuffer
  picks up fresh ranges. Phase 2-Approach B (bidirectional sync)
  fixes this.
- **No `InsertText` / `DeleteText` routing.** Approach A is
  "edit-textarea-then-commit"; per-keystroke text ops route to
  source-file sessions only under Approach B.
- **No external-edit sync.** While the multibuffer is open, edits
  to the source file from another tab don't update the
  multibuffer's snapshot. Approach B (or a subscriber on
  `com.nexus.editor.changed.<source_relpath>`) closes this.

### BL-141 Phase 2 — read-write multibuffer (Medium)

Today edits in an excerpt return an error. Phase 2 translates text
ops at "multibuffer position N" into character-position edits against
the corresponding source file's `Session` and propagates the result
back into the multibuffer's snapshot. Touches the transaction model
(operations need a source-file projection), the session lookup
(multibuffer session needs to acquire + dispatch to source sessions),
and `save` (saves every dirty source). Also adds bidirectional sync
so source-file edits update the multibuffer's snapshots while the
view is open.

## Original scope (preserved for context)

**Source**: Zed capability assessment — see [../research/zed-capability-assessment.md](../research/zed-capability-assessment.md). Filed 2026-05-15.
**Effort**: Medium–Large. Extends the block-tree model; touches editor open flow and LSP wiring.
**Crates**: `nexus-editor` (excerpt block type, synthetic session), `nexus-storage` (multi-file read), shell editor plugin.

Zed's multibuffer is a single editable tab containing *excerpts* (ranges) from multiple different files — useful for project-wide diagnostics (every error, editable in place), find-all-references results, and rename refactoring preview. Nexus's block-tree model is excerpt-friendly: a synthetic document composed of `ExcerptBlock` nodes, each referencing `(relpath, line_start, line_end)`, can be constructed without changing the on-disk model.

**Scope:**

- New `ExcerptBlock` variant in `nexus-editor`'s block types: `{ source_relpath, line_start, line_end, content_snapshot }`.
- New `com.nexus.editor::open_excerpts` IPC handler: accepts `[{ relpath, line_start, line_end, label? }]`, constructs a synthetic read-write buffer, returns a session ID. Edits within an excerpt dispatch through `apply_transaction` on the source file's session; saves to the source file. Read-only mode available for reference excerpts (e.g. definition peek).
- Shell editor plugin: `nexus.editor.openExcerpts(items)` — opens the synthetic buffer in a new tab with a `📄 N files` title.
- Consumers:
  - **Diagnostics panel** ("Open all in multibuffer" button) — replaces the current single-file-at-a-time diagnostics flow.
  - **Find-all-references** LSP result — open references as an excerpt view.
  - **Rename refactoring** preview — show all affected locations before confirming.

**Definition of done:**
- `open_excerpts` returns a synthetic session; reads show correct content from source files
- Edits within an excerpt round-trip through `apply_transaction` on the source file and persist on save
- "Open diagnostics in multibuffer" renders all LSP errors across the project in one tab, each error editable
- Excerpt separators show file path + line range header
- 8 unit tests: construction, multi-file reads, edit round-trip, read-only guard, empty excerpt list, overlapping ranges dedup
