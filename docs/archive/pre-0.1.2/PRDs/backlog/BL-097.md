# BL-097: IPC schema versioning rollout

**Status:** closed — 2026-05-06

**Source**: Kernel Integration Assessment (2026-05-06) — gap #6
**Files**: `crates/nexus-bootstrap/src/lib.rs`, `crates/nexus-plugins/src/loader.rs`, `crates/nexus-bootstrap/tests/ipc_versioning.rs`

The PRD's text proposed a request-envelope `schema_version: u32` field (Option B in ADR 0021). [ADR 0021](../adr/0021-ipc-handler-versioning.md) adopted Option A instead — version in the command name (`<cmd>.v<N>`) — for explicit reasons that didn't change between then and now (composes cleanly with `#[serde(deny_unknown_fields)]`; lets each `*Args` struct stay single-purpose; no internal version-branch code in handlers). BL-097's actual remaining work was the rollout the ADR §"Other subsystems" deferred: opting every subsystem into `with_v1_aliases` so every bare command name has an explicit `.v1` pin, and adding a live-registry deprecation guard.

**What shipped:**

- Every `core_manifest_with_ipc(...)` call in `nexus-bootstrap` now wraps its command list in `with_v1_aliases(&[...])`. Coverage: `com.nexus.security`, `database`, `editor`, `theme`, `ai`, `skills`, `templates`, `formats`, `workflow`, `linkpreview`, `comments`, `agent`, `mcp.host`, `git`, `terminal`. Storage was already done in ADR 0021's pilot PR.
- New `PluginLoader::list_ipc_commands() -> Vec<(plugin_id, command_name)>` accessor.
- Two new live-registry tests in `crates/nexus-bootstrap/tests/ipc_versioning.rs`:
  - `live_registry_has_v1_alias_for_every_bare_command` — for every registered bare `<cmd>`, asserts the same plugin also registered `<cmd>.v1`. Catches a future subsystem author who forgets `with_v1_aliases`.
  - `live_registry_passes_deprecation_window_guard` — runs the existing synthetic-input `deprecation_window_violations` against the live registry. Vacuous today (no `.v2` handlers exist) but acts as a forward guard so when `.v2` ships, dropping `.v1` early surfaces immediately.

**Why the PRD-text envelope field was not adopted.** ADR 0021 §"Decision" already documents this: a request-envelope `_schema_version: u8` weakens `deny_unknown_fields` (it must be `#[serde(default)]`), pushes `match version` branches into every handler, and duplicates the routing primitive that already exists at the dispatcher level. Adopting both conventions would have been actively confusing.

**Deferred from the PRD-text DoD:**

- `IpcRequest.schema_version` envelope field — see above; superseded by ADR 0021.
- `scripts/check_ipc_drift.sh` `schema_versions.json` snapshot — not added. The drift script already catches IPC schema changes via `git diff` against the generated TS bindings + JSON Schemas; a `schema_versions.json` would duplicate the signal. When `.v2` handlers actually ship, an ADR follow-up can revisit.
- `nexus-plugin-api` changelog format spec — no `.v2` handlers exist yet to bump; the convention belongs in `DEPRECATED.md` (per ADR 0021 §"Naming") when a subsystem first does this for real.
