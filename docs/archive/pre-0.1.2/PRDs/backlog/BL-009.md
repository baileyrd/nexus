# BL-009: Whole-File `.mermaid` Viewer

**Status:** closed — 2026-04-28

**Delivered:** `shell/src/plugins/community/mermaid/` extends BL-008's fenced renderer with a whole-file viewer for `.mermaid` files. The plugin's `activate()` now registers a `mermaid` view-type via `viewRegistry.register(VIEW_TYPE, creator)` and claims the `.mermaid` extension via `viewRegistry.registerExtensions(['mermaid'], 'mermaid')` (same pattern as `nexus.canvas`'s `.canvas` claim). New `MermaidPaneView` extends `ViewBase`: on `onOpen` it reads the file via `kernel.invoke('com.nexus.storage', 'read_file', { path })` (decoded via `TextDecoder('utf-8')`), caches the source so toggling modes is instant, and renders the SVG via the existing `renderMermaid()` helper that BL-008 already used. A "View Source" / "Render" toolbar button in the top-right toggles between the rendered SVG and a `<pre>` view of the raw source (caching means no re-IPC on toggle). Errors during read or render fall back to the same `buildErrorBox` BL-008 ships. Stylesheet additions in `mermaid.css` cover the pane chrome (`.nexus-mermaid-pane`, `.nexus-mermaid-toolbar`, `.nexus-mermaid-source`, `.nexus-mermaid-placeholder`); existing `.nexus-mermaid-diagram` is reused so fenced and whole-file renders look identical. Disposal: `deactivate()` runs both the fenced-renderer and view-registry disposers; `onClose()` clears the container and drops the cached source. Plugin stays default-off (`DEFAULT_OFF_PLUGINS`); users opt in via Settings → Plugins. Edit-in-place is explicitly out of scope for v1 — the spec called this out, and "View Source" is the escape hatch. Coverage: pre-existing typecheck and lint pass; no new unit tests since the surface is purely the viewRegistry hook + a thin DOM wrapper around an already-tested render function. Out of scope per the BL-009 plan: no editor handoff (open in CodeMirror via the existing extension override flow), no zoom/pan controls (mermaid SVG inherits page zoom; canvas-style pan would be a separate plugin), no auto-reload on file change (next iteration; today the user closes/reopens). Plugin docs and `plugin.json` description unchanged — informational only since the loader path goes through the shell catalog.
