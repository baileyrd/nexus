# BL-083: Forge-to-forge import and migration tool

**Status:** closed — 2026-05-06

**Source**: Storage Integration Assessment (2026-05-06) — gap #5
**Files**: `crates/nexus-storage/src/{import.rs (new),lib.rs,core_plugin.rs,Cargo.toml}`, `crates/nexus-bootstrap/src/lib.rs`, `crates/nexus-cli/src/{main.rs,commands/forge.rs}`

New `nexus-storage::import` module with two pure functions: `plan_import(src, dst) -> ImportPlan` (walks the source forge, hashes every file, classifies vs. the destination) and `apply_import(src, dst, plan, options) -> ImportReport` (executes the plan with one of three conflict strategies). Both skip the `.forge/` index subtree — the destination's index is rebuilt after apply via `StorageEngine::rebuild_index`.

Conflict strategies:

- `Skip` (default) — leave the destination unchanged, record the path in the report.
- `Overwrite` — replace destination bytes with source.
- `Rename` — write the source file to `<stem>.imported.<n>.<ext>` so both versions coexist; per-parent counter caps at 1000 to prevent pathological loops.

**Engine surface:** `StorageEngine::plan_import(source_root)` and `StorageEngine::apply_import(source_root, plan, options)`. The destination is always the engine's own forge root — no `--into` at the IPC layer (callers spin up the destination engine and call this on it).

**IPC + CLI:**

- `com.nexus.storage::import_forge` (handler 56). Args: `{source, dry_run, on_conflict}`. Returns the `ImportPlan` when `dry_run=true`, the `ImportReport` after applying, and triggers a `rebuild_index` after a successful apply so the destination index reflects the imported files immediately.
- `nexus forge import <source> [--dry-run] [--on-conflict skip|overwrite|rename]`. End-to-end smoke against three hand-built forges (one with a fresh file, one identical, one conflicting) confirmed the dry-run report shape, the `rename` strategy laid down `diff.imported.1.md` alongside the original `diff.md`, and the `skipped (identical)` count was correct.

**Tests:** seven unit tests in `crates/nexus-storage/src/import.rs` cover the classification (copies / skips_identical / conflicts), the `.forge/` exclusion, all three conflict strategies, the new-file copy path, and the missing-source error.

**Deferred from the original DoD:**

- *Progress events* on the kernel bus during the copy phase — no UI consumer yet; the synchronous JSON return is the operational shape today. Wire when a shell import wizard surfaces.
- *`--into <dest>`* at the CLI level — the engine method is destination-scoped, so a future CLI change to spin up a transient destination engine for a different forge root is additive (no IPC re-shaping needed).
