# BL-113: Protocol-host contribution model for LSP / DAP / MCP / ACP

**Status:** closed ŌĆö 2026-05-15

**Source**: BL-081 review (2026-05-13) ŌĆö full design in [ADR 0027](../adr/0027-protocol-host-contribution-model.md).
**Effort**: Large.
**Crates**: `nexus-lsp`, `nexus-dap`, `nexus-mcp`, `nexus-acp`, `nexus-plugins` (contribution loader), `nexus-bootstrap`.
**Related**: BL-081 (merged alongside on PR #163); BL-144 / BL-145 (Phase 4 ŌĆö ACP host + inbound server, shipped 2026-05-15).

Lifts adapter declarations from per-host flat TOML to a shared plugin contribution point (`contributes.protocolHosts.{lsp,dap,mcp,acp}`). Host crates stay core and protocol-only; plugins contribute adapter definitions through `[[registrations.protocol_hosts.<host>]]`. Flat TOML stays as legacy fallback for the three existing hosts during the transition; ACP lands greenfield (no `acp.toml`).

**Phase 0a shipped 2026-05-14.** ADR 0027 accepted (open questions pinned to decisions). New `[[registrations.protocol_hosts.{lsp,dap,mcp,acp}]]` manifest section: public types (`ProtocolHostsContribution`, `LspProtocolHostReg`, `DapProtocolHostReg`, `McpProtocolHostReg`, `AcpProtocolHostReg`) live in `nexus-plugins::manifest`; new `nexus-plugins::contributions` module exposes `ContributedAdapter<T>` + `collect_contributions(&[&PluginManifest])`. 10 unit tests across parsing + aggregation.

**Phases 2a + 3a shipped 2026-05-14.** Host-side merge primitives on `main`. `LspHostConfig::merge_contributed(Vec<(LspServerSpec, plugin_id)>)` and `McpHostConfig::merge_contributed(Vec<(name, McpServerSpec, plugin_id)>)` accept contributed adapter specs and merge with **TOML-wins** precedence. Both return a typed skip report (`LspMergeSkip` / `McpMergeSkip`) carrying `(name, plugin_id, reason)`. MCP factors per-spec validation out of `validate` into a shared `validate_spec` rule so contributions go through the same transport-aware checks as TOML entries. New `nexus-bootstrap::protocol_host_specs` is the only place in tree that maps `ContributedAdapter<*ProtocolHostReg>` ŌåÆ host spec; preserves contribution order; MCP transport string (`http` / `ws|websocket` / unknown ŌåÆ `stdio`) parsed defensively. 8 + 4 unit tests.

**Phases 1aŌĆō1e shipped 2026-05-15 ŌĆö on `main` via PR #163.** The DAP track end-to-end: merge primitives ŌåÆ runtime register/unregister IPC ŌåÆ bootstrap-side wiring helper ŌåÆ CLI integration on community-plugin load + per-plugin enable/disable.

- Phase 1a ŌĆö DAP merge primitives mirroring 2a/3a. `DapHostConfig::merge_contributed` with `DapMergeSkipReason::{TomlOverride, InvalidName, InvalidCommand}`, TOML-wins, preserves input order. New `dap_contribution_to_spec` / `dap_contributions_to_specs` in `nexus-bootstrap::protocol_host_specs`; shell-only fields (`display_name`, `root_markers`, `launch_config_schema`, `variable_renderers`) dropped per "host stays protocol-only". 6 unit tests.
- Phase 1b ŌĆö host-side runtime register/unregister. New `DapHostConfig.contributed_by: HashMap<name, plugin_id>` provenance tracking; new `register_contributed` / `unregister_contributed` returning `Result<DapAdapterSpec, UnregisterError::{NotFound, TomlEntry, NotOwnedByPlugin{actual_owner}}>`. New IPC handlers 20 (`register_adapter`) + 21 (`unregister_adapter`), strict-shape wire types, 4 new ts-rs bindings + 4 new JSON schemas. `DapCorePlugin.config` switched from `Option<Arc<DapHostConfig>>` to `Arc<RwLock<DapHostConfig>>`; async dispatch takes a per-future snapshot. 11 new tests.
- Phase 1c ŌĆö bootstrap-side wiring. New `nexus-bootstrap::dap_contribution_wiring::wire_dap_contributions` / `unwire_dap_contributions_for_plugin` issue IPC once per contribution. Best-effort: per-adapter outcomes don't short-circuit. `DapWireStatus` maps every host reply status plus `DispatchError` / `UnexpectedReply`. 4 module unit tests + 5 integration tests against `MinimalForge`.
- Phase 1d ŌĆö CLI integration at load time. New `App::wire_dap_contributions()` in `nexus-cli` handles the dual-borrow choreography. Called from `dispatch_external` after `plugins.load_all()`.
- Phase 1e ŌĆö runtime enable/disable hooks. New `App::wire_dap_contributions_for_plugin(plugin_id)` + `unwire_dap_contributions_for_plugin(plugin_id)`. The `nexus plugin enable / disable` CLI commands call them at the lifecycle boundary. Shared `log_dap_wire_outcomes(result, verb)` helper.

**Phases 2b + 3b shipped 2026-05-15 ŌĆö on `main` via PR #163.** The LSP and MCP tracks mirror Phase 1 layer-for-layer.

- Phase 2b ŌĆö LSP runtime register/unregister. New `LspHostConfig.contributed_by`; new `register_contributed` / `unregister_contributed`. New IPC handlers 13 + 14 on `com.nexus.lsp`, 4 new ts-rs bindings + 4 new JSON schemas. `LspCorePlugin.config` ŌåÆ `Arc<RwLock<LspHostConfig>>`. New bootstrap helper `lsp_contribution_wiring`. 5 LSP module tests + 5 bootstrap integration tests.
- Phase 3b ŌĆö MCP runtime register/unregister. Same shape, adapted for MCP's `BTreeMap<name, McpServerSpec>` keying and per-transport validator (stdio needs `command`, http/websocket need `url`). New IPC handlers 11 + 12 on `com.nexus.mcp.host`; register reply carries `reason: Option<String>` because MCP's per-transport validator emits a free-form message. `McpHostPlugin.config` ŌåÆ `Arc<RwLock<McpHostConfig>>`. New bootstrap helper `mcp_contribution_wiring` with `Invalid(String)` variant. 5 MCP module tests + 5 bootstrap integration tests.

CLI integration shared across all three protocols. `App::wire_{lsp,mcp}_contributions()` / `_for_plugin()` / `unwire_{lsp,mcp}_contributions_for_plugin()` mirror DAP. `dispatch_external` wires all three after `plugins.load_all()`; `nexus plugin enable / disable` wires/unwires per-plugin contributions in the same lifecycle order (wire AFTER enable, unwire BEFORE disable). Three log-outcome helpers in `nexus-cli::commands::plugin` emit `tracing::info` on `Ok` and `tracing::warn` otherwise; wiring failures don't block user-facing operations.

Bootstrap-side handler-id alias registration extended: `com.nexus.lsp` exports `execute_command` + the two BL-113 verbs; `com.nexus.mcp.host` exports `register_tool` / `unregister_tool` / `list_dynamic_tools` + the two BL-113 verbs.

**Phase 4 closed** ŌĆö greenfield ACP host shipped as BL-144 (outbound) and BL-145 (inbound) on 2026-05-15.

**Capability surface for `register_adapter` resolved**: no new capability at the verb level; the contribution path is the manifest-declared shape and the bootstrap wiring helper is the only intended caller. Runtime usage capabilities (`process.spawn`, `net.connect`) ride on the contributing plugin's existing grants and are checked at the `launch` / `connect` boundary. See ADR 0027 ┬¦Open Question #3. Hard enforcement at the verb level would need kernel-side caller-identity threading; filed as a hardening follow-up in BACKLOG.md.
