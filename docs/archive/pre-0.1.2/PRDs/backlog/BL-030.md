# BL-030: Bases per-surface undo / redo

**Status:** closed — 2026-04-28

**Delivered:** PRD-10's bases surface now has the per-base ring-buffered undo/redo the BACKLOG entry specified. New [`shell/src/plugins/nexus/bases/activeBases.ts`](../../shell/src/plugins/nexus/bases/activeBases.ts) mirrors the `canvas/activeCanvas.ts` registry-of-one pattern: every mounted `BasesView` registers a `BasesHandle { undo, redo, cut, copy, paste }` on focusin and clears it on focusout, and the bases plugin's command handlers dispatch to the active handle via `withActiveBases(...)`. The plugin's `index.ts` contributes two new commands (`nexus.bases.undo` / `nexus.bases.redo`) plus three keybindings (`mod-z` / `mod-shift-z` / `mod-y`) gated on `when: 'bases.focused'`, replacing the duplicate `BasesTable.tsx` keydown branch that only fired when the table itself had focus. Every base-mutating call site that bypassed the `pushHistory` ring buffer is now wrapped in paired forward/reverse `HistoryEntry`s: schema editor mutations (rename / retype / toggle-required / update-options / update-expression / delete-column / add-column), view CRUD (create / update / delete), and the non-Table view drag handlers (`BasesBoard` drop, `BasesCalendar` create-on-day). Schema retypes and column deletes snapshot per-record values for lossless undo; bases > 50,000 records refuse to push the destructive entry and surface a "history truncated" hint via a new `lastUndoError` field on `BasesTabState` rendered as a dismissable banner. Toolbar undo/redo buttons now live on `BasesView` for parity across all view modes. Coverage: shell tests pass with 10 new BL-030-specific tests in `basesStore.test.ts` (history cap, redo-stack-clear-on-push, cell-edit / add-row / soft-delete / schema-rename / schema-delete-with-cell-restore round-trips, view CRUD round-trips, focus-gated command dispatch). Per-surface undo only — global cross-surface Cmd-Z stays the explicit non-goal documented in the BACKLOG 2026-04-28 design note. Shipped commit `f42020c`.
