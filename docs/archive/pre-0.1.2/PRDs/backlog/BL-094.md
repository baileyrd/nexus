# BL-094: Audit event persistence

**Status:** closed — 2026-05-06

**Source**: Kernel Integration Assessment (2026-05-06) — gap #2
**Files**: `crates/nexus-kernel/src/{audit_store.rs (new),audit.rs,lib.rs,Cargo.toml}`, `crates/nexus-bootstrap/src/{audit_sqlite.rs (new),lib.rs,Cargo.toml}`, `crates/nexus-security/src/{ipc.rs,core_plugin.rs}`

Audit events now persist to `<forge>/.forge/.kernel/audit.db` with 90-day retention, queryable via `com.nexus.security::query_audit_log` (handler 5). Each `audit::log_*` helper appends to a global `AuditStore` in parallel with its existing `tracing::*` call — emission stays infallible (DB errors are `tracing::warn!`'d and swallowed). Architecture: kernel exposes the trait + types + `OnceLock<Arc<dyn AuditStore>>` global accessor (`install`/`append`/`query`); SQLite impl (`SqliteAuditStore`) lives in `nexus-bootstrap` since the kernel is backend-agnostic per the `dep_invariants` test that forbids `rusqlite` in `nexus-kernel`. Schema: `audit_events (id, ts_ms, event_type, plugin_id, detail_json)` with indexes on ts_ms / event_type / plugin_id; query supports optional filters with default limit 1000. Bootstrap initialises the store before kernel construction so boot-time capability grants land in the DB. 8 tests total: 2 kernel-side using a `FakeStore`, 6 in bootstrap exercising the SQLite round-trip.
