**Status:** closed — 2026-05-07

**Source**: Editor Integration Assessment (2026-05-06) — gap #1

**Related**: PRD-08 (database view spec), PRD-10 (database engine), BL-012 (DB query blocks splits 1–5 — landed previously and provided the foundation)

The original BL-069 entry described "the widget doesn't fetch or display data," which was outdated by the time the work started. The Rust handler (`HANDLER_EXECUTE_DATABASE_VIEW = 12` in `nexus-editor::core_plugin`) was already wired end-to-end through `com.nexus.storage::base_load` → `crate::database_view::config_to_view` → `com.nexus.database::apply_view`; the shell widget already fetched + cached + rendered through the BL-012 splits 1–4 work. The remaining gap was layout fidelity: every view type rendered as either a flat HTML `<table>` or a stack of grouped table-sections — no real Kanban columns, no Calendar grid, no Gallery cards, and cells were rendered with a type-blind stringify-and-truncate.

This pass closes the layout gap.

- **Type-aware cell formatter (`databaseViewFormat.ts`).** `formatCell(value, fieldDef?)` maps every `nexus_types::bases::FieldType` to a renderable string. `text` / `long-text` / `url` / `email` / `uuid` pass through. `number` / `currency` / `percent` go through `Intl.NumberFormat` with `locale` + `currency` overrides honoured from the schema; percent appends `%` *without* `style: percent`'s implicit ×100 (records store the displayable value already — 45 means 45%, not 4500%). `date` / `time` / `datetime` render in ISO form so calendar bucketing and sort comparisons stay deterministic. `checkbox` renders `✓` for `true`, empty for `false`. `select` pulls `.label` / `.name` / `.id` from object selectors; `multi-select` joins with `,`; `relation` does the same with array support. Missing field def falls back to the legacy primitive-or-stringify (capped at 200 chars) so widgets without schema metadata wired through don't regress.
- **View-type dispatch in `renderApplied`.** Switches on `applied.view_type` first (`kanban` / `calendar` / `gallery` / `table` / `list` / `timeline`), with the original `viewConfig` threaded through so layout renderers can read the variant-specific fields (`column_by`, `date_field`, `title_field`) that get erased in the apply_view projection. Falls back to the layout-shape switch when the view type is one we don't have a dedicated renderer for.
- **Kanban (`renderKanban`).** Horizontal flex of `<section.cm-md-dbview-kanban-column>` elements, one per group key. Each column's header carries the group value plus a count badge; records render as compact `<article.cm-md-dbview-card>` instances via the shared `buildCard` helper. Empty columns show an em-dash placeholder.
- **Calendar (`renderCalendar`).** 7×6 month grid anchored on the median group-key date so the visible month lands on the densest area in the data without a navigation control (deferred). Weekday headers (Sun–Sat); each cell labels the day-of-month and lists matching records as pills; cells from the previous / next month are dimmed via `--other-month`. The `MISSING_GROUP_KEY` sentinel (`"(none)"`) routes to a separate "Undated" bucket below the grid.
- **Gallery (`renderGallery`).** CSS grid of cards (`auto-fill` columns, `minmax(200px, 1fr)`). Title comes from `view_type.title_field` when set, falling back through `recordTitle`'s precedence (explicit field → first text-shaped field → `record.id`). Body shows up to 5 labeled rows of remaining fields; missing / empty values are skipped so a sparse record doesn't render padded blanks.
- **`buildCard` + `recordTitle` shared helpers.** Both Kanban and Gallery render their records through the same card builder so the title-resolution precedence stays consistent across layouts. `recordTitle` is exported for tests to pin.
- **Explicit 30 s IPC timeout.** `EditorKernelClient.executeDatabaseView` now passes `30_000` ms to `kernel.invoke` per the BL-069 DoD bullet. The kernel default is also 30 s but the spec wanted an explicit value so a future default change can't silently tighten the budget.
- **CSS.** All new classes (`cm-md-dbview-kanban` / `-column` / `-heading` / `-label` / `-count` / `-empty`; `cm-md-dbview-card` / `-title` / `-body` / `-row` / `-label` / `-value`; `cm-md-dbview-gallery`; `cm-md-dbview-calendar` / `-month` / `-grid` / `-dow` / `-cell` / `-cell--other-month` / `-day` / `-pill` / `-pill--undated` / `-undated`) styled in `livePreview.css` against the existing `--background-*` / `--divider-color` / `--interactive-accent` token palette so themes can re-skin without code changes.

**Pre-existing test rewrite.** The `grouped layout renders one section per group with record counts in the heading` test in `databaseViewWidget.test.ts` was asserting against the legacy `.cm-md-dbview-group` table-section DOM that BL-012 split 3 produced. With the view-type dispatch landing, an `applied.view_type === 'kanban'` response now drives the new Kanban renderer instead — so the test was rewritten to assert against `.cm-md-dbview-kanban-column` / `-label` / `-count` / `-card`. The legacy grouped DOM still fires, just for `view_type=list` / `view_type=timeline` (neither user-driven in this codebase yet) and as the last-resort fallback for a `grouped` layout under `view_type=table` (which `apply_view` wouldn't actually produce — Table emits `Flat`).

## BL-069follow-up: calendar navigation ✅ (2026-05-08)

**Source**: BL-069 closure note (2026-05-07) — third of the four record-mutation / UX deferrals; sibling of the kanban drag-to-reorder + inline cell editing follow-ups that landed earlier the same day
**Files**: `shell/src/plugins/nexus/editor/cm/databaseViewWidget.ts` (new exported `deriveCalendarAnchor` + `stepMonth` helpers; `renderCalendar` factored into `buildCalendarHeader` / `buildCalendarGrid` / `buildUndatedSection` and now owns mutable visible-month state with `‹` / Today / `›` controls; new optional `now: Date` arg for test determinism); `shell/src/plugins/nexus/editor/cm/databaseViewWidget.test.ts` (+7 tests covering anchor derivation, month stepping, header DOM shape, and prev / next / today / round-trip behaviour); `shell/src/plugins/nexus/editor/livePreview.css` (header becomes a flex row with prev / label / today / next children; `.cm-md-dbview-calendar-nav` button styling)
**Related**: BL-069 follow-up: inline cell editing ✅ 2026-05-08; BL-069 follow-up: kanban drag-to-reorder write-back ✅ 2026-05-08; BL-069 (DB query executor — calendar layout shipped without nav)

The original BL-069 closure note framed calendar navigation as "deferred until a user complains about the heuristic" — the median-of-data anchor lands the user on the densest area without any controls, but exposes no way to walk forward or back. This pass adds the controls without changing the default-anchor behaviour.

- **`deriveCalendarAnchor(dates, now)` and `stepMonth(year, month, delta)`.** Two pure helpers carved out of the original inline math. `deriveCalendarAnchor` returns the median `{year, month}` from the dated record list, falling back to `now` when nothing is dated (so an empty calendar still has a sensible Today). `stepMonth` walks the visible month forward / backward, wrapping the year boundary cleanly using arithmetic on `year * 12 + month` (no `Date` math, no DST surprises). Both exported for unit tests.
- **`renderCalendar` carries mutable visible-month state.** The function still returns a single `HTMLElement`, but inside the closure `currentYear` / `currentMonth` are mutable; a local `render()` rebuilds the wrapper's contents (`replaceChildren(header, grid, undated?)`) on each navigation event. The dated → group `Map` is computed once outside `render()` because it doesn't depend on the visible month — a navigation rebuild is cheap.
- **Header is now a three-button flex row.** `‹` (prev), the existing month + date_field label, "Today", `›` (next). Each button is a `<button type=button>` so it can't accidentally submit a form; click handlers `stopPropagation` so a click inside the editor decoration doesn't escape into CM6's selection logic. CSS layers a hover background and a small border-radius — they read as buttons in both ember themes without committing to a specific accent.
- **`now: Date = new Date()` is injected for testability.** Production callers pass nothing and pick up the current local date; the test suite passes deterministic dates so the Today-button assertion doesn't depend on wall-clock state. The choice to use local-shaped `new Date()` (vs. UTC) is deliberate: the user thinks of "today" in their timezone; the grid math itself stays UTC-anchored to keep `parseIsoDate` round-trips clean.
- **Undated bucket survives navigation.** Built once outside `render()`; appended on every rebuild. So a March 2026 → April 2026 step doesn't make the Inbox bucket flicker out and back in.

**Tested**: `pnpm --filter nexus-shell test` 1065/1065 (was 1058, +7); `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` 0 errors. New tests cover `deriveCalendarAnchor` (median-wins + empty-fallback-to-now), `stepMonth` (forward / backward / December→Jan year wrap / January→Dec year wrap / multi-step wrap), header DOM (prev / today / next buttons present + label still carries the date_field annotation), prev click steps back, next click advances (twice, off the data range, undated bucket unaffected), Today snaps to `now`'s month even when initial visible month is the data median, and a four-step round-trip (next/next/prev/prev) restores all four pills (3 dated + 1 undated). The pre-existing calendar grid-shape test (`grid.children.length === 49`) keeps passing because the grid layout is unchanged.

## BL-069follow-up: inline cell editing ✅ (2026-05-08)

**Source**: BL-069 closure note (2026-05-07) — second of two record-mutation deferrals; sibling of the kanban drag-to-reorder follow-up that landed earlier the same day
**Files**: `shell/src/plugins/nexus/editor/cm/databaseViewWidget.ts` (new exported `isEditableType` / `parseCellInput` / `makeEditableCell`; `renderApplied` / `renderFlat` / `renderGrouped` / `renderGallery` / `buildRow` / `buildCard` thread `MutationDeps` through to leaf renderers); `shell/src/plugins/nexus/editor/cm/databaseViewWidget.test.ts` (+10 tests covering the new helpers + DOM-level edit / commit / cancel flows + read-only fallbacks); `shell/src/plugins/nexus/editor/livePreview.css` (cell wrapper / hover highlight / editing outline / saving + error states / editor input reset)
**Related**: BL-069 follow-up: kanban drag-to-reorder write-back ✅ 2026-05-08 (same `MutationDeps` plumbing); BL-069 (DB query executor — kanban / calendar / gallery layouts), BL-012 (DB query blocks foundation)

The kanban-drag-to-reorder pass landed `MutationDeps` (`{ update(recordId, fields), refresh() }`) and `EditorKernelClient.updateBaseRecord` but only wired them to the kanban DnD path. This follow-up extends the same surface to per-cell editing across every layout that renders cells: tables (flat + grouped), kanban cards, gallery cards. The IPC + cache + refresh plumbing is unchanged — the new code is the inline-editor lifecycle and the type-aware editor controls.

- **Editable-type gate.** `isEditableType(type)` returns `true` for `text` / `long-text` / `url` / `email` / `uuid` / `number` / `currency` / `percent` / `checkbox` / `date` / `time` / `datetime` / `select`. Fields without a `def.type` (legacy schemas where `lookupFieldDef` returns `{}`), or fields whose type is `multi-select` / `relation` / `lookup` / `formula` / `rollup` route to a read-only `<span>` — the storage layer can't safely round-trip those shapes through a single text input, and a multi-select picker is its own UI scope. Read-only cells render via the existing `formatCell` path so the BL-069 type-aware formatter still drives the display.
- **`makeEditableCell` lifecycle.** The wrapper is a `<span class="cm-md-dbview-cell">` with `data-field` + `data-record-id` attributes for testing. Default state: `<span class="cm-md-dbview-cell-display">` carrying the formatted text; hover highlights via a CSS oklab-mixed background. Click swaps the display for a type-specific editor control (input / select); Enter commits, Escape restores, blur commits. A `settled` flag rejects re-entry so a blur firing right after Enter doesn't double-commit. An identity check (`parsed === value` plus a `null ↔ null` equivalence) short-circuits the IPC when the user pressed Enter on an unchanged value — no IPC, no refresh, just restore.
- **Type-aware editor controls.** `text` / `long-text` / `url` / `email` / `uuid` → `<input type=text>`. `number` / `currency` / `percent` → `<input type=number>` parsed via `Number()`; non-finite results route to a `SAME_VALUE` symbol sentinel that the caller treats as a no-op (caps user typos like "abc" → no IPC, no error toast, just restore). `date` / `time` / `datetime` → native `<input type=date|time|datetime-local>` seeded from `formatCell` (with the datetime variant rewriting the formatter's space-separated form to the `YYYY-MM-DDTHH:MM` shape the input element wants). `select` → `<select>` populated from `def.options` plus a leading "—" empty option for clearing the field; commits on `change` so single-click → pick → done.
- **Checkbox is special-cased.** No popover, no editor swap — the wrapper takes the click directly and dispatches the toggled boolean through the same `commitCellMutation` path. The `✓` glyph from `formatCell` doubles as the affordance.
- **Empty input clears the field.** `parseCellInput` collapses whitespace-only strings to JSON `null`, mirroring the kanban-drop convention: storage clears the field rather than storing an empty string.
- **Saving / error visual states.** While the IPC is in flight the wrapper gains `cm-md-dbview-cell--saving` (opacity 0.6). On failure the wrapper gains `cm-md-dbview-cell--error` (red outline) and the editor is replaced back with the pre-edit display so the user can click and retry; success calls `mutate.refresh()` which rebuilds the entire layout (the wrapper is replaced wholesale, error class included). Toast-class surfacing of the failure message lands when the BL-069 mutation UI grows past inline edits.
- **Card layouts opt in via the same threading.** `buildCard` (used by kanban + gallery) now accepts an optional `MutationDeps` and threads it into each body row — labels stay as static `<span class="cm-md-dbview-card-label">`, but the value side becomes an editable wrapper when the field type qualifies. The card's `<header class="cm-md-dbview-card-title">` stays read-only because the title is a presentation choice (first text field as fallback) rather than a single canonical field.

## BL-069follow-up: kanban drag-to-reorder write-back ✅ (2026-05-08)

**Source**: BL-069 closure note (2026-05-07) — first of two record-mutation deferrals (drag-to-reorder + inline cell editing)
**Files**: `shell/src/plugins/nexus/editor/kernelClient.ts` (new `updateBaseRecord` method routing to `com.nexus.storage::base_record_update`); `shell/src/plugins/nexus/editor/cm/databaseViewWidget.ts` (new `MutationDeps` type, exported `KANBAN_DRAG_MIME` / `MISSING_GROUP_KEY` / `KanbanDragPayload` / `readKanbanDragPayload` / `applyKanbanDrop`; `renderApplied` and `renderKanban` accept the mutate callback; `toDOM` builds a `refresh` closure that invalidates the cache and re-renders); `shell/src/plugins/nexus/editor/cm/databaseViewWidget.test.ts` (+8 tests for the new helpers + DnD-enabled / read-only DOM shape + end-to-end drop event); `shell/src/plugins/nexus/editor/livePreview.css` (drop-target ring + draggable cursor + dragging opacity)
**Related**: BL-069 (DB query executor — kanban / calendar / gallery layouts shipped without write-back), BL-012 (DB query blocks foundation)

The BL-069 closure note framed kanban drag-to-reorder as **blocked on a missing `com.nexus.database::update_record` IPC verb**. Re-checking the storage surface: the verb already exists as `com.nexus.storage::base_record_update` (handler 41) and routes end-to-end through `BaseEngine::base_record_update` to disk. The actual gap was shell-side wiring — neither the editor's database-view widget nor the layout renderers had a path to the existing IPC. This pass closes that gap for kanban; inline cell editing inherits the same wiring and can land next.

- **`EditorKernelClient.updateBaseRecord(databasePath, recordId, fields)`.** New method on the existing client (the widget already has it injected for `executeDatabaseView`, so threading through a separate bases-plugin client would be churn). Routes to `com.nexus.storage::base_record_update`. Response is typed `unknown` because the widget invalidates its cache and re-fetches via `executeDatabaseView` to get the freshest layout — the typed `BaseRecord` shape isn't needed at the call site.
- **`MutationDeps` — write-back surface threaded into renderers.** New interface `{ update(recordId, fields), refresh() }` passed as an optional fifth arg to `renderApplied` → `renderKanban`. Layouts that don't support write-back (table / list / timeline / gallery / calendar) ignore it; kanban opts in. Refresh is a thin closure built in `toDOM` over the existing fetch path: `cache.invalidate(this.key); body.replaceChildren(renderPending()); cache.run(...).then(renderApplied)`.
- **Pure helpers — `readKanbanDragPayload` + `applyKanbanDrop`.** Extracted the drop-decision logic out of the DOM event handler so the unit tests can pin every branch without standing up DragEvent / DataTransfer (happy-dom's coverage of those is partial). `readKanbanDragPayload` rejects malformed JSON and wrong-type fields with `null` so a misfired drop never panics. `applyKanbanDrop` early-returns `{ updated: false }` when the source and destination are the same column, maps the `MISSING_GROUP_KEY` ("(none)") sentinel to JSON `null` so a drop into the "Undated" bucket clears the field rather than writing the literal `"(none)"` string, and otherwise calls `mutate.update` followed by `mutate.refresh`.
- **`renderKanban` DOM event wiring.** Cards become `draggable`, get `data-record-id` + `data-source-group-key` attributes, and gain CSS classes `cm-md-dbview-card--draggable` (cursor: grab) + `cm-md-dbview-card--dragging` (opacity 0.45 while held). Columns get `dragover` (preventDefault to enable drop, only for the Nexus MIME type — wider drops bounce so a stray text drag doesn't become a record patch), `dragleave` (clears the highlight), and `drop` (calls `applyKanbanDrop` fire-and-forget; on update failure logs to the console — toast-class surfacing lands when BL-069's mutation UI grows past kanban). Drop targets get a `cm-md-dbview-kanban-column--drop-target` accent ring during the drag.
- **DnD gating.** Drag-to-reorder is only enabled when both `mutate` is provided AND a `groupField` is known (the `view_type.column_by` for kanban views, or `viewConfig.group_by` as a fallback). Without `groupField` the writer wouldn't know which key to patch; without `mutate` there's nowhere to write to. Either-missing case falls back to static cards — exactly the pre-BL-069-tail behaviour.
- **Custom MIME type (`application/x-nexus-dbview-kanban`).** Picking a Nexus-specific drag type keeps the drop handler from accepting arbitrary text drags from outside the board (a `text/plain` drag from a sibling document, say). Two kanban widgets on the same page can coexist because the payload carries the `record_id`, not a DOM reference.
