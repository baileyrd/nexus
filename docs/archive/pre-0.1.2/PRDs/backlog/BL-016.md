# BL-016: AI tool registration for LLM function-calling

**Status:** closed — 2026-04-28

**Source**: PRD-12 §8.1 (`ToolRegistry`, `ToolExecutor`, built-in `read_file` / `write_file` / etc.).
**Effort**: Medium. **Crate**: `nexus-ai`.
**Delivered:** Phase-1 keystone landed in three sequenced commits; downstream tracks (BL-010, BL-011, BL-027, BL-035, BL-036, BL-044) gain native function-calling on Anthropic, OpenAI, and Ollama through one shared registry rather than per-provider scaffolding.

- **Sub-task 1 — `ToolRegistry` + `ToolExecutor` core** _(commit `de4eb81`)_: new module [`crates/nexus-ai/src/tools/`](../../crates/nexus-ai/src/tools/) with three files. [`registry.rs`](../../crates/nexus-ai/src/tools/registry.rs) defines `ToolSchema { name, description, input_schema: serde_json::Value }` (JSON-Schema-shaped tool description), `ToolError` (`NotFound(name)` / `ExecutionFailed(msg)` / `InvalidInput(msg)`), the `#[async_trait] ToolExecutor` trait with `execute(&self, input: Value) -> Result<String, ToolError>`, `RegisteredTool { schema, executor: Arc<dyn ToolExecutor> }` (cheap-clone via Arc-shared executor), and `ToolRegistry` with `new` / `register` (overwrites by name) / `schemas` / `len` / `is_empty` / `contains` / async `execute(name, input)`. The schema is not enforced inside the registry — providers paste the raw `Value` the model produced and the executor does its own validation. [`functions.rs`](../../crates/nexus-ai/src/tools/functions.rs) ships `ReadFileTool` / `WriteFileTool` as thin wrappers over `com.nexus.storage::read_file` / `write_file`, dispatched through `KernelPluginContext::ipc_call` per CLAUDE.md invariants 2 + 3 (no direct dep on `nexus-storage`). `register_storage_builtins(&mut registry, ctx)` is the convenience seam sub-task 2's dispatch loop calls. 13 unit tests cover registry mechanics (register / overwrite / schemas / execute round-trip / `NotFound` / executor-error propagation / clone / serde) and built-in schema shape; the `dep_invariants` test still passes (no new direct dependencies into forbidden subsystems).

- **Sub-task 2 — Anthropic + OpenAI tool wire format** _(commit `91e9c0f`)_: provider trait gains `chat_turn_with_tools` plus cross-provider `ChatTurn` / `ToolCall` / `ChatTurnOutput` types in [`provider.rs`](../../crates/nexus-ai/src/provider.rs); the default impl falls back to plain text streaming so Ollama (sub-task 3) keeps compiling untouched. [`anthropic.rs`](../../crates/nexus-ai/src/anthropic.rs) serializes `tools[]` and switches to the block-array message form, parses interleaved `text` / `tool_use` content blocks, batches consecutive tool-result turns into a single user message, and tolerates unknown block types for forward-compat. [`openai.rs`](../../crates/nexus-ai/src/openai.rs) wraps tools in `{type:"function", function:{...}}` envelopes, encodes `tool_calls[].function.arguments` as a JSON string (and decodes the same field back into a `Value` on response), and prefixes tool errors with `[error]` since OpenAI lacks an `is_error` flag. `AiCorePlugin` now owns an `Arc<ToolRegistry>` populated in `wire_context` via `register_storage_builtins`, so every frontend reaches `read_file` / `write_file` through the same IPC path. The dispatch loop in [`core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs) caps at 8 rounds to bound runaway tool-use chains. Tests grew 44 → 70 covering Anthropic + OpenAI serialization, response parsing including the `arguments`-string quirk and unknown-block tolerance, and a scripted-provider end-to-end exercising the dispatch loop, error mapping, and round cap.

- **Sub-task 3 — Ollama tool-call wire format** _(commit `57d43a6`)_: replaces the default `chat_turn_with_tools` fallback for `OllamaProvider` in [`ollama.rs`](../../crates/nexus-ai/src/ollama.rs) with a native streaming tool-call implementation, so an Ollama-backed forge gets the same dispatch-loop semantics as Anthropic and OpenAI. Wire format matches Ollama's `/api/chat` tool spec — request `tools[]` uses the OpenAI-style `{type:"function", function:{...}}` envelope; assistant turns serialize `tool_calls[]` with `function.arguments` as a JSON **object** (the key wire-level distinction from OpenAI, which encodes arguments as a JSON-encoded string); tool-result turns map to `role: "tool"` messages with `tool_call_id` set; failures get an `[error] ...` prefix mirroring the OpenAI adapter; system prompts go as a synthetic `role: "system"` message because Ollama has no top-level system field. The streaming reader parses NDJSON chunks with optional `message.tool_calls`, synthesizes a stable `ollama_call_<n>` id when the model omits one (most builds do; preserves a real id when supplied), and flushes any trailing buffered line at EOF so a server that closes mid-line doesn't drop the last record. Tests grew 70 → 86: `tool_format_tests` cover schema serialization, turn → message translation, error prefixing, the arguments-as-object quirk, and request-body shaping; `streaming_dispatch_tests` drive a hand-rolled HTTP server through the real `chat_turn_with_tools` to cover id synthesis, multi-chunk text aggregation, and HTTP error propagation.

`cargo test -p nexus-ai` 86 passing across all three splits; `cargo test -p nexus-bootstrap --test dep_invariants` clean; `cargo build --workspace` clean. The Phase-2 `stream_chat` engine consumes this surface via the `tools: "auto" | "none"` arg toggle (BL-010 / BL-011 / BL-034 ship that path).

### F-8.1.2: Boundary-bound `pluginId` for sandboxed plugins ✅ (2026-04-28)

**Delivered:** Closes the second F-8.1.x red-tier item by deriving the authoritative `pluginId` from the iframe boundary instead of trusting any string the plugin (or host wiring) self-asserts. Pre-fix, [`shell/src/main.tsx`](../../shell/src/main.tsx) built one `PluginAPI` instance with `pluginId: 'community-sandbox'` and shared it across every sandboxed plugin via `SandboxOrchestratorOptions.api`. The accompanying comment claimed "each router actually enforces its own pluginId when it dispatches API calls" — but [`shell/src/host/sandbox/router.ts`](../../shell/src/host/sandbox/router.ts) `dispatchMethod` only used `this.pluginId` for error envelopes; every `this.api.storage.set(...)` / `this.api.events.emit(...)` / `this.api.activityBar.addItem(...)` call ran against the shared instance, so storage namespacing (`plugin:community-sandbox:<key>`), `eventBus.emit('activityBar:itemAdded', { ...config, pluginId: 'community-sandbox' })` tagging at [`PluginAPI.ts:422`](../../shell/src/host/PluginAPI.ts), and `PluginRegistry.track('community-sandbox', …)` ownership all collapsed concurrent sandboxed plugins into one bucket — with a corresponding cross-plugin storage leak / impersonation surface. **The fix has three pieces.** First, [`SandboxOrchestrator`](../../shell/src/host/sandbox/SandboxOrchestrator.ts) now takes `apiFactory: (pluginId: string) => PluginAPI` instead of a single shared `api: PluginAPI`. `SandboxInstanceImpl.start()` calls the factory with the orchestrator-set `pluginId` (the same id the spec passed at `orchestrator.load(spec)` time, which is always the `manifest.id` from the plugin's `plugin.json`), so every router gets its own `PluginAPI` whose `localStorage` keys, event payloads, and registry-track entries derive from the boundary id. Second, [`shell/src/main.tsx`](../../shell/src/main.tsx)'s sandbox wiring drops the false-claim comment and plugs in the factory: `apiFactory: (sandboxedPluginId) => buildPluginAPI(reg, { pluginId: sandboxedPluginId, isCore: false })`. The literal `'community-sandbox'` label is gone; nothing in the codebase still bakes that string into a real `PluginAPI`. Third, new helper [`assertValidPluginId`](../../shell/src/host/PluginAPI.ts) is the choke point — `buildPluginAPI` calls it on every construction. Empty strings, non-string inputs, and colon-bearing ids all throw before any side-effecting key is computed. The `:` ban closes the namespace-escape vector: a plugin built with `pluginId: "evil:hijack"` would otherwise share the `plugin:evil:` `localStorage` prefix with the legitimate `evil` plugin; rejecting `:` keeps each plugin pinned to its own sub-tree. Real plugin ids in this codebase are dotted (`com.nexus.editor`, `community.hello-world`) and never contain colons, so the constraint is non-breaking for shipped plugins. **Test coverage**: 9 new tests across two files. [`shell/src/host/PluginAPI.test.ts`](../../shell/src/host/PluginAPI.test.ts) (new — 7 tests) locks `assertValidPluginId` behaviour for empty / non-string / colon-bearing / valid dotted ids, asserts `buildPluginAPI` rejects the same shapes, and exercises end-to-end `localStorage` isolation between two `PluginAPI` instances built with `'plugin.a'` vs `'plugin.b'` (both writing the same logical key `'shared-key'` see distinct values; `apiA.storage.clear()` does not wipe `apiB`'s namespace). [`shell/src/host/sandbox/sandboxE2E.test.ts`](../../shell/src/host/sandbox/sandboxE2E.test.ts) gains 2 tests under a new `F-8.1.2 — pluginId binding at the sandbox boundary` section: `apiFactory` is called once per `orchestrator.load(spec)` with the spec's `pluginId` (verified via captured-call array `['plugin.a', 'plugin.b']`), and two distinct routers receive distinct `PluginAPI` instances (`assert.notEqual(apiA, apiB)`). The existing test harnesses in `orchestrator.test.ts` and `sandboxE2E.test.ts` were updated from `api: makeApi()` to `apiFactory: () => makeApi()` with no behavioural change — the per-test `makeApi` helpers stay; tests get one factory wrapper instead of a direct API. Mechanical change, not semantic. **Coverage:** `pnpm --filter nexus-shell test` 415/416 (was 406/407; +9 net new), `pnpm --filter nexus-shell typecheck` clean, `pnpm --filter nexus-shell lint` clean (sandbox / PluginAPI surfaces have zero new warnings), `pnpm --filter nexus-shell build` succeeds. **Out of scope per the F-8.1.2 plan:** legacy non-sandboxed plugins still receive a `PluginAPI` built with their `manifest.id` (the existing path through `ExtensionHost`); plugins themselves never call `buildPluginAPI` directly, so the self-assertion vector for legacy plugins is purely theoretical. The validation gate in `assertValidPluginId` is the single ingress check that catches both surfaces uniformly. With F-8.1.1 + F-8.1.1-fo1 + F-8.1.2 closed, the iframe-sandbox red-tier items from `docs/archive/planning/UI-AUDIT.md` are fully resolved.

### F-8.1.1-fo1: Precompiled sandbox runtime bundle ✅ (2026-04-28)
