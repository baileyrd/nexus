# BL-123: Slim `apply_transaction` response for text-only ops (Phase 1 of TYPING-LATENCY-PLAN)

**Status:** closed — 2026-05-14

**Source**: Typing-latency analysis — `docs/roadmap/TYPING-LATENCY-PLAN.md`
**Files**: `crates/nexus-editor/src/core_plugin.rs` (`ApplyTransactionResponse` enum, slim/full branch in `handle_apply_transaction`, doc update on `HANDLER_APPLY_TRANSACTION`, response-shape test + helpers); `shell/src/plugins/nexus/editor/kernelClient.ts` (`ApplyTransactionResponse` type, `applyTransactionRevision` / `applyTransactionSnapshot` helpers, `applyTransaction` return-type change); `shell/src/plugins/nexus/editor/kernelClient.test.ts` (slim + full test cases); `shell/src/plugins/nexus/editor/cm/transactionBridge.ts` (kind-switch on response, slim path skips setSnapshot, full path strips discriminator); `shell/src/plugins/nexus/editor/cm/transactionBridge.test.ts` (slim/full coherency tests, mock returns updated to wire shape); `experiments/perf/baselines/2026-05-14.json` (post-BL-123 baseline overwrites the BL-122 one — see numbers below)
**Related**: BL-122 (the regression gate this PR exercises), BL-124/125/126 (the remaining typing-latency fixes; this PR is the biggest single win in the sequence)

The motivating observation: BL-122's baseline showed `editor.apply_transaction.{small,medium,large}` at 39 / 330 / 24190 µs p50 on a 10 / 100 / 5000-block doc — clearly N-linear, with the snapshot serialize dominating. The webview's transaction bridge already discards the snapshot via the `skipReconcile` shortcut for text-only ops (`shell/src/plugins/nexus/editor/cm/transactionBridge.ts:362-367` pre-PR). So the kernel was doing N-linear work for a payload the bridge then threw away.

**What landed.**

- **`ApplyTransactionResponse` tagged union** in `crates/nexus-editor/src/core_plugin.rs`. Serde representation `#[serde(tag = "kind", rename_all = "snake_case")]`:
  - `Slim { revision: u64 }` for text-only ops — wire shape `{"kind": "slim", "revision": 5}`.
  - `Full(EditorSnapshot)` for structural ops — wire shape `{"kind": "full", "relpath": "...", "tree": {...}, ...}` (flattened EditorSnapshot fields next to the `kind` discriminator).
  - Helpers `revision()` (works on both variants) and `snapshot() -> Option<&EditorSnapshot>` so the existing Rust tests can pull the right thing out without matching on the enum every time.
  - `#[allow(clippy::large_enum_variant)]` because Slim is 8 bytes and Full carries an EditorSnapshot (block tree). Boxing Full would force a heap allocation for every structural op response where the snapshot's own internal allocations already dominate — wash.
- **Slim/full branch in `handle_apply_transaction`**. `text_only` flag set when every op in the transaction is `Operation::InsertText` or `Operation::DeleteText` (and the op list is non-empty — empty op lists still get the full path so callers can detect the no-op via revision parity). Recorded in the BL-122 `tracing::info_span!("apply_transaction", ...)` as a new `text_only` field, alongside the existing `op_count` / `bytes_in` / `bytes_out`. `UpdateAnnotations` stays on the full path: the bridge's optimistic mirror doesn't track annotations, so the snapshot is the only authoritative source for the post-apply annotation list.
- **`EditorKernelClient.applyTransaction` return type** flips from `Promise<EditorSnapshot>` to `Promise<ApplyTransactionResponse>`. Two new helpers in `kernelClient.ts`:
  - `applyTransactionRevision(response)` — pulls the revision out regardless of variant.
  - `applyTransactionSnapshot(response)` — extracts the inner `EditorSnapshot` for the full variant (strips the `kind` discriminator), `null` for slim.
- **`transactionBridge.dispatchTransaction` kind-switch**.
  - Slim path: read `response.revision`, push it into `editorStore.sessionRevision`, return. **No `setSnapshot?.(...)` call** — the sessionManager cache stays at its pre-tx contents. Block IDs and structure are unchanged for text-only ops, so the downstream consumers (drag-bridge, comments, block-link nav) keep correct structure. Stale text content doesn't matter because those consumers either re-read the tree on demand (`comments` falls back to `editorClient.getTree(relpath)` when the cache is missing) or use `getTree`/`stamp_block` directly.
  - Full path: destructure `{ kind: _kind, ...snapshot } = response`, push the clean `EditorSnapshot` through `setSnapshot?.(snapshot)`, then run the existing reconcile path (skip via `skipReconcile`, else `getMarkdown` + replace).

**Downstream-consumer audit.** Three callers of `sessionManager.getSnapshot(relpath)` outside the bridge itself:
1. `blockRefDragBridge.getSnapshot` — drags read block IDs from the snapshot's `tree.root_blocks` + `blocks` map. Text-only ops don't move blocks around; both IDs and `stable_id`s are unchanged. Stale snapshot is safe.
2. `nexus.editor`'s comment bridge (`editor/index.ts:1397`) — comment-on-block reads `tree.root_blocks[targetIdx]`. Same invariant: text-only ops don't change root_blocks. The handler already falls back to `editorClient.getTree(relpath)` when no cached snapshot exists, so a missing cache is also safe.
3. `EditorView.tsx:944` — passes a fresh `getSnapshot` closure into the drag bridge, not consumed directly.

The block-link nav path (`resolveBlockLink` IPC) reads the kernel-side session directly, not the shell-cached snapshot, so it never sees the stale data even on the slim path.

**Baseline shift on `editor.apply_transaction.*` (`experiments/perf/baselines/2026-05-14.json`).** Same dev WSL2 host as BL-122; same scenarios and iteration counts.

| scenario | iterations | p50 µs (BL-122) | p50 µs (BL-123) | speedup |
|----------|-----------:|----------------:|----------------:|--------:|
| `editor.apply_transaction.small` (10 blocks) | 500 | 39 | 4.01 | ≈ 10x |
| `editor.apply_transaction.medium` (100 blocks) | 300 | 331 | 4.45 | ≈ 74x |
| `editor.apply_transaction.large` (5000 blocks) | 50 | 24190 | 4.01 | ≈ 6000x |

The curve flattens to a horizontal line at ~4 µs across all three doc sizes — exactly the shape the BL predicted. The flat ~4 µs floor is the cost of the mutex acquire + undo-tree execute + the trivial `serde_json::to_value(&Slim { revision })` serialize.
