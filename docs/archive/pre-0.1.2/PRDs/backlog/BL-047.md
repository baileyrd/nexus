**Delivered:** AI-MEMORY-LAYER-PLAN piece 6 lands as the **one-off cron path** described in the BL-047 PRD — a workflow-driven cron job that walks recent capture-note markdown files, asks the AI provider to summarise them, and writes the result to a `Digests/` file. BL-028 (template engine) is the follow-up upgrade. **Library** ([`crates/nexus-workflow/src/digests.rs`](../../crates/nexus-workflow/src/digests.rs), 530 lines): new `DigestConfig { enabled, daily_cron, weekly_cron, scope_path, digests_dir }` with serde-derived defaults (`enabled = false` so users opt in; `daily_cron = "0 9 * * *"`; `weekly_cron = "0 9 * * 1"`; `digests_dir = "Digests"`). `DigestKind { Daily, Weekly }` is the IPC-arg enum. `digest_window(kind, now_utc)` is the pure window helper — daily = `[now-24h, now)`, weekly = `[now-7d, now)`. `output_path(kind, now_utc, dir)` returns `<dir>/Daily-YYYY-MM-DD.md` (chrono `%Y-%m-%d`) or `<dir>/Weekly-YYYY-Www.md` using ISO-week numbering via `now_utc.iso_week()` so the 2027-01-01 boundary correctly emits `Weekly-2026-W53.md`. `build_digest_prompt(kind, sources)` is the pure prompt builder — emits a tight instruction ("produce a concise markdown digest with overview / themes / actions"), a `# Sources (N file(s))` header, and one fenced markdown block per source with the path as the section heading. **Orchestrator** (`pub async fn run_digest(ctx, cfg, kind, now_utc) -> Result<DigestRunReport, PluginError>`): walks the configured scope (forge root when `scope_path` is `None`) recursively via `com.nexus.storage::list_dir` (HANDLER_LIST_DIR = 27), filters markdown files whose `modifiedMs` falls inside the digest window, reads bodies via `com.nexus.storage::read_file` (HANDLER_READ_FILE = 2), skips anything under `digests_dir/` to avoid feedback loops, builds the prompt, calls `com.nexus.ai::ask` (HANDLER_ASK = 1), ensures the digest directory exists via `com.nexus.storage::create_dir` (HANDLER_CREATE_DIR = 29; pre-existing-dir errors are swallowed), then writes the digest body via `com.nexus.storage::write_file` (HANDLER_WRITE_FILE = 8). The output file gets YAML frontmatter (`digest_kind`, `generated_at`, `source_count`, `sources` list) so downstream indexing sees it as a normal markdown note. Empty-window short-circuits return `DigestRunReport { written: false, source_count: 0, model: None, … }` without calling the AI provider — saves a round trip and avoids asking the model to summarise nothing. The recursive walker uses `Box::pin` to bound the async future size. All filesystem and AI access flows through kernel IPC; the workflow crate never touches the filesystem or AI provider directly. **IPC handler** ([`crates/nexus-workflow/src/core_plugin.rs`](../../crates/nexus-workflow/src/core_plugin.rs)): new `HANDLER_RUN_DIGEST: u32 = 6` on `com.nexus.workflow`, args `{ kind: "daily"|"weekly" }`, returns `DigestRunReport`. Wired through `dispatch_async` (the sync `dispatch` arm returns the same "use dispatch_async" error pattern the existing HANDLER_RUN uses). The plugin gains a `digest_config: DigestConfig` field and a new `WorkflowCorePlugin::open_with_digest_config(workflows_dir, cfg)` constructor; the legacy `open(workflows_dir)` delegates with `DigestConfig::default()` for backwards compat. **Cron loop** (`spawn_digest_scheduler` + `digest_scheduler_loop`): spawned from `wire_context` (mirrors the existing `spawn_cron_schedulers` and `spawn_file_event_triggers` patterns; on_start runs before context wiring, so the digest loop cannot use it). Disabled when `DigestConfig.enabled == false` — the loop never spawns and `tracing::debug!`s the reason. When enabled, the loop wakes every 60s (capped sleep), computes the next fire across both schedules via `digests::next_fire`, and dispatches `com.nexus.workflow::run_digest` via `ctx.ipc_call` so semantics match the manual-trigger path. Sleeps 61s past each fire so a one-minute boundary doesn't fire twice. The loop's `JoinHandle` is parked in the existing `scheduler_handles` Vec and aborted on plugin Drop alongside the cron and file-event handles. **Bootstrap manifest** ([`crates/nexus-bootstrap/src/lib.rs`](../../crates/nexus-bootstrap/src/lib.rs) `com.nexus.workflow` block): `("run_digest", HANDLER_RUN_DIGEST)` registered alongside the existing `list` / `get` / `reload` / `validate` / `run` entries; CLI / TUI / MCP / shell all reach it through `context.ipc_call(...)` with no per-frontend opt-in. New `load_digest_config(forge_root)` helper reads `<forge>/.forge/config.toml` and deserialises the `[digests]` block via `toml::from_str` (toml moved from dev-dep to runtime dep on `nexus-bootstrap`); missing file / missing block / parse failure all fall back to `DigestConfig::default()` so existing forges work with no config changes — a parse failure logs at warn but never blocks boot. **Coverage**: `cargo test -p nexus-workflow` 78 passing (was 65 pre-BL-047; +13 net): 11 pure-logic tests in `digests::tests` covering defaults (off + known crons), daily window = last 24h, weekly window = last 7d, daily output path with ISO date, weekly output path with ISO week, weekly output path at the year-53 boundary (2027-01-01 → `Weekly-2026-W53.md`), trailing-slash strip on `digests_dir`, prompt includes sources + paths + count, weekly prompt label, `next_fire` picks the earliest schedule (Sunday morning daily wins over Monday morning weekly), and `DigestKind::from_str` round-trip + reject. 2 stub-IPC integration tests (`run_digest_walks_files_calls_ai_and_writes` — three fixtures with mixed mtimes + extensions; only the in-window `.md` makes it through; AI is called once; `write_file` fires; report carries the model + path; `run_digest_short_circuits_when_no_files_in_window` — stale-only fixture; AI is NOT called; report shows `written: false`). The stub `IpcDispatcher` mirrors the BL-041 `indexing_daemon::tests` pattern — records every call and returns canned responses for the storage / AI commands the pipeline issues. `cargo clippy -p nexus-workflow --all-targets -- -D warnings` clean (required four module-scoped allow lints in `digests.rs`: `unnecessary_wraps` for the `Option<String>` config defaults that are part of the wire shape, `should_implement_trait` for `DigestKind::from_str` which mirrors the JSON arg name not `std::str::FromStr`, `format_push_string` for hot-path string building dwarfed by the IPC round-trips, and `cast_possible_truncation` for the `u64`-to-`u8` byte coercion in `read_file` where JSON numbers are constructively `0..=255`). `cargo test -p nexus-bootstrap` 11 passing; `cargo clippy -p nexus-bootstrap --all-targets -- -D warnings` clean. No IPC drift required — args/return stay inline `serde_json::Value`. **Scheduling-thread design choice**: spawned from `wire_context` rather than `on_start` because `on_start` runs before the kernel context is attached, and the loop needs `ctx.ipc_call` to dispatch through `com.nexus.workflow::run_digest` (matches the existing `spawn_cron_schedulers` rationale documented in `core_plugin.rs`'s module docstring). The 60s wake interval is intentionally simple — a precise sleep-until-next-fire would shave milliseconds at the cost of clock-skew brittleness; the current design re-evaluates every minute so a config change or system suspend resync naturally. **Known limitations**: (1) `DigestConfig` is loaded once at bootstrap and never re-read — toggling `enabled` or changing crons requires restarting Nexus; a follow-up could subscribe the loop to a config-change event (the workflow plugin already has a `reload` handler the digest path could mirror). (2) The walker scans the whole forge subtree on every fire — a large forge with thousands of markdown files pays the IPC cost even when only a handful are in-window; a future optimisation could push the mtime filter into a single storage IPC like `list_recent` or use the existing search index. (3) Capture-notes scope is implicit (any `.md` in `scope_path`) rather than tag-based or BL-043 `memory.inboxPath`-aware — the BACKLOG description called this out as the pragmatic v1 choice; BL-028 templates will let users author per-digest source filters declaratively. (4) The output file format is hard-coded — frontmatter shape, prompt wording, and section structure are all baked into the Rust code; BL-028 will move these into user-editable templates. (5) The AI call uses `ask` (RAG) which adds embedding cost to the digest pipeline even though the prompt already carries the source bodies — `stream_chat` would be cheaper but pulling in the chat-message shape from the workflow crate is out of scope for v1; the AI provider is free to ignore the empty RAG side and just summarise the prompt. (6) The cron loop has no per-fire de-duplication beyond the 61s post-fire sleep — if the system clock jumps backwards (NTP correction) the same minute boundary could re-fire; a fired-at watermark in KV would close that gap if it ever bites.
