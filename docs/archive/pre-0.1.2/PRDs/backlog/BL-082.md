# BL-082: Symlink handling in watcher and reconcile

**Status:** closed — 2026-05-06

**Source**: Storage Integration Assessment (2026-05-06) — gap #4
**Files**: `crates/nexus-storage/src/{reconcile.rs,watcher.rs}`

`reconcile::scan_dir_recursive` now consults `entry.file_type()` (which doesn't follow symlinks) instead of `Path::is_dir` / `is_file` (which do). Any entry whose type is `is_symlink()` is logged at info-level and skipped — neither recursed into (so a symlink pointing at a directory inside the forge can't double-index its contents) nor read as a file (so an external-target symlink can't follow the link out of the sandbox).

`watcher.rs`'s event loop adds the same gate: any debounced event whose path's `symlink_metadata().file_type().is_symlink()` is true is dropped at debug level. Reconcile already excludes symlinks from the walk; the live event stream now stays consistent with that.

**End-to-end smoke** against a forge with `note.md` + a `ln -s note.md alias.md` symlink: `nexus forge reindex` reports `Files: 1` (only `note.md` indexed, alias correctly skipped). Two new unit tests in `reconcile::tests` (gated to `cfg(unix)` since Windows symlinks need elevation) cover the intra-forge and external-target cases.

**Deferred from the original DoD:**

- `file_type = "symlink"` tagging in the `files` table — the simpler "skip entirely" invariant matches the no-double-index goal without a schema change. Restore if a future use case wants to surface "you have N un-indexed symlinks" in the UI.
- `query_files(include_symlinks: bool)` — moot under the skip-entirely policy (no symlinks in the table to filter).
- Canonical-path dedup for "symlinks within the forge root, follow + dedup". The shipped policy is "skip all symlinks regardless of target", which is stricter than the PRD text but provides the same no-double-index guarantee with simpler logic. Operators who need symlinks in the index can replace them with hard links (no policy change needed).
