# BL-049: NB block-links navigator (`[[…#^block-id]]`)

**Status:** closed — 2026-04-30

**Source**: [../notion-block-ux-plan.md](../notion-block-ux-plan.md) out-of-scope follow-up.
**Effort**: Medium. **Crate / Package**: `nexus-editor` (resolver) + `shell/src/plugins/nexus/editor/` (navigation UX) + `nexus-storage` (backlinks-to-block helper) + `shell/src/plugins/nexus/backlinks/` (filter UI).
Click-through and back-link surfacing for block-anchored links. **Hard-gated on the block-id stability ADR** — without stable cross-session ids, block links rot on every edit downstream of an insert.

- **BL-049 phase 4 — `backlinks_to_block` IPC + shell filter UI** _(shipped 2026-04-30)_: closes BL-049 by giving the `backlinks_to_block` Rust helper an IPC surface and a user-facing filter chip. **Rust** ([`crates/nexus-storage/src/core_plugin.rs`](../../crates/nexus-storage/src/core_plugin.rs)): new `HANDLER_BACKLINKS_TO_BLOCK = 55` (`com.nexus.storage::backlinks_to_block`) takes `{ path, block_id }` and returns `Vec<BacklinkResult>` filtered to inbound links whose fragment is `^<block_id>` (case-insensitive). Engine adds `StorageEngine::backlinks_to_block` as a thin delegate so the locking discipline matches the existing `backlinks` method. Bootstrap manifest entry registered alongside `backlinks` so CLI / TUI / shell all reach the new handler uniformly. Two cargo dispatch tests cover (a) missing-`block_id` rejection and (b) empty-result for an unknown path so empty-state rendering on the shell side stays simple. **Shell** ([`shell/src/plugins/nexus/backlinks/`](../../shell/src/plugins/nexus/backlinks/)): the store gains a `blockFilter: string \| null` field with `setBlockFilter` action; the plugin's `load` and `refresh` paths route through `backlinks_to_block` when the filter is set and through the unfiltered `backlinks` handler otherwise. Switching active files clears the filter (a UUID stamped in file A is meaningless against file B), and a new store-subscription in `index.ts` re-issues `load` whenever the filter toggles so the panel re-queries the kernel. UX entry-point: `FragmentPill` is now clickable for block anchors — clicking sets the store's `blockFilter`, which the header surfaces as a separate `ActiveBlockFilterChip` carrying an `×` to clear. Heading-anchor pills stay non-interactive (no matching IPC surface yet). The pill click stops propagation so the surrounding "open file" handler doesn't fire. Coverage: 2 new cargo dispatch tests (missing-arg + empty-result) and 4 new node:test cases pinning the store's filter behaviour (default null, round-trip set/clear, clear() resets alongside the rest of the state, subscribe sees the prev/next transition pair). End-to-end "click pill → filtered list" path is exercised through the existing store + IPC contracts; the React render side is structural and not unit-tested.

- **BL-049 phase 3 — backlinks fragment surfacing + block-anchored filter** _(shipped 2026-04-30)_: The kernel knowledge graph already stored every link's `fragment` field on `EdgeData` (heading slug or `^<block-id>`); BL-049 phase 1's resolver and phase 2's click navigator both use it. Phase 3 propagates it through the IPC surface so the shell's backlinks panel can show "this incoming link points at *which* block / heading?" without an extra round trip. **Rust** ([`crates/nexus-storage/src/graph.rs`](../../crates/nexus-storage/src/graph.rs)): `BacklinkResult` gains an optional `fragment: Option<String>` field with `#[serde(default, skip_serializing_if = "Option::is_none")]` so older snapshots stay JSON-compatible — pre-phase-3 callers see the same wire shape, post-phase-3 callers can rely on the field. `KnowledgeGraph::backlinks(path)` now copies the fragment from the matching `EdgeData`. New `KnowledgeGraph::backlinks_to_block(path, block_id)` filters down to inbound links whose fragment is `^<block_id>` (case-insensitive, since on-disk markers are lowercase via `format_stable_id_marker` but a hand-typed link could be upper); the helper backs a future "show only backlinks to *this* block" filter chip in the panel. **Shell** ([`shell/src/plugins/nexus/backlinks/`](../../shell/src/plugins/nexus/backlinks/)): the `KernelBacklink` decoder picks up the new `fragment` key, normalises non-string / empty values to `null`, and the `Backlink` interface gains `fragment: string \| null`. The view renders a compact pill below the source name when the fragment is present — `^<8-char-prefix>…` for block anchors (so the pill stays legible — full id remains available via tooltip) and `#<heading>` for heading anchors. Pure `FragmentPill` component is exported alongside `BacklinksView` so future per-block filter chips can reuse the styling. Coverage: 3 new cargo tests on the graph (`backlinks_carries_fragment_field` — round-trip through the result struct; `backlinks_to_block_filters_by_uuid_case_insensitive` — three-edge scenario covering uppercase fragment match + heading-fragment exclusion + unknown-id empty result), and 5 new node:test cases on the shell decoder (`fragment` round-trip, BL-043 backwards-compat null mapping, self-reference filtering still drops irrespective of fragment, defensive non-string / empty-string rejection, mixed block + heading fragments preserved). `cargo test -p nexus-storage --lib graph` 23 passing (was 21; +2 since the third test reuses fixtures); `cargo clippy --workspace --all-targets -- -D warnings` clean; `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` 0 new warnings; `pnpm --filter nexus-shell test` 677 passing (was 672; +5).

- **BL-049 phase 2 — CM6 click navigation + reveal-block scroll** _(shipped 2026-04-30)_: closes the click-through loop. New CM extension [`shell/src/plugins/nexus/editor/cm/blockLinkNav.ts`](../../shell/src/plugins/nexus/editor/cm/blockLinkNav.ts) provides a `StateField`-backed `Decoration.mark` source that underlines every parsed `[[<file>#^<uuid>]]` (rebuilds on `docChanged` only — selection moves don't re-decorate), plus a `ViewPlugin` `mousedown` handler that intercepts plain left-clicks within a link range and routes them to `deps.onNavigate(link)`. Click routing is split into the pure `handleBlockLinkMousedown(state, pos, event, deps)` so chord-click / right-click / out-of-range fall-through paths are unit-testable without driving CM's hit-testing internals. The `revealBlockInView(view, blockId)` helper searches the doc for the `<!-- ^<uuid> -->` stable-id marker (case-insensitive — on-disk markers are lowercase but a hand-typed link could be upper) and dispatches a `selection` + `scrollIntoView` effect; returns `false` when the marker isn't present so the caller knows to fall back to the resolver-supplied `root_index`. The plugin's `activate()` builds an `onBlockLinkNavigate` callback that (1) calls `editorClient.resolveBlockLink` to validate the link — a stale id surfaces as a notification rather than silently scrolling nowhere — (2) emits `files:open` with the target path, (3) records the pending `(relpath, blockId)` reveal, and (4) emits a `nexus.editor:reveal-block` event whose handler tries `revealBlockInView` immediately, then again after 80 ms / 250 ms so the reveal lands whether the file was already open or had to be loaded fresh. New CSS class `.cm-md-block-link` underlines + recolours the source range using the existing `--accent` token. Threaded through `EditorRuntime.onBlockLinkNavigate` so the live-mode extension stack only wires the click handler when activate() supplied one (test drivers without a full plugin runtime don't get the side-effect path). Coverage: 9 new node:test cases — 2 decoration tests (one mark per link, ignores plain wikilinks + heading anchors), 3 reveal tests (scrolls to marker line, returns false when absent, case-insensitive UUID match), 4 click-routing tests (plain left-click navigates with `preventDefault`, every chord variant + right/middle button falls through, out-of-range click falls through, pipe-aliased label flows through into the callback). `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` 0 new warnings; `pnpm --filter nexus-shell test` 624 passing (was 615; +9).

- **BL-049 phase 1 — resolver IPC + shared parser** _(shipped 2026-04-30)_: ships the kernel-side foundation that the click-through navigation UX (phase 2) and the backlinks panel (phase 3) sit on top of. New IPC handler `com.nexus.editor::resolve_block_link` (handler id 13) takes `{ file_relpath, block_id }` (UUID per ADR 0017's `<!-- ^<uuid> -->` marker) and returns `{ found, block, root_index }`. Two-tier lookup: when the file is already open as an editor session the handler reads the in-memory block tree (so unsaved edits flow through); otherwise it parses the file from disk transiently without polluting the session map. `root_index` is the position in `tree.root_blocks` of the target block's root ancestor — the resolver walks `parent_id` up the tree before recording the index, which is the granularity the shell's "scroll into view" path needs before per-block source-position metadata lands. Async + sync dispatch paths share the same `resolve_in_session` / `resolve_in_tree` helpers; sync routes through `fs::read_to_string` for the context-less unit-test fallback, async routes through `com.nexus.storage::read_file`. Wired through [`nexus-bootstrap`](../../crates/nexus-bootstrap/src/lib.rs). Shell-side: new pure module [`shell/src/plugins/nexus/editor/blockLinks.ts`](../../shell/src/plugins/nexus/editor/blockLinks.ts) defines the canonical `[[<file>#^<uuid>]]` syntax + optional `|label` alias (`parseBlockLinks`, `serializeBlockLink`, `blockLinkAt`); the regex requires a v4-shaped UUID after `#^` so a bare `[[A.md#some-heading]]` (heading anchor) doesn't masquerade as a block link. UUID casing is normalised to lowercase on parse so the on-disk marker (which `format_stable_id_marker` emits lowercase) and any user-typed link agree. New typed `EditorKernelClient.resolveBlockLink(file, id)` wrapper plus the `ResolveBlockLinkResponse` interface mirrors the Rust handler's wire shape. Coverage: 5 cargo tests on the resolver (in-session happy path, fs-fallback for closed sessions via stamp + save + close + reopen, not-found for unknown id, root-walk for nested blocks via a synthetic tree, invalid-uuid rejection); 13 node:test cases on the shell parser (no-op on plain markdown, bare + pipe-aliased forms, heading-anchor rejection, malformed-UUID rejection, casing normalisation, multi-link ranges, regex reentrancy, offset translation, `blockLinkAt` happy/miss, serializer round-trip with and without label, serializer rejection of bad inputs); 1 new kernel-client test pinning the snake_case wire shape. `cargo test -p nexus-editor` 206 passing (was 201; +5); `cargo clippy --workspace --all-targets -- -D warnings` clean; `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` 0 new warnings; `pnpm --filter nexus-shell test` 615 passing (was 601; +14).
