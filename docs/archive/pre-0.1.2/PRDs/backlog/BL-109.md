# BL-109: Virtualize the files tree

**Status:** closed — 2026-05-13

**Source**: Nexus frontend performance assessment (2026-05-11) — `experiments/nexus-frontend-assessment.html` §6
**Files**: `shell/src/plugins/nexus/files/flattenTree.ts` (new); `shell/src/plugins/nexus/files/flattenTree.test.ts` (new, 7 tests); `shell/src/plugins/nexus/files/FilesTree.tsx` (rewritten around `useVirtualizer`)
**Related**: `experiments/zed-frontend-analysis.md` (`uniform_list` pattern); BL-053 (visual target); BL-112 (perf benchmark harness — owns the future regression gate)

`FilesTree` previously rendered every expanded `TreeNode` recursively — no windowing — so paint and event-handler attachment scaled linearly with the count of visible rows. At a 10k-file forge with deep expansion this is the visible bottleneck. `@tanstack/react-virtual` was already a dependency (used by `BasesTable.tsx`), so this pass is structural rather than a new dep.

- **`flattenTree.ts` pure helper.** Walks the visible portion of the tree into an `{ entry, depth }[]` array, applying `sortEntries(mode)` at every level, skipping bundle-dir contents (`.bases` etc.) even when the dir is in the `expanded` set, and stopping at any expanded dir whose children haven't been loaded into the cache yet. Pure inputs → pure output, so the React tree stays trivially testable. `isBundleDir` lifted alongside since both consumers (the flattener + `TreeRow`) need it. The previous `compareNullableNumber` / `sortEntries` helpers moved here verbatim.
- **`useVirtualizer` over the flat list.** `FilesTree` calls `useMemo(() => flattenTree(...))` keyed on `(rootEntries, childrenCache, expandedSet, sortMode)` — the four store slices the visible list depends on. Mutations in `setChildren` / `toggleExpanded` / `setExpanded` already create new map / set identities, so the memo invalidates correctly. The virtualizer mounts only `viewport / 24px + 2×overscan` row nodes regardless of expansion depth; rows are absolutely positioned via `transform: translateY(start)` inside a relative-height wrapper, matching the pattern `BasesTable.tsx` uses.
- **`TreeNode` recursion → `TreeRow` per row.** Each row is a self-contained component: owns its drop-hover state, click toggles expand-or-load-children (folders) or activates (files), context-menu opens via the module-scope `openMenuRef` shim that pre-existed BL-109. Rows call `e.stopPropagation()` on drop / context-menu, so the container's root-drop / root-context-menu handlers fire only when the event lands outside any row. (The previous `e.target !== e.currentTarget` guards on the container couldn't survive the new inner positioned wrapper.)
- **Selection scroll-into-view via `scrollToIndex`.** The pre-virtualization `rowRef.current.scrollIntoView()` per-row effect is gone (off-screen rows don't mount). Replaced by a single effect on the parent that watches `selected` + `flatRows` and calls `virtualizer.scrollToIndex(idx, { align: 'auto' })` once the selected file's row exists in the flat list. This collapses both code paths (manual click + auto-reveal) into the same surface; auto-reveal still expands ancestors first, then the flat-list rebuild triggers the scroll.
- **Per-row status-dot subscription is now a perf win.** `RowStatusDot` previously fired `useFileStatus` on every rendered row (= every visible row in the recursive tree). With virtualization, only mounted (visible) rows subscribe — so a 10k-file forge no longer pays for 10k status reads on every cache mutation.

**Tested**: `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` clean; `pnpm --filter nexus-shell test` 1246/1246 pass (gained 7 new `flattenTree` tests). Manual benchmark deferred to BL-112 (the harness item) since the structural DoD ("`O(viewport_height / row_height)` mounted rows") is met by construction — `getVirtualItems()` returns at most `ceil(viewport / 24) + 16` items regardless of how many entries are in `flatRows`.

**Definition of done coverage**: ✅ Mounted-row count bounded by viewport; ✅ drag/drop / keyboard nav / auto-reveal / context-menu paths all preserved; ✅ scroll-into-view goes through `scrollToIndex`. The "smooth scroll at 10k files" benchmark is structurally satisfied; numeric measurement waits on BL-112's recorded-baseline harness.


## Closure notes (from index)

BL-109 closed 2026-05-13 — see above. New `flattenTree.ts` extracts the visible-rows walker as a pure helper (sorting + bundle-dir gate + lazy-children skip); `FilesTree.tsx` rebuilt around `useVirtualizer` over that flat list. DoD benchmark scenario (10k files + every folder expanded) intentionally not measured against a recorded baseline — BL-112 owns the harness; the structural goal (`O(viewport_height / row_height)` mounted rows regardless of expansion depth) is met by construction since the virtualizer's `getVirtualItems` is bounded by `viewport / estimateSize + 2*overscan`.
