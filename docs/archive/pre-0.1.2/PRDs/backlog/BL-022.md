# BL-022: Skill in-app editor UI

**Status:** closed — 2026-04-29

**Delivered:** PRD-13 §16's in-app editor lands in `shell/src/plugins/nexus/skills/`. Pre-BL-022 the Skills panel was read-only — mutations required editing `.skill.md` on disk and calling `reload` by hand. The new editor surfaces an "Edit", "Delete", and "+ New skill" affordance, round-tripping every save through the existing `com.nexus.storage::write_file` + `com.nexus.skills::reload` pair (no bespoke "save_skill" IPC — the storage handler stays the single source of write truth per ADR-0005). **Wire-shape augmentation**: `com.nexus.skills::list` and `::get` now include a `relpath` field on each entry — `Vec<u8>` builders in [`core_plugin.rs`](../../crates/nexus-skills/src/core_plugin.rs) `dispatch_list` / `dispatch_get` augment each `Skill` JSON value with the forge-relative path computed from `SkillRegistry::path_for(...)` against the configured root. Append-only on the wire — pre-BL-022 consumers ignore the new field. New helper [`relpath_for(root, abs)`](../../crates/nexus-skills/src/core_plugin.rs) returns the canonical `.forge/skills/<sub>/<name>.skill.md` shape with forward slashes (the storage handler canonicalises to the platform separator on its end). **New components**: [`SkillEditor.tsx`](../../shell/src/plugins/nexus/skills/SkillEditor.tsx) is a standalone form taking only a `kernel: SkillsKernelAPI` prop — frontmatter goes through a typed `<FrontmatterForm>` (id, name, description, version, author, created, plus comma-separated `tags` / `applicable_contexts` / `triggers` / `depends_on` via `<ListField>`); body goes through a `<BodyField>` textarea. The component is split into per-row sub-components specifically so future surfaces (BL-046's "code-aware capture" UI patterns per the BACKLOG risk #5 note) can swap the body editor without touching the frontmatter form. The editor is mounted both inline inside the expanded `<SkillRow>` (when `editingId === skill.id`) and as a standalone panel pinned to the top of the listing (when `editingId === '__new__'`). **Store extensions** ([`skillsStore.ts`](../../shell/src/plugins/nexus/skills/skillsStore.ts)): new `SkillDraft` interface; new state `editingId: string | null`, `draft: SkillDraft | null`, `saveError`, `saving`; new actions `openEditor(id)` / `openNewSkill()` / `cancelEditor()` / `patchDraft(patch)` / `saveDraft(api)` / `deleteSkill(api, id)`. `openEditor` hydrates the draft straight from the listing snapshot (the kernel returns the full body in `list`, so no extra round trip). `openNewSkill` seeds a starter draft with today's date as `created` and `version: 0.1.0`. `patchDraft` keeps the relpath in lockstep with the id slug for new skills (the user picks the slug; the file path follows). **Serialization**: new pure `serializeDraft(draft) -> string` re-emits the `.skill.md` format — `---\n<yaml frontmatter>\n---\n<body>` — with a stable key order (id-like fields first, then descriptive, then list fields) so a save round-trip is deterministic and diff-friendly. Conservative scalar quoting handles the YAML-reserved bare words (`true`, `false`, `null`, `yes`, …) and numeric-looking strings; multi-line scalars fall back to JSON-style. New `validateDraft(draft) -> string | null` checks every required frontmatter field plus the kebab-case id pattern (mirrors the kernel-side `parse_skill_text`'s required-field set) so the user catches problems before disk I/O — and `saveDraft` no-ops the IPC call when validation fails. **Save flow**: `saveDraft` calls `com.nexus.storage::write_file({ path, bytes: number[] })` (UTF-8 encoded via `TextEncoder`), then `com.nexus.skills::reload({})` to make the registry pick up the new file. Reload failure logs a warning but doesn't fail the save (the file is on disk; the next `refresh` will see it). On success the editor closes and the listing refreshes via a new module-local `subscribeSkillsChanged(listener)` channel — chosen over `eventBus.emit` to keep `skillsStore.ts` clean of `host/EventBus` imports (the `plugin-import-hygiene.test` guards that boundary). The `subscribeSkillsChanged` API mirrors the singleton-module pattern used by `contextContributors` in `nexus.ai`. **Delete flow**: `deleteSkill(api, id)` confirms via `window.confirm`, calls `delete_file` + `reload`, and optimistically prunes the listing so the row vanishes before the next refresh round-trip lands. **Edit/Delete gating**: both buttons hide for skills loaded out-of-tree (empty `relpath`) — the kernel can't write back to a skill it didn't load from disk (built-in skills bundled in the binary, hypothetical future plugin-supplied skills). **BL-021 surface**: the expanded panel now renders a `depends_on` chip row alongside `tags` / `contexts` / `triggers` so the inheritance chain is visible on the listing. **Coverage**: `pnpm --filter nexus-shell test` 462 passing (was 456; +6 net). New tests in [`skillsStore.test.ts`](../../shell/src/plugins/nexus/skills/skillsStore.test.ts): `serializeDraft round-trips frontmatter + body`, `serializeDraft quotes scalars that look numeric or reserved`, `validateDraft surfaces required-field messages`, `openEditor seeds the draft from the listing snapshot`, `saveDraft: write_file then reload, then editor closes` (asserts the IPC call ordering), `saveDraft: validation failure sets saveError without IPC`. `pnpm --filter nexus-shell typecheck` and `lint` clean (no new warnings). `cargo test -p nexus-skills` still 51 passing — the `relpath` augmentation is a wire-shape addition only. **Out of scope**: live preview during edit (existing `<RenderForm>` flow is the rendering surface — invoked from the same expanded panel without round-tripping through the editor); markdown WYSIWYG (textarea is the v1 body editor); rename-as-move semantics (changing the id of an existing skill writes a new file; the old one needs to be deleted explicitly).
