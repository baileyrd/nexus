# BL-040: AI-DIR semantic search

**Status:** closed — 2026-04-29

**Delivered:** AI-INTEGRATION-DIRECTIONS direction 3 lands as a new `com.nexus.ai::semantic_search` IPC handler plus a default-off shell palette plugin that fans out to it alongside the existing keyword search. **Backend** ([`crates/nexus-ai/src/core_plugin.rs`](../../crates/nexus-ai/src/core_plugin.rs)): new `HANDLER_SEMANTIC_SEARCH: u32 = 13` on `com.nexus.ai`, args `{ query: String, limit?: usize (default 10) }`, returns `{ matches: Vec<ChunkMatch> }`. The handler mirrors `handle_ask`'s embedder-build path (`build_embedding_provider(&embed_cfg)`) but skips the chat half entirely — the embedded query vector is forwarded to `com.nexus.storage::vector_query` (handler 18) via the existing `vectorstore::search` IPC client. Library-side, [`rag::semantic_search`](../../crates/nexus-ai/src/rag.rs) is a thin alias for `rag::retrieve` exposed under the handler-matching name so the function is grep-friendly and unit-testable independent of the dispatcher. **Manifest** ([`crates/nexus-bootstrap/src/lib.rs`](../../crates/nexus-bootstrap/src/lib.rs)): one new `("semantic_search", HANDLER_SEMANTIC_SEARCH)` entry on the `com.nexus.ai` core manifest so all four frontends (CLI, TUI, MCP, shell) reach it through the same `ipc_call` path. **Shell palette** ([`shell/src/plugins/nexus/semanticSearch/`](../../shell/src/plugins/nexus/semanticSearch/)): new default-off `nexus.semanticSearch` plugin contributing one palette command (`nexus.semanticSearch.run`, title "Search by Meaning", category "Search"). On invocation it `api.input.prompt`s for a query, then `Promise.all`s `com.nexus.storage::search` (keyword) + `com.nexus.ai::semantic_search` (semantic) — both are wrapped in `.catch(() => empty)` so a one-side failure (e.g. AI plugin off) still surfaces the other side's hits. The pure [`mergeResults(keyword, semantic, limit = 30)`](../../shell/src/plugins/nexus/semanticSearch/merge.ts) function normalises each list to `[0, 1]` by dividing by the in-list max (guarded against zero), then blends co-occurring `file_path`s at `0.5 * keyword + 0.5 * semantic` and damps single-source hits to `0.5 * normalised` — so a co-occurrence beats a single-source hit even when the raw scores would not, which is the BL-040 ranking rule's whole point. The merged list is sorted descending, capped at 30, and the top hit is opened via the existing `files:open` event the keyword search sidebar already uses; an `api.notifications.show` confirms the open or reports zero results. Snippet preference: keyword excerpt (FTS-highlighted) wins over semantic chunk text when both exist. The plugin is registered in `DEFAULT_OFF_PLUGINS` (paired with `aiPlugin`) so users don't see the command before the embedding backend is configured. **Coverage**: `cargo test -p nexus-ai` 99 passing (+5 net): three new in-module tests in [`rag.rs`](../../crates/nexus-ai/src/rag.rs) end-to-end the dispatcher path with a `StubEmbedder` + a `StubDispatcher` that intercepts `com.nexus.storage::vector_query` and asserts on the recorded args (target / command / `limit` / embedding vector); two new tests in `core_plugin.rs::semantic_search_dispatch_tests` exercise the `HANDLER_SEMANTIC_SEARCH` arm of `dispatch_async` via the missing-query and missing-embed-config error paths so we cover the routing without making a network call (a third pin-test asserts `HANDLER_SEMANTIC_SEARCH == 13` so the wire id can't drift silently). `cargo clippy -p nexus-ai --all-targets -- -D warnings` clean. `cargo test -p nexus-bootstrap` 11 passing; `cargo clippy -p nexus-bootstrap --all-targets -- -D warnings` clean. `pnpm --filter nexus-shell typecheck` clean; `pnpm --filter nexus-shell lint` clean (no new warnings); `pnpm --filter nexus-shell test` 471 passing (was 463) — 8 new merger tests in [`merge.test.ts`](../../shell/src/plugins/nexus/semanticSearch/merge.test.ts) covering co-occurrence blending, single-source damping, the "co-occurrence beats single-source" invariant, the limit cap, empty input, divide-by-zero guard, and snippet preference (kw excerpt > semantic chunk text, with fallback). The co-located `*.test.ts` is re-imported from [`shell/tests/semantic-search-merge.test.ts`](../../shell/tests/semantic-search-merge.test.ts) so the workspace test runner (which globs `tests/*.test.ts`) picks it up. No IPC-boundary types changed (args/results stay inline `serde_json::Value`), so the drift check is a no-op for this BL. **Out of scope** per the task allowance: the `?:` palette-input prefix (the dedicated palette command alone is sufficient to ship the surface — the prefix is a UX polish a follow-up can add); a dedicated semantic-search sidebar view (today the result is opened; a results-list view + dedup-by-block UI is what BL-039 / BL-045 will likely want and we'd rather build it once); and any pre-warm of the embedding model (per the `LocalEmbedding` test discipline, the backend lazily fetches BGE on first call — a UI hint that the first query is slower belongs to the BL-019 onboarding pass).
