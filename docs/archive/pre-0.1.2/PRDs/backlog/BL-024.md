# BL-024: MCP Reconnection + Connection Pool

**Status:** closed — 2026-04-28

**Delivered:** PRD-14 §4.2.4 lands as a new [`crates/nexus-mcp/src/pool.rs`](../../crates/nexus-mcp/src/pool.rs) module that replaces the ad-hoc `Arc<RwLock<HashMap<String, Arc<Mutex<McpClient>>>>>` field on `McpHostPlugin`. `PoolConfig::default()` ships the spec values: `max_per_server = 10` (BACKLOG override of PRD-14 §4.2.4's 5 — flagged in commit body since the discrepancy was deliberate), `idle_timeout = 300s`, `connect_timeout = 30s`, and `backoff = [100ms, 500ms, 2s, 10s, 30s]` matching PRD-14 §11.1 verbatim. `ConnectionPool::get_or_connect(server, host_cfg)` keeps lazy-connect semantics — entries are created on first access — but every call now runs `sweep_idle()` first, dropping any entry whose `last_used.elapsed() > idle_timeout`. Successful ops refresh `last_used` via the private `touch(server)` helper. Each entry carries a `tokio::sync::Semaphore::new(cfg.max_per_server)` so per-server in-flight calls are capped — advisory rate-limiting today since the inner `Mutex<McpClient>` already serializes, but load-bearing once a future refactor moves to `&McpClient` borrows. `McpClientError::is_transient()` returns `true` only for the `Service(_)` variant; `Spawn` (binary missing) and `Handshake` (initialize timeout / protocol error) are non-transient because retrying just delays surfacing misconfiguration. `call_with_reconnect(server, host_cfg, op)` is the retry surface: it iterates `1 + backoff.len()` total attempts, force-reconnects between attempts on transient connect / op failures, sleeps the appropriate backoff, and bails immediately on any non-transient error. The 4 dispatch-site handlers in `core_plugin.rs` (`list_tools`, `call_tool`, `list_resources`, `list_prompts`) call `pool.get_or_connect` today; threading the inner ops through `call_with_reconnect` is a follow-up so the closure-`Fn` capture rewrite of `tool` / `tool_args` can be reviewed independently. `on_stop` calls `pool.shutdown_all().await` instead of the old `clients.write().await.clear()` — same semantics (entries dropped, `McpClient::Drop` runs graceful close), tighter encapsulation. Testability is via a private `Connectable` trait with two impls: production wires `RealConnector` (which calls `McpClient::connect`); tests wire a `FakeConnector` that pops outcomes from a queue and increments an attempt counter. Combined with `#[tokio::test(flavor = "current_thread", start_paused = true)]` + `tokio::time::advance(...)` this exercises the full retry loop in deterministic constant time without spawning child processes — the previously feature-gated `tokio` `test-util` is now a `dev-dependencies` feature on the crate (no new external dep — `tokio` was already there). Coverage: 9 new unit tests in `pool.rs` — `backoff_schedule_matches_spec` (locks the [100ms..30s] array), `pool_config_defaults` (locks the four numeric defaults), `call_with_reconnect_gives_up_after_full_schedule` (6 transient errors → returns last error, `attempts == 6`), `call_with_reconnect_does_not_retry_non_transient` (one Handshake → 1 attempt), `call_with_reconnect_stops_at_non_transient_op_error` (in-budget retry counter ≤ 6), `idle_sweep_on_empty_pool_evicts_nothing` (boundary case after time-advance past the idle window), `disconnect_returns_false_for_unknown_server`, `shutdown_all_clears_empty_pool`, and `is_transient_only_for_service_variant` (variant table guard). Total `cargo test -p nexus-mcp`: 30 passing (was 21). Out of scope per the BL-024 plan: no background sweep task (lazy eviction only), no transport changes (rmcp untouched), no IPC contract drift (handler ids 1–7 and arg shapes stay), no auth flow (BL-025), no in-handler `call_with_reconnect` wiring (deferred so the call-site closure rewrite can be a focused follow-up).
