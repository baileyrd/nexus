# BL-015: Soft-deleted bases — trash view UI

**Status:** closed — 2026-04-28

**Delivered:** PRD-10's soft-delete trash UX was already most of the way there — the audit trail showed `BaseRecord.deletedAt` wired end-to-end (table actions, kernel filter at `BasesView.tsx`, IPC handlers `base_record_soft_delete` / `base_record_restore`) and a Trash filter toggle in the header (`BasesView.tsx:145-164`) that swaps the live record set for the soft-deleted set, plus toolbar actions in `BasesTable.tsx:408-427` for **Restore** and **Delete forever** that only render when `trashOpen && selectedRecordId`. The default Backspace path goes through `handleSoftDeleteRow` (lands records in the trash, restorable); hard-delete is gated behind the trash view so a reflexive Backspace can't permanently destroy data. Both directions are integrated with the bases undo stack (`pushHistory` with paired forward/inverse) so soft-delete and restore round-trip via Cmd-Z. The remaining gap closed in this pass: an empty-state for the trash. Previously, switching into trash mode with zero soft-deleted records dropped through to whichever view was active (table / calendar / timeline / etc), which renders awkwardly on the non-table views (empty months, empty lanes). New `TrashEmptyState` component in `BasesView.tsx` short-circuits before any of the per-mode views mount when `trashOpen && trashCount === 0`, showing a plain "The trash is empty." message and a "Back to live records" button that flips `setTrashOpen(relpath, false)`. Out of scope for v1: bulk "Empty trash" action (one-shot delete-all-forever) — sensible follow-up but not on the BL-015 critical path. Coverage: pre-existing tests for soft-delete/restore IPC and undo stack continue to pass; no new unit tests since the empty-state is a pure-DOM short-circuit with no IPC. Total `pnpm --filter nexus-shell test`: 370 passing (1 pre-existing skip), `pnpm --filter nexus-shell typecheck`: clean.
