# BL-070: Vim keybindings mode

**Status:** closed — 2026-05-06

**Source**: Editor Integration Assessment (2026-05-06) — gap #2a; PRD-08 §9
**Files**: `shell/src/plugins/nexus/editor/cm/vimKeymap.ts` (new), `shell/src/plugins/nexus/editor/cm/{extensions.ts,CodeMirrorHost.tsx}`, `shell/src/plugins/nexus/editor/{index.ts,EditorView.tsx,runtime.ts}`, `shell/package.json`
**Related**: BL-071 (Emacs mode — parallel)

`@replit/codemirror-vim` (^6.3.0) is now mounted on top of the default keymap when the user flips the new `nexus.editor.keybindings` setting from `'default'` to `'vim'`. Normal/Insert/Visual modes ship with the package; the status panel renders via `vim({ status: true })`.

**Per-view ex command dispatch.** `Vim.defineEx` is process-global, so the standard "register once, look up the active view's callbacks" pattern is implemented via a CM6 `StateField` (`vimContextField`) that each mount initializes with `{ relpath, onSave, onClose }`. The four registered handlers (`:write`, `:quit`, `:wq`, `:x` — including their canonical short-prefix forms) read `cm.cm6.state.field(vimContextField)` at dispatch time, so two tabs open simultaneously route their ex commands to their own callbacks. Test coverage in `vimKeymap.test.ts` exercises this case end-to-end with two mounted views.

**Wiring.** `BaselineExtensionsOptions` gained `keybindings?: 'default' | 'vim'` plus a `vim?: VimKeymapOptions` payload. `CodeMirrorHost` accepts those and threads them into `baselineExtensions`. `EditorView.tsx` reads the setting via `runtime.getKeybindings()` at render time and includes the value in the host's `key` prop, so a settings flip + tab reopen remounts cleanly with the new keymap. `:w` calls `runtime.kernelClient.saveSession(relpath)`; `:q` calls `runtime.confirmAndClose(relpath)`, so the dirty-tab confirmation flow still applies.

**Slash-command coexistence.** PRD-08's worry about `/` collision resolves by construction: the editor's slash-command extension only fires when `/` is typed at the start of an empty paragraph. In Vim Normal mode the typed `/` reaches vim's search layer first; in Insert mode it reaches the slash-command extension exactly as it does in default mode. No interceptor needed.

**Tests.** Five unit tests in `vimKeymap.test.ts` (and re-exported via `tests/vim-keymap.test.ts` for the top-level glob):

- `getCM` returns the CM5 wrapper for a vim-enabled view.
- `:w` fires `onSave` exactly once.
- `:q` fires `onClose` exactly once.
- `:wq` fires save then close in order.
- Two mounted views with different `relpath`s dispatch to their own callbacks.

All 846 shell tests pass (was 841; +5 vim).

**Deferred from the original DoD:**

- *Emacs option in the same toggle* — the schema's `options` array currently lists `['default', 'vim']`. BL-071 will add `'emacs'` and the matching `emacsKeymap.ts`; the rest of the plumbing (settings read, host prop, key-based remount) is shared.
- *Live re-mount on settings flip* — today the user flips the setting and reopens the tab. CM6 compartments could swap the keymap in place, but the vim layer's modal state can't be cleanly hot-swapped (a stranded Insert-mode wrapper is worse than the close-and-reopen friction). Documented in the schema's description.
