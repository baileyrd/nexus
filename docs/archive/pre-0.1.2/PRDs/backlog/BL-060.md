# BL-060: Ad-hoc command history — IPC exposure and shell UI

**Status:** closed — 2026-05-07

**Source**: Terminal Integration Assessment (2026-05-06) — gap #3
**Files**: `crates/nexus-terminal/src/{core_plugin.rs, lib.rs}`, `crates/nexus-bootstrap/src/lib.rs`, `crates/nexus-cli/src/{main.rs, commands/proc.rs}`, `shell/src/plugins/nexus/terminal/{index.ts, HistoryView.tsx, HistoryPaneView.tsx, historyStore.ts, terminal.css}`, three new ts-rs bindings under `packages/nexus-extension-api/src/generated/ipc/`
**Related**: PRD-09 §10; `SqliteAdHocStore` shipped Phase M; BL-055 / BL-056 (still owe a dispatch-side `adhoc_record` for full first-class IPC surface)

Exposes the existing `procmgr_adhoc_history` SQLite store over IPC, surfaces a `nexus proc history` CLI verb, and ships a sibling sidebar leaf to Saved Commands that lists ad-hoc rows with re-run / promote / delete affordances.

- **IPC.** Four new handlers on `com.nexus.terminal`: `adhoc_list` (id 19, `{ limit?: u32 }` → `Vec<AdHocRecord>` ordered by `executed_at` desc — defaults to 100), `adhoc_get` (id 20, `{ id }` → `AdHocRecord | null`), `adhoc_delete` (id 21, idempotent — returns `{ id }` regardless of whether the row existed, mirroring the underlying `DELETE` no-op), `adhoc_promote` (id 22 — wraps `promote_adhoc_to_saved` and returns the freshly inserted `SavedCommand`). The DoD-suggested ids 18–21 were the *intent* but BL-059's `open_in_terminal` already occupies id 18, so the contiguous block slipped to 19–22; ids are append-only per the module-level invariant.
- **Plugin wiring.** `TerminalCorePlugin` gains a new `adhoc: Option<Mutex<SqliteAdHocStore>>` field plus a `with_adhoc_store(...)` builder; bootstrap opens both stores against the same `<forge>/.forge/procmgr.sqlite` file (separate `Connection`s, separate tables — `procmgr_commands` vs. `procmgr_adhoc_history`) and logs-and-degrades on open failure exactly the way the saved-store path already did. The `adhoc_*` handlers return a clear `ad-hoc history store not attached` error when the runtime was built without a forge (tests, embedded).
- **CLI.** `nexus proc history [--limit N] [--json]` wraps `adhoc_list`. Default rendering is a fixed-width table — `WHEN` (formatted from `executed_at` via `chrono`), `STATUS` (success / failure / timeout), `RUNS` (the dedupe counter), `COMMAND` (truncated to fit), `CWD`. `--json` returns the raw response untouched for piping.
- **Shell UI.** New `HISTORY_VIEW_TYPE = 'terminal-history'` leaf revealed by the `nexus.terminal.history.show` palette command. Layout intentionally mirrors `SavedCommandsView` (same `.nexus-saved-commands*` classes) so the user's eye doesn't relearn the row affordances when moving between Saved and History panels. Each row shows the command literal in monospace, a colored status chip (✓ success / ✗ failure / ⏱ timeout), the run counter, a relative-time stamp (`5m ago` / `2d ago` / falls back to `YYYY-MM-DD` past 30 days), and the cwd. Action row (hover-revealed via the same opacity pattern as BL-066): Run → `send_input` of the command literal; Promote → inline name field, defaults to the first-token program name (`/usr/bin/foo args` → `foo`), then `adhoc_promote` and a saved-commands cache reload; Delete → `adhoc_delete` with optimistic prune. `historyStore.ts` mirrors `savedCommandsStore.ts` shape so subsequent shell work has one pattern to follow.
- **Lifecycle.** Cache resets on `workspace:closed` so the next forge doesn't see stale rows from the previous one's `procmgr_adhoc_history` table. The single in-flight `loading` flag prevents overlapping reloads when the user spam-clicks the refresh icon.

**Tests.**

- 10 new dispatch-boundary tests in `core_plugin.rs::tests` cover: missing-store error, default-limit ordering, explicit-limit truncation, get-unknown-id returns null, get-known-id round-trip, idempotent delete, delete-then-get-null, promote produces correct `SavedCommand` (slug, shell, cmd, cwd, icon all faithful), promote-unknown-id surfaces the underlying `no adhoc row` error, invalid-args reported as `ExecutionFailed`. Seeding goes through `SqliteAdHocStore::record` *before* the plugin consumes the store via `with_adhoc_store(...)` since the plugin owns the only handle once attached.
- All 290 nexus-terminal unit tests pass; full ts-export regen passes; shell typecheck and the new files lint cleanly (`HistoryView.tsx`, `HistoryPaneView.tsx`, `historyStore.ts` are absent from the `pnpm lint` output).

**Deliberately deferred:**

- **Recording new ad-hoc rows over IPC.** The procmgr layer still calls `SqliteAdHocStore::record` directly — there's no `adhoc_record` handler. This becomes load-bearing only when running ad-hoc commands becomes a first-class IPC verb (BL-056 territory). Today, the read / delete / promote surface is what unblocks the user-visible feature.
- **Status filtering, time-range constraints, free-text search.** The DoD only required a recency-ordered list; the table view + the existing `recent(limit)` covers the "I ran this earlier today, what was it again" case. Search across sessions is BL-063 (FTS5 index) and operates on output, not on this command history.
