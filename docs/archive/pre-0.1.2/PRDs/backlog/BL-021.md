# BL-021: Skill `depends_on` composition resolver

**Status:** closed — 2026-04-28

**Delivered:** PRD-13 §5's four sub-tasks land as new [`crates/nexus-skills/src/compose.rs`](../../crates/nexus-skills/src/compose.rs) plus a new IPC handler and a planner-side consumer. **Sub-task 1 — parse `depends_on`** was already wired (`SkillMeta.depends_on: Vec<String>`); the new module just consumes it. **Sub-task 2 — topological sort + cycle detection**: iterative DFS with the standard white/gray/black colouring. A back-edge into a `Gray` node reconstructs the offending cycle path from the active stack — `ComposeError::Cycle { path: Vec<String> }` carries `[a, b, c, a]` so the operator sees the loop without re-walking. Self-cycles, indirect cycles, and unresolvable ids each surface as a distinct typed variant (`Cycle` / `MissingDependency { from, missing }` / `UnknownRoot`). Diamond inheritance dedupes naturally — visiting a `Black` node short-circuits — so `d → b → a; d → c → a` produces `[a, b, c, d]` with `a` appearing once. **Sub-task 3 — prompt-fragment merge order**: post-order DFS puts dependencies first and the root last, mirroring how class inheritance reads top-down. Within a layer, children are visited in `depends_on` declaration order so the output is stable across runs. The merged body emits each fragment under a `## Skill: <name> [<id>]` heading separated by blank lines, giving the model a clear boundary to latch onto. **Sub-task 4 — conflict-warning UX**: non-fatal `ComposeConflict` enum with two variants today: `ParameterClash { parameter, skill_ids }` (two ancestors declare the same parameter name with differing `param_type` or `default`) and `RestrictionsDisagree { field, skill_ids }` (`modify_files` / `delete_content` / `execute_code` / `allowed_tools` levers carrying contradictory values across the closure). Conflicts ride alongside `fragments` + `merged_body` on `ComposedSkill` so the Skills UI can surface a banner without losing the merged composition. **IPC handler 7 — `compose`**: new [`HANDLER_COMPOSE: u32 = 7`](../../crates/nexus-skills/src/core_plugin.rs) registered in [`nexus-bootstrap/src/lib.rs`](../../crates/nexus-bootstrap/src/lib.rs) under the `com.nexus.skills` manifest. Args: `{ id }`. Returns `ComposedSkill` JSON: `{ root_id, fragments: [{ id, name, body }], merged_body, conflicts: [{ kind, ... }] }`. Handler ids are append-only — pre-BL-021 callers (`list` / `get` / `list_by_context` / `triggered_by` / `reload` / `render`) are untouched. **Planner integration**: [`nexus-agent`](../../crates/nexus-agent/src/core_plugin.rs)'s `system_prompt_with_skills` now calls a new `compose_skill_body(ctx, id)` for each triggered skill BEFORE falling back to the single-skill `render_skill_body`. The composed merged body replaces the per-skill section so an inheritance chain like `concise → markdown-style → rust` contributes every layer's instructions in topo order. Errors (cycle / missing dep / IPC unavailable) silently degrade to the pre-BL-021 single-skill path so a broken dep graph never blocks planning. Conflicts are emitted via `tracing::warn!` with `skill_id` + `conflict_count` so operators can spot them in logs without us having to plumb an event channel through to the planner UI; the Skills panel will surface them through the IPC response when BL-022 lands. **Coverage**: `cargo test -p nexus-skills` 51 passing (was 42; +9 net). New tests: `compose_returns_self_only_when_no_deps`, `compose_orders_deps_before_dependents`, `compose_diamond_dedupes_to_single_visit`, `compose_reports_cycle_with_full_path`, `compose_reports_self_cycle`, `compose_reports_missing_dependency`, `compose_reports_unknown_root`, `compose_surfaces_parameter_clash`, `compose_surfaces_restriction_disagreement`. `cargo clippy -p nexus-skills -p nexus-agent -p nexus-bootstrap --all-targets -- -D warnings` clean. **Out of scope**: parameter-substitution composition (compose returns raw bodies; per-fragment `render` substitution is a follow-up if needed); UI surfacing of conflicts inside the Skills panel (folded into BL-022).
