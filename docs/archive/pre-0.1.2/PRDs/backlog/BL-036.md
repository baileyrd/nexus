# BL-036: AMB margin suggestions + inline correction

**Status:** closed — 2026-04-30

**Source**: [../AI-AMBIENT-COPILOT-PLAN.md](../AI-AMBIENT-COPILOT-PLAN.md) patterns 6 + 9.
**Effort**: Medium (~1 week, four phases). **Crate**: `shell/src/plugins/nexus/{ai,editor}/`.
Background pass over the active document surfaces non-blocking margin glyphs ("rephrase", "tighten", "fact-check") that expand into accept/dismiss diffs in the gutter; spelling and grammar share the same engine but render as wavy underlines reachable through a right-click menu rather than glyphs (gutter-clutter avoidance for high-volume kinds).

- **BL-036 phase 1 — engine + store** _(shipped 2026-04-30)_: lays the engine plumbing with no UI mounted yet. New module [`shell/src/plugins/nexus/ai/marginSuggestStore.ts`](../../shell/src/plugins/nexus/ai/marginSuggestStore.ts) — zustand store with the `Suggestion` shape (5 kinds — `rephrase` / `tighten` / `fact-check` / `spelling` / `grammar` — anchored char ranges, generation counter for drift detection) and the pass lifecycle (`beginPass / setSuggestions / setError / dismiss / accept / clear`). The same `currentRequestId` staleness guard the chat / Cmd+I stores use drops late results from a superseded pass. New module [`shell/src/plugins/nexus/ai/marginSuggest.ts`](../../shell/src/plugins/nexus/ai/marginSuggest.ts) — `requestPass(api, docPath, docText)` invokes `com.nexus.ai::stream_chat` (no RAG retrieval — same shape as Cmd+I, same handler the chat surface uses for tool-calling) with a `margin-…` session prefix and a 30s budget tighter than chat's 60s (a margin pass that takes a full minute means the user has typed a paragraph since it started, so the result is almost certainly stale). `parseSuggestionsResponse(text, docText, generation, requestId)` parses the model's JSON-array response, anchors each suggestion by locating its `original` substring inside the doc (drops hallucinated entries where the substring doesn't appear), dedupes by `kind|original`, caps at 6 per pass, and tolerates a stray ```json fence the model occasionally emits despite the "no markdown" instruction. The engine is single-flight by virtue of the store's request-id staleness check — concurrent `requestPass` calls supersede each other; debouncing is the caller's job (the CM extension in phase 4). Errors are written to `lastError` and the promise resolves to `[]` so background fire-and-forget callers don't have to wrap in try/catch. Coverage: 28 new node:test cases (16 engine + 12 store) covering the parser's anchor / dedupe / cap / fence / line-tracking paths, transport errors, generation-bump-per-call, and the stale-pass drop.

- **BL-036 phase 2 — margin-glyph + diff-card CM extension** _(shipped 2026-04-30)_: the rendering surface. New module [`shell/src/plugins/nexus/editor/cm/marginSuggestions.ts`](../../shell/src/plugins/nexus/editor/cm/marginSuggestions.ts) — a CM6 `StateField<{ suggestions, expandedId }>` with three effects (`setResolvedEffect` from the store sync, `expandEffect` for diff-card open/close, `dropOneEffect` for accept/dismiss) and a `provide: EditorView.decorations.compute` that emits per-kind `Decoration.mark` underlines so the suggested span is visually anchored. `resolveSuggestions(store, docText)` validates each store entry against the live doc text (drops drift); the StateField re-validates after every doc-change transaction by remapping `[from, to)` via `tr.changes.mapPos` and confirming the slice still equals `original` — accepting a stale suggestion would clobber edits, never do it. The `MarginSuggestionsView` ViewPlugin renders an absolute-positioned right-margin glyph layer (one button per `rephrase` / `tighten` / `fact-check` suggestion using `↻` / `✂` / `?` icons; spelling / grammar deliberately get no glyph) and a floating diff-card overlay showing original vs replacement plus Accept / Dismiss buttons. Subscribes to the zustand store and dispatches `setResolvedEffect` whenever suggestions change for the bound `relpath` — multiple tabs share the singleton store; a stale tab paints nothing. `buildAcceptTransaction(suggestion)` returns a CM TransactionSpec (`changes` + a co-fired `dropOneEffect`) so the glyph disappears atomically with the doc edit; returns `null` for fact-check (annotation-only — Accept on the card just dismisses, no doc edit). The transaction goes through the editor's transaction-bridge pipeline like any user edit so kernel undo/redo covers margin-suggestion accepts. `installMarginSuggestStyles()` mirrors `installBlockHandleStyles` — colour vars fall back to dark-theme defaults. Wired into [`shell/src/plugins/nexus/editor/EditorView.tsx`](../../shell/src/plugins/nexus/editor/EditorView.tsx)'s live-mode extension list as `marginSuggestionsExt({ relpath: tab.relpath })` after `linkSuggestExt`; source-mode tabs don't mount it — the AMB pass over a raw markdown doc is a live-mode UX. Also added [`shell/tests/margin-suggest.test.ts`](../../shell/tests/margin-suggest.test.ts) re-export wrapper picking up the phase-1 engine tests (which were missed in the phase-1 commit) AND the new phase-2 extension tests under the default `pnpm test` glob. Coverage: 15 new node:test cases over `resolveSuggestions` (drift, out-of-bounds, ordering), `buildDecorations` (sort + empty), StateField effects (set / expand / drop / unknown-id guards), doc-change drift mapping (preserve outside, drop inside, collapse card on any edit), and `buildAcceptTransaction` (replace + drop spec, fact-check null). `pnpm --filter nexus-shell test` runs 747 cases (was 704; +28 phase-1 picked up via the wrapper, +15 phase-2). Behaviour today is dormant in production — extension is mounted but renders nothing until something writes to the store.

- **BL-036 phase 3 — wavy squiggle + right-click menu** _(shipped 2026-04-30)_: closes the user-visible scope. The phase-2 placeholder dotted underline for `cm-margin-suggest--{spelling,grammar}` is replaced with an inline-SVG wavy underline rendered via `background-image` (a single `<svg viewBox='0 0 6 3'><path d='M 0 1.5 Q 1.5 0 3 1.5 T 6 1.5'>` repeated horizontally, encoded inline so there's no external request and theme overrides happen by swapping the stroke colour). Two tints — warning `#fbbf24` for spelling, accent `#a78bfa` for grammar — keep the kinds visually distinct without a second palette token. Pure CSS swap inside `installMarginSuggestStyles`; the decoration class wiring landed in phase 2 so no JS changes. Right-click anywhere inside an active suggestion span surfaces a small native menu at the click coords with Accept fix / Show diff / Dismiss rows. Critically, this is the ONLY way to reach spelling / grammar suggestions for review — those kinds don't get a margin glyph, so the right-click is the gate; for rephrase / tighten / fact-check the menu is supplementary to the glyph + diff-card path. Pure helpers extracted so the spec is testable without DOM: `suggestionAtPos(suggestions, pos)` does an inclusive-from / exclusive-to hit-test (matches `Decoration.mark` semantics — a click at the right edge of a span belongs to the next character), with first-match-wins on overlap for deterministic menu UX; `buildContextMenuRows(suggestion)` returns the per-kind row spec — rephrase / tighten emit Accept + Dismiss, fact-check emits Dismiss only (no replacement to apply), spelling / grammar emit Accept + Show diff + Dismiss with Show diff being the unique row vs the glyph kinds (since no glyph means right-click is the only way to see the proposed fix before applying). Click behaviour falls through to the same `buildAcceptTransaction` / `dropOneEffect` machinery the diff card uses. Default platform menu is suppressed ONLY when the click lands inside a suggestion — clicks outside still get paste / spell-check / etc. Coverage: 11 new node:test cases (5 for `suggestionAtPos` covering inclusive/exclusive bounds, no-match paths, empty list, overlap determinism; 6 for `buildContextMenuRows` covering each kind plus "Dismiss is always last" so the CSS separator stays correct). `pnpm --filter nexus-shell test` 758 (was 747; +11). **Out of scope (phase 4)**: skipped registering Accept/Dismiss against `aiActionRegistry`. BL-035 (commit `60a0657`) explicitly defers the editor selection-menu wiring, so registering against the registry today is invisible — forward-compatible follow-up once that menu lands.

- **BL-036 phase 4 — idle-debounced trigger + settings** _(shipped 2026-04-30)_: makes the feature actually run in production. New module [`shell/src/plugins/nexus/ai/marginApi.ts`](../../shell/src/plugins/nexus/ai/marginApi.ts) — module-scoped `setMarginApi` / `getMarginApi` PluginAPI handle, mirroring `ghostApi.ts` (BL-034) and `cmdIApi.ts` (BL-032) so the editor-side CM extension (which runs far from the AI plugin's activate closure) can lazily reach `requestPass`. New module [`shell/src/plugins/nexus/editor/cm/marginSuggestTrigger.ts`](../../shell/src/plugins/nexus/editor/cm/marginSuggestTrigger.ts) — a CM ViewPlugin that resets a `setTimeout` on every `docChanged` (selection moves and viewport scrolls don't count as typing), and on idle expiry calls `requestPass(api, relpath, docText)`. Pure helpers exposed for tests: `readMarginSuggestSettings()` reads from `configStore` with `MARGIN_SUGGEST_DEFAULTS` fallback; `shouldFirePass(settings, docText, relpath)` returns a `SkipReason` string (`'disabled' | 'doc-too-short' | 'doc-too-long' | 'untitled'`) or `null` when the trigger should fire — surfacing the reason rather than a bare boolean lets tests assert on WHY a pass was skipped. Settings re-read at fire time so a flip-to-disabled mid-debounce takes effect (the timer is also cancelled outright when `enabled` becomes false). Untitled tabs are skipped (no kernel session to anchor citations against, mirroring the predicate in `editor/sessionManager.ts`). Four settings keys added to the AI plugin's `contributes.configuration.schema` in [`shell/src/plugins/nexus/ai/index.ts`](../../shell/src/plugins/nexus/ai/index.ts): `ai.marginSuggest.enabled` (boolean, default `false` — opt-in for v1 because each pass is a model call; enabling implicitly would surprise users with provider-config dialogs on first idle), `ai.marginSuggest.idleMs` (number, default `5000`), `ai.marginSuggest.minDocChars` (default `200`), `ai.marginSuggest.maxDocChars` (default `8000` — caps token cost on book-length notes). Defaults agree with `MARGIN_SUGGEST_DEFAULTS` in the trigger module by construction (shared constant). `setMarginApi(api)` is called from the AI plugin's `activate()` next to the existing `setGhostApi(api)`. Wired into `EditorView.tsx`'s live-mode extension list as `marginSuggestTriggerExt({ relpath: tab.relpath })` after `marginSuggestionsExt`. Coverage: 15 new node:test cases — 8 pure (kill-switch / untitled / doc-length boundaries / configStore round-trip) and 7 end-to-end mounting a real `EditorView` under happy-dom with `t.mock.timers` (default-disabled dormancy, fire-after-idleMs, debounce coalescing across rapid edits, doc-too-short skip, untitled skip, null-handle short-circuit when AI plugin not yet active, and the live-edit settings re-read mid-debounce). The mocked-timer suite uses a `flushMicrotasks()` helper that drains via `Promise.resolve()` rather than `setTimeout(0)` because mocked timers freeze the latter and would hang the test. Typecheck clean; lint produces no new warnings. **Out of scope (follow-ups)**: (1) on-mount pass for read-only review (today the trigger only fires after typing — opening a doc and reading without editing surfaces nothing); (2) per-kind enable toggles (single kill switch covers all five kinds today); (3) registering Accept/Dismiss against `aiActionRegistry` once BL-035's selection-menu wiring lands (commit `60a0657` explicitly deferred that wiring; registering today would be invisible).
