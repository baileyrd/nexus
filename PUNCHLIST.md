## Architecture review (2026-04-16) ŌĆö microkernel adherence

- [x] **`nexus-app/src/forge.rs` routes through the kernel for file-tree ops.** All seven commands (`read_forge_file`, `write_forge_file`, `create_forge_file`, `create_forge_dir`, `rename_forge_entry`, `delete_forge_entry`, `list_forge_dir`) now call `ipc_call("com.nexus.storage", ŌĆ”)` via `editor::KernelRuntime`. New storage handlers `list_dir` / `create_file` / `create_dir` / `rename_entry` / `delete_entry` added (ids 27ŌĆō31) on `StorageEngine` with `resolve_within` / `resolve_target` path-confinement. Writes go through the plugin's `atomic_write` (temp + fsync + rename). Coverage: `crates/nexus-bootstrap/tests/forge_ipc.rs` (7 tests).
- [x] **`nexus-editor::core_plugin` routes FS I/O through `com.nexus.storage`.** Adopted the existing `CorePlugin::wire_context` + `dispatch_async` pattern that `com.nexus.ai` already uses (no `CorePlugin::dispatch` signature change needed ŌĆö punch list's option (a) assumed a trait change that turned out to be unnecessary). Editor captures its `KernelPluginContext` via `wire_context`; async `open`/`save` handlers `ipc_call("com.nexus.storage", "read_file"/"write_file", ŌĆ”)` so reads/writes go through storage's atomic-write + index-update + capability-check envelope. Sync dispatch kept as a unit-test-only fallback with the fsync gap fixed (`sync_all` on temp file + best-effort parent-dir fsync). Coverage: new `save_routes_through_storage_and_updates_index` test in `crates/nexus-bootstrap/tests/editor_ipc.rs` proves save updates the storage SQLite index (would stay stale if save still used `std::fs`).
- [x] **Collapsed bases SQL into `nexus-storage`; `nexus-database` is now a pure library.** `schema.rs` / `query.rs` / `relation.rs` moved from `crates/nexus-database/src/` to `crates/nexus-storage/src/bases/` ŌĆö same SQL, same tests, now living alongside the only `rusqlite` owner. `nexus-database` keeps types/validation/formulas/CSV (no rusqlite dep) and is reclassified as a library crate; its no-op `DatabaseCorePlugin` (empty dispatch, never called) is deleted along with the bootstrap registration. Added `("nexus-database", "rusqlite")` to `dep_invariants::FORBIDDEN`. ARCHITECTURE.md ┬¦3/┬¦4.2/┬¦6.1/┬¦7 updated to show storage as the single SQLite owner and nexus-database as a leaf library.
- [x] **`nexus-cli` links `nexus-database` directly for CSV/formulas.** Reintroduced `com.nexus.database` as a pure-logic core plugin in `crates/nexus-database/src/core_plugin.rs` exposing three handlers (`csv_import` / `csv_export` / `formula_eval`) ŌĆö no SQLite, just a thin IPC wrapper around the existing CSV import/export and formula-evaluation helpers. Registered it in `nexus-bootstrap/src/lib.rs` alongside the other core plugins, added typed client helpers in `nexus-bootstrap/src/database.rs`, rewired `nexus-cli/src/commands/bases.rs` to call them via `ipc_call("com.nexus.database", ŌĆ”)`, dropped the direct `nexus-database` + `csv` deps from `nexus-cli/Cargo.toml`, and extended `dep_invariants::FORBIDDEN` with `("nexus-cli", "nexus-database")` + `("nexus-tui", "nexus-database")` so this can't regress. ARCHITECTURE.md ┬¦4.2 / ┬¦6.1 updated to reflect the new IPC surface. Coverage: `crates/nexus-bootstrap/tests/database_ipc.rs` (4 tests) plus the existing unit tests on the plugin itself.
- [x] **De-dup `resolve_within` path-confinement logic.** New `nexus_types::paths::resolve_within` is the single owner of forge-relative path-component validation. `nexus-storage/src/lib.rs` and `nexus-editor/src/core_plugin.rs` now delegate to it (storage maps the shared `PathError` ŌåÆ `StorageError::PermissionDenied`; editor adds its own `canonicalize` pass for the sync unit-test fallback). `nexus-app/src/forge.rs`'s prior copy was already gone post-storage-IPC migration, so no change there. `KernelPluginContext::confine_path` is left alone ŌĆö different signature (`Path` + absolute paths + always canonicalize), not a relpath validator. Coverage: 5 new unit tests in `nexus-types::paths` covering empty/normal/parent/current-dir/absolute cases; all existing storage + editor + bootstrap IPC tests still pass.

- [x] **Dual FS watchers on the forge root.** `nexus-app/forge.rs` starts a `notify_debouncer_mini` watcher (`lib.rs` setup block) to push `forge:fs-changed` to the frontend. `StorageEngine::open_internal` also starts its own `Watcher` on the same root. Two independent OS-level watches are running simultaneously. The Tauri-side watcher is the wrong layer: `StorageCorePlugin` should emit a `NexusEvent::Custom { type_id: "storage.file.changed" }` on its internal watcher events, and the Tauri event forwarder should relay that to the frontend. Removing the `notify_debouncer_mini` dep from `nexus-app` would also guarantee the frontend sees change notifications only *after* the index has been updated (current order is undefined).
- [x] **Theme engine is now a core plugin.** New `ThemeCorePlugin` in `crates/nexus-theme/src/core_plugin.rs` owns the `ThemeEngine` state behind a `Mutex` and registers as `com.nexus.theme` with 11 IPC handlers (`get_available_themes` / `apply_theme` / `compute_variables` / `get_available_snippets` / `toggle_snippet` / `reorder_snippets` / `get_theme_config` / `set_mode` / `apply_config` / `set_plugin_overrides` / `reload`). Every mutating handler publishes a `com.nexus.theme.changed` event to the kernel bus with the updated `ThemeConfig` as payload, so plugins can subscribe via `EventFilter::CustomPrefix("com.nexus.theme.")` and react to theme-driven changes. `nexus-app/src/commands.rs` now dispatches via `ipc_call("com.nexus.theme", ŌĆ”)` on the kernel runtime instead of a shell-owned `EngineState`. `nexus-app/src/lib.rs` spawns a `start_theme_event_forwarder` background thread that relays bus events to the frontend as `theme:changed` Tauri events; `app/src/App.tsx` listens for those and calls a new `syncFromEngine` action on the theme store to reflect plugin-driven mutations in the UI. Coverage: 6 new unit tests in `core_plugin.rs` + 4 integration tests in `crates/nexus-bootstrap/tests/theme_ipc.rs`.
- [x] **Community plugins can now reach core plugin IPC.** Community plugins (in `PluginManager`) and core plugins (in bootstrap's `SharedPluginLoader`) stay in separate registries by design ŌĆö different trust levels, different backends ŌĆö but the dispatcher injected into community WASM plugins is now a new `CompositeIpcDispatcher` (`crates/nexus-plugins/src/composite.rs`) that tries the community manager first and falls through to the bootstrap loader on `IpcError::PluginNotFound`. Only `PluginNotFound` triggers fall-through; `CommandNotFound` propagates as-is so a community plugin can't shadow a core command. `PluginState` gains a write-once `core_loader: FallbackCell` populated in the Tauri `setup` closure after `build_cli_runtime` returns, which means community WASM plugins can now `host::invoke_command("com.nexus.storage", "read_file", ŌĆ”)` from inside their sandbox. Coverage: 5 unit tests in `composite.rs` (primary-short-circuit, fall-through, command-not-found passthrough, empty-fallback, late-install) plus 5 integration tests in `crates/nexus-bootstrap/tests/community_to_core_ipc.rs` that use an `EmptyCommunityDispatcher` as a stand-in for the real WASM community dispatcher and prove theme/storage core plugins are reachable.

## UI architecture review (2026-04-16) ŌĆö editor-shell pattern

### Code gaps

- [x] **Tabs don't dispatch on `Tab.surface` / `Tab.contentType`.** `app/src/components/layout/PaneView.tsx:17` hardwires `<FileViewer />` whenever `openFileStore.file` is set, ignoring the `surface` + `contentType` fields that already exist on `Tab` ([bindings/Tab.ts](app/src/bindings/Tab.ts)). Wire `PaneView` through `useContentType(tab.contentType)` the same way `SidePanelBody` ŌåÆ `PanelView` already does for side panels, so plugins can register tab surfaces (graph view, canvas editor, terminal, custom preview) through the existing contribution registry. Covers PRD-07 ┬¦7 acceptance criterion "Tab system supports drag-to-reorder, drag-to-split, context menu" + the missing tab-content dispatch.
- [x] **PRD-08 ┬¦14.1ŌĆō14.3 editor extension points now land through the contribution registry.** Three new APIs on `contributions` (`registerEditorBlockType` / `registerEditorDecorationProvider` / `registerEditorKeybinding`) plus matching `list*` / `subscribe*` readers and a `useEditorBlockTypes()` React hook. `EditorSurface.tsx` holds decoration providers and editor-scoped keybindings in CM6 `Compartment`s and reconfigures them whenever the registry emits, so a hot-reloaded plugin's contributions swap in without remounting the editor. Script plugins reach the registries through a new `editor` surface on `NexusPluginContext` (`ctx.editor.registerBlockType` / `registerDecorationProvider` / `registerKeybinding`); each returns a `Disposable` the plugin calls from `onStop`. Keybindings dispatch through `contributions.invokeCommand`, so the existing command registry is the single authority over what runs. Slash commands (┬¦14.4) were already landed.
- [x] **Settings tabs now register through the contribution registry.** New `contributions.registerSettingsTab({ id, title, icon, group, component, order? })` API + `useSettingsTabs()` React hook added in `app/src/contributions/registry.ts`. The three built-ins (General / Hotkeys / Plugins) register themselves in `app/src/contributions/builtins.ts` alongside `files` / `outline` content-types. `SettingsModal.tsx` dropped the hardcoded `BUILTIN_TABS` array and now resolves both the rail list and the active-tab component out of the registry; plugin-contributed tabs continue to flow through `useSettingsStore.pluginTabs` ŌåÆ `PluginSettingsTab` (backend-owned) but are merged into the rail display. Ordering is deterministic via the optional `order` hint. `tsc -b` + `vite build` pass clean.
- [x] **Ribbon `togglePanel` action is a no-op.** `app/src/components/layout/Ribbon.tsx:44-48` logs `console.log` and does nothing. Plugin ribbon items can't currently show/hide a side panel. Either drop the `togglePanel` kind from `RibbonAction` or wire it to `useLayoutStore.togglePanelVisibility("left"/"right", panelId)` with a `side` field on the action.

- [ ] **Rust-owns-state model conflicts with PRD-08 ┬¦4 CM6 architecture.** PRD-08 ┬¦4 specifies CodeMirror 6 as the live document model: CM6 holds the text, Rust provides decorations and a `BlockPositionMap`. The current implementation inverts this ŌĆö `EditorCorePlugin` owns the authoritative `BlockTree` and every mutation requires a Tauri IPC round-trip (`editor_apply_transaction`) that serializes and returns the *full* block tree. For large documents this is O(blocks) per keystroke and cannot achieve real-time editing performance. Two paths forward: (a) adopt the PRD model ŌĆö CM6 owns character-level state in the frontend; Rust is called at coarser granularity (commit, save, semantic analysis); or (b) formally amend PRD-08 ┬¦4 to reflect the IPC-snapshot model and design a debounce/batching layer so keystrokes are coalesced before dispatch. A decision is needed before building the collaborative-editing or AI-inline-edit layers, both of which assume a defined document-state owner.

### PRD gap ŌĆö no owner for plugin-contributed tab surfaces

- [ ] **Add a PRD section (or new PRD) owning "Tab Surface Registration".** The `Surface` enum in PRD-07 ┬¦5.1 includes `"custom"` and PRD-06 defines `.canvas`, `.bases`, `.mdx` as first-class formats, but **no PRD specifies the registration API that lets a plugin contribute a new tab surface kind or claim a file extension.** Today, opening `notes/board.canvas` falls through to `FileViewer` ŌåÆ CodeMirror and displays raw JSON, because there's no `contributions.registerTabSurface(kind, component)` or `contributions.registerFileHandler(ext, surfaceKind)` contract. Options: (a) extend PRD-07 with a new ┬¦7.4 "Tab Surface Registration" paralleling ┬¦8.2 "Panel Registration", or (b) write a standalone extensibility-surfaces PRD covering tab surfaces + file-extension handlers + editor surface registration in one place. Without this, Canvas/Bases/future document editors have no architectural home.

## Editor-shell capability gaps (2026-04-16) ŌĆö vs VS Code / Obsidian / IntelliJ

### Spec'd in a PRD, not yet implemented

- [ ] **CLI subcommands from plugins (PRD-04 ┬¦1.1 `cli_subcommands` + PRD-05 ┬¦19.1).** `nexus-cli/src/main.rs` is pure clap-derive; no extension point wires plugin-registered subcommands into the argv parser. Required to match "plugins can add `nexus myplugin <args>`" in PRD-05 ┬¦19.1. Blocks the "CLI = headless entry point to every plugin" story in PRD-05 ┬¦1.

### Half-specced: manifest keys exist, but no UI/wiring spec in PRD-07

- [ ] **Menu-bar extension point (PRD-04 ┬¦1.1 `menu_items` has no home in PRD-07).** PRD-04 declares `registrations.menu_items` (File/Edit/View contributions, with `menu` + `index`) but PRD-07 doesn't specify a menu bar component, native-vs-custom strategy, or a plugin registration API for it. The current Tauri shell has no menu bar at all. Either add a ┬¦7.5 "Menu Bar" section to PRD-07 covering the component, native/platform-menu integration, and the `contributions.registerMenuItem(menu, item)` API, or drop `menu_items` from PRD-04. Related: macOS native menus vs Windows/Linux custom menu bar decision.
- [ ] **Context-menu extension point (PRD-07 ┬¦13.1 lists the component; no plugin API).** No `contributions.registerContextMenuItem(scope, item)` is specced. Plugins can't add right-click entries to the file tree, tabs, editor selection, etc. Add to PRD-07 alongside the menu-bar work.
- [ ] **Protocol/URL handlers (PRD-04 ┬¦1.1 `protocol_handlers` manifest has no UI wiring spec).** Manifest declares `scheme = "nexus", path = "/analyze"` but no PRD specifies how incoming `nexus://ŌĆ”` URLs are dispatched to plugins, how OS-level scheme registration happens, or the in-app `contributions.registerUriHandler(scheme, handler)` API. Needs a PRD-07 or PRD-04 section mapping manifest ŌåÆ runtime dispatch.

### Not in any PRD ŌĆö new spec work needed

- [ ] **Webview / iframe panels.** VS Code `WebviewPanel` and Obsidian's HTML-returning panels let plugins render arbitrary HTML/React trees ŌĆö richer than the current "contentType ŌåÆ React component" model where the component is baked into the plugin's bundled JS. No PRD addresses sandboxed webviews for WASM plugins or iframe-based content from community plugins. Needed for: Dataview-style rich panels, embedded diagrams, external viewer embedding. Likely a new PRD-07 section or an extensibility-surfaces PRD.
- [ ] **Plugin-facing host UI APIs (notifications, quick-pick, input prompts).** PRD-07 ┬¦13.1 lists `Toast`, `CommandPalette`, `Popover` as UI components, but no PRD exposes them as an API that WASM or script plugins can call ŌĆö e.g., `nexus_notify(level, message)`, `nexus_quick_pick(items) ŌåÆ choice`, `nexus_input_box(prompt) ŌåÆ string`. VS Code's `vscode.window.showInformationMessage` / `showQuickPick` / `showInputBox` are everyday plugin tools. Needs a section in PRD-04 (host capability surface) or PRD-07.
- [ ] **Snippets.** Text-expansion snippets (VS Code `snippets.json`, Obsidian's text snippets plugins). No PRD spec; nothing in the editor engine (PRD-08) addresses it. Decide scope: editor-only (PRD-08 extension) or first-class plugin contribution (PRD-04 new registration key `snippets`).
- [ ] **Language Server Protocol (LSP) integration.** Not mentioned in any PRD. Matters if Nexus wants to host code blocks with IntelliSense or be used as a general editor. Would live in PRD-08 ┬¦14 as a new ┬¦14.5 or as a separate PRD.
- [ ] **Debug adapter protocol (DAP).** Not specced. Out of scope for a knowledge tool today, but worth an explicit "not in roadmap" note so the extensibility surfaces don't accidentally foreclose it.
- [ ] **Task runners.** No PRD covers `tasks.json`-style build/run tasks. PRD-09 (Terminal) handles processes but not the task-definition + keybinding + output-pane story. Either a ┬¦9.N in PRD-09 or a new Phase-3 PRD.
- [ ] **Tree-data providers.** VS Code's `TreeDataProvider` lets a plugin populate a generic tree widget (file tree, outline, etc.). Nexus currently has `FileTree` and `Outline` as hardcoded built-in panel content-types. A `contributions.registerTreeDataProvider(viewId, provider)` API would let plugins ship custom tree panels without each writing a bespoke React component. Add to PRD-07 ┬¦8 (Sidebar Architecture) alongside the ┬¦8.2 Panel Registration spec.
