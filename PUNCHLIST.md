## Architecture audit (2026-04-16) — follow-ups

Findings surfaced by the microkernel + editor-shell audit that weren't already tracked above.

- [x] **`host::emit_event` is not capability-gated.** `crates/nexus-plugins/src/host_fns.rs:275` calls `event_bus.publish_plugin` with no capability check, so any loaded WASM plugin can publish bus events regardless of manifest. ## Microkernel hardening — 2026-04-16 audit findings

Findings from `docs/MICROKERNEL-AUDIT.md` not yet tracked. Ordered by audit priority. The three 🔴 items and F-9.2.1 are blockers before any public plugin marketplace.

### 🔴 Red — blockers for untrusted plugin distribution

- [ ] **Install-time capability prompt for HIGH-risk capabilities (F-5.1.1).** `crates/nexus-plugins/src/loader.rs:1222-1236` — `build_capabilities` auto-grants the union of `required + optional` with no user interaction. `nexus-security::risk_level` classifies `net.http`, `fs.read.external`, `fs.write.external`, `process.spawn`, `ipc.call` as HIGH but that classification is never consumed by the loader. Fix: at `PluginLoader::load`, compute `high_risk_caps = (required ∪ optional).filter(risk == High)`; emit `com.nexus.security.capability_prompt` and block until user responds; persist decision to `<plugin_dir>/granted_caps.json` keyed by (plugin, version); re-prompt on version bump for newly-requested HIGH-risk caps.
- [ ] **Fix TOCTOU in `host::write_file` and `KernelPluginContext::write_file` (F-5.3.1, F-5.3.2).** `crates/nexus-plugins/src/host_fns.rs:386-412` canonicalizes the *parent* directory then writes to the non-canonical `absolute` path — a symlink race between those two operations is a sandbox escape for any plugin holding `FsWrite`. Same pattern in `crates/nexus-kernel/src/context_impl.rs:174-186`. `ForgePathValidator` in `crates/nexus-security/src/path.rs` already does this correctly (canonicalize target, prefix-check, symlink tests included) but is not used by either write site. Fix: route both through `ForgePathValidator`; open the file via the canonical path returned by the validator.

- [x] **Enforce `max_execution_ms` via wall-clock deadline (F-8.4.1).** `WasmSandbox::dispatch` now arms an epoch-interruption watcher thread before each call: sleeps `max_execution_ms` then calls `engine.increment_epoch()`, which wasmtime converts to `Trap::Interrupt`. `map_trap_error` maps both `OutOfFuel` and `Interrupt` to `PluginError::ExecutionTimeout`. - [ ] **Deterministic plugin shutdown order (F-4.2.2).** `crates/nexus-plugins/src/lib.rs` — `shutdown` iterates `all_plugin_ids()` in hashmap order. Fix: stop plugins in reverse-registration order; emit an audit event for each `on_stop` failure in addition to the `warn!`.
- [ ] **`reload_plugin` retry + rollback (F-4.3.1).** `crates/nexus-plugins/src/loader.rs` — on a failed new-sandbox build the plugin transitions to `Crashed` with no retry. A mid-write WASM file triggers this. Fix: retry once after 100 ms; if both attempts fail, keep the previous (pre-reload) sandbox so the plugin remains functional.
- [ ] **Emit `PluginReloading` during live reload (F-4.4.1).** `crates/nexus-plugins/src/error.rs:121-124` declares `PluginError::PluginReloading` but it is only constructed in tests. An IPC call arriving mid-reload races the backend `Mutex` and surfaces as `CapabilityDenied`. Fix: set an `AtomicBool reloading` flag per plugin; return `PluginError::PluginReloading` from `dispatch_ipc_checked` when set.
- [x] **Plugin discovery respects `plugin_search_paths` from kernel config (F-3.3.1).** `crates/nexus-bootstrap/src/lib.rs:122` hardcodes `.forge/plugins` regardless of `plugin_search_paths` in `.nexus/config.toml` (which the kernel does load — `crates/nexus-kernel/src/config.rs`). ## Suspected issues — not fully investigated

Threads from `docs/MICROKERNEL-AUDIT.md §Suspected Issues` that warrant a targeted code walk.

- [ ] **Subscriptions persistence integrity.** `crates/nexus-plugins/src/loader.rs` `load_disabled_subscriptions` silently defaults to an empty set on any `serde_json` parse error — a corrupt `subscriptions.json` re-enables every subscription without warning. Consider checksumming or failing loudly.
- [ ] **Fuel replenishment per call vs. per lifetime.** Verify whether `dispatch` (WASM sandbox) refills fuel between handler invocations or expects plugins to be short-lived per-dispatch. A long-lived plugin with `fuel = 1_000_000` that accumulates calls will eventually return `OutOfFuel` on every handler.

## UI audit (2026-04-16) — follow-ups

Findings from `docs/UI-AUDIT.md` not yet tracked above. IDs reference the audit. The 🔴 items plus F-9.1.1 are blockers before any untrusted-plugin distribution.

### 🔴 Red — cannot ship to untrusted users without these

- [ ] **Move JS script plugin execution into a sandboxed iframe (UI F-8.1.1).** `app/src/plugins/scriptRuntime.ts:61-67` loads plugin modules via `URL.createObjectURL(new Blob([...], { type: "application/javascript" }))` + `import(url)` straight into the main WebView thread. Combined with F-5.1.2 (no CSP) and F-2.2.1 (JS caps unenforced), an untrusted plugin has full DOM + `invoke` + `localStorage` + `fetch` access. Fix: host JS plugins in an `<iframe sandbox="allow-scripts">` (no `allow-same-origin`) with a typed `postMessage` protocol; expose `NexusPluginContext` as a message-passing proxy. Large effort (1–2 eng-months) but required by the stated trust model.
- [ ] **Enable a strict Tauri CSP (UI F-5.1.2).** `crates/nexus-app/tauri.conf.json` sets `"security": { "csp": null }`, removing the last backstop against injected-script exfiltration. Fix: enable at least `default-src 'self' tauri:; script-src 'self'; connect-src 'self' tauri: https:; frame-src tauri:`. Gate Vite HMR's `'unsafe-eval'` behind `NODE_ENV=development`. Audit every plugin-contributed inline `style=` for strict-CSP breakage.
- [ ] **Drop `allow-same-origin` from webview-panel iframes (UI F-5.1.1).** `app/src/contributions/registry.ts:1195` builds panel iframes with `sandbox = "allow-scripts allow-same-origin"`. The inline comment claims cross-origin safety, but the documented `htmlUrl` example in `nexusContext.ts:176` is same-origin, so `document.cookie` / `window.top` remain reachable. Fix: drop `allow-same-origin`; serve panels from a synthetic `tauri-plugin://<id>/...` scheme validated at registration time; expose host-mediated storage through `ctx` for panels that need persistence.
- [ ] **Bind `pluginId` at the sandbox boundary, not in the JS context (UI F-8.1.2).** `app/src/plugins/nexusContext.ts:184` — `createNexusContext(pluginId)` trusts the string; any plugin can re-create a context claiming another plugin's id, affecting `ctx.events.emit`, `ctx.ui.notify` `source`, settings namespace, and per-plugin disposables. Fix: derive identity at the iframe/worker boundary (depends on F-8.1.1); reject any host call whose asserted id disagrees with the boundary id.

> F-9.1.1 (validate `api_version` at load time) is the UI twin of the microkernel 🟠 item of the same ID already tracked above — no duplicate entry.

### 🟠 Orange — substantive design gaps, schedule before next external release

- [ ] **Extract `@nexus/extension-api` TS package with semver (UI F-2.1.1 / F-9.2.1).** The host API a JS plugin sees is the runtime shape of `NexusPluginContext` + `contributions/index.ts`; there is no stable importable type package, so every `NexusPluginContext` change silently breaks every script plugin on next load. Fix: extract `NexusPluginContext`, contribution DTOs (`EditorBlockType`, `TreeDataProvider`, `MenuItem`, `UriHandler`, `WebviewPanelConfig`, `Snippet`, …) and minimal runtime validators into `packages/nexus-extension-api/`; publish semver; make `@nexus/extension-api@^1` the only supported import path. Separate deliverable from the Rust `nexus-plugin-api` crate already tracked in the microkernel section.
- [ ] **Enforce manifest `capabilities.required` / `optional` in `createNexusContext` (UI F-2.2.1).** `app/src/plugins/nexusContext.ts:184-226` hands out `settings` / `events` / `ipc` / `editor` / full `ui` unconditionally regardless of manifest. A plugin that did not declare `ui.registerContextMenuItem` can still register context menu items. Fix: read declared capabilities at load time and build a `NexusPluginContext` that exposes only the requested surface; stub the rest with a throw-on-call that logs the capability error. Pairs with microkernel F-5.1.1.
- [ ] **Activation events for script plugins (UI F-3.2.1).** `[lifecycle] on_init = true` forces eager activation on shell start; with N script plugins, `onInit` runs serially on the UI thread before chrome renders. Fix: extend manifest with `[activation] on_command = [...]` / `on_content_type = [...]` / `on_uri_scheme = [...]`; only read a script plugin's module when its first matching activation event fires. Pairs with UI F-10.3.1.
- [ ] **Keybinding-conflict UI in Settings (UI F-4.1.1).** `app/src/keybindings/KeybindingDispatcher.tsx:21-25` documents a silent first-wins policy with only a `console.warn`. Fix: (a) surface a conflict event on the registry; (b) add a keybinding-conflict panel in Settings listing all conflicts with rebind affordances; (c) persist resolutions via the existing `keybindings.set_keybinding_override` command.
- [ ] **Declarative plugin-panel primitives — `registerPanelView` (UI F-5.2.1).** `app/src/components/panels/PluginPanel.tsx` renders plugin output as `<pre>{result.content}</pre>`, which is safe but useless for interactive UI. Pattern used by Raycast and Sketch.
- [ ] **Per-plugin error boundary in the frontend (UI F-7.2.1).** If a script plugin's `onInit` throws, the exception propagates up `registerPluginContributions` and can abort registration for every plugin loaded after it. Fix: wrap every plugin lifecycle call (`onInit`, `onStart`, `onStop`, `dispatch`) in a per-plugin try/catch that logs against the plugin, marks it "failed" in a `plugins:status` store, and offers a disable/uninstall action surfaced in the Settings → Plugins tab (UI F-10.1.1).
- [ ] **UI-thread time budget on plugin command dispatches (UI F-8.2.1).** `contributions.invokeCommand(id)` calls the handler synchronously from the UI thread; a 5 s plugin command freezes the palette, ribbon, and menu bar. - [ ] **Memory budget / accounting for script plugins (UI F-8.3.1).** WASM plugins have `memory_mb = 8` in their manifest; script plugins have no equivalent and allocate against the WebView heap directly. A plugin that accumulates a 500 MB structure OOMs the whole shell. Fix: track `performance.measureUserAgentSpecificMemory()` per plugin-iframe when F-8.1.1 lands; warn on budget overrun. Depends on F-8.1.1.
- [ ] **Deprecation channel for the extension API (UI F-9.3.1).** No `DEPRECATED.md`, no `@deprecated` JSDoc on contribution DTOs, no per-plugin deprecation log. Fix: adopt the VS Code pattern of `@deprecated` JSDoc tags on contribution types plus a per-plugin deprecation report generated at load time. Cheap once `@nexus/extension-api` lands (UI F-2.1.1).
- [ ] **"Show Running Extensions" tab in Settings (UI F-10.1.1).** No UI surface enumerates loaded plugins, activation state, capabilities, recent errors, or contribution counts. Fix: ship a Settings → Plugins tab listing each plugin's id / name / version / runtime tier / trust level / status (loaded / failed / deactivated) / declared+used capabilities / contribution counts per extension point / last-error / slowest command.
