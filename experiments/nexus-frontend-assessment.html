Nexus frontend — performance assessment

Honest read of the Nexus desktop shell against the bar Zed sets in experiments/zed-frontend-analysis.md.

1. Executive summary

Nexus's frontend is a Tauri 2 + React 18 + Vite WebView app with a CodeMirror 6 editor, an xterm.js + WebGL terminal, a Zustand-backed plugin shell, and ~53 first-party plugins lazy-loaded as Vite chunks. It is architecturally closer to VS Code minus Electron than to Zed: the rendering path is still the platform compositor through a Chromium/WebKit DOM, not a GPU scene graph. The differentiator vs. VS Code is the host runtime (native Tauri shell, no Node main process, no Electron bundle) and the disciplined plugin boundary (IPC-only kernel access, WASM-sandboxed community plugins).

Realistic expectations versus the Zed/VS Code numbers in the source doc: cold start and memory should land roughly in the middle — 2–3× better than VS Code, 2–4× worse than Zed — with input latency and frame rate bounded by the WebView's compositor (16 ms / 60 FPS practical ceiling, with React reconciliation eating into that budget). The two largest concrete performance risks today are (a) the non-virtualized files tree and (b) the React reconciliation cost on hot paths like outline / search results. Both are fixable without architectural change.

2. The stack, mapped to Zed's pipeline

Zed's GPUI pipeline is prepaint → paint → present against a Metal/DX/Vulkan backend at a 120 FPS target. Nexus's analogue is the browser engine: React reconciliation → DOM/CSSOM → compositor, vsync-locked at 60 FPS in every platform WebView Tauri targets. The architectural layers map roughly as:

3. Where Nexus is doing the right things

3.1 Lazy, code-split plugin loading

shell/src/plugins/catalog.ts uses dynamic import() per plugin, and vite.config.ts groups heavy vendor libs into named chunks (vendor-codemirror, vendor-xterm, vendor-mermaid, vendor-react). Default-off plugins are not loaded at all until the user opts in. This is the right pattern for keeping the initial bundle small, and it mirrors how VS Code handles its extension host (but at a finer grain — per-plugin chunks rather than a single extension-host process). Boot wall-clock scales with DEFAULT_ON_PLUGINS, not with the full 53-entry catalog.

3.2 Per-platform build targets

// shell/vite.config.ts
target: process.env.TAURI_PLATFORM === 'windows' ? 'chrome105'
      : process.env.TAURI_PLATFORM === 'macos'   ? 'safari15'
      : 'es2022'

Branching the esbuild target per OS means each platform bundle ships only the transforms that platform's WebView actually needs — smaller bytes, faster parse, no dead polyfills. This is a small win that is easy to skip, and Nexus didn't.

3.3 IPC-only kernel boundary

Every backend capability the UI consumes routes through context.ipc_call(plugin_id, command, args). That keeps the frontend's job constrained to "render and dispatch", and makes the WebView fundamentally cheap: it doesn't load heavy data into JS heap, it asks the kernel and renders the result. This is the architectural twin of Zed's snapshot model — the frame's data is a query result, not a live mutation target.

3.4 GPU-accelerated terminal

@xterm/addon-webgl is on. The terminal is one of the two glyph-heavy surfaces (the editor being the other), and getting it onto WebGL closes a real gap; this matches Zed's "let the GPU draw the letters" principle in the one place the browser cleanly allows it.

3.5 Sandboxed community plugins

The SandboxOrchestrator mounts each community plugin in an iframe with a shared bootstrap runtime (Blob-cached once per session), wired through postMessage and capability-gated by the kernel. This delivers exactly the Zed pitch quoted in the source doc — "extensions can't freeze the main thread" — and the implementation is sound: per-plugin PluginAPI, storage namespacing on the authoritative pluginId, and a single shared runtime URL to amortize the bootstrap cost.

4. Where Nexus is leaving performance on the table

4.1 Files tree is not virtualized high impact

shell/src/plugins/nexus/files/FilesTree.tsx renders the tree recursively — every expanded TreeNode is a real React component with its own subscriptions, refs, and DOM nodes. There is no windowing. The Zed analysis calls out uniform_list as the file-tree pattern, and the source doc explicitly flags it as something Nexus should adopt regardless of framework. At ~200 visible rows the cost is invisible; at a 10k-file forge with deep expansion, paint and event-handler attachment scale linearly. @tanstack/react-virtual is already a dependency (used in BasesTable.tsx) so the fix is local.

4.2 React StrictMode double-renders in dev dev-only, but misleading

main.tsx wraps both <App /> and <PopoutShell /> in <React.StrictMode>. This is correct for catching effect bugs but it doubles every effect and render in dev. When profiling locally, this masks real wins and inflates apparent costs. Worth remembering when reading tauri:dev traces.

4.3 No explicit per-frame snapshot for cross-subsystem state

Zed's EditorSnapshot is the answer to "the LSP reply arrived mid-frame and you painted a torn view." Nexus mostly avoids the problem by routing through the IPC kernel (one-shot queries, not live refs), but features that compose multiple stores at render time — e.g. editor + git blame + LSP diagnostics + outline — could benefit from a deliberate useSnapshot() pattern that picks a coherent tuple per frame, rather than independent selectors that each re-render on their own cadence. This isn't a bug today; it's the kind of thing that bites once any one of those streams gets noisy.

4.4 Mermaid is heavy budget the chunk

mermaid + d3 + dagre are grouped into vendor-mermaid; good. But Mermaid pulled in eagerly the first time a markdown file contains a diagram fence will add noticeable hitch to the first paint of that document. Worth verifying the diagram extension defers Mermaid until a ```mermaid block is actually visible, not on editor mount.

4.5 React vs. fine-grained reactivity

The source doc explicitly recommends Solid.js for the editor shell as "the pragmatic shortcut where it doesn't hurt the product." Nexus chose React. That's defensible — the ecosystem is bigger, hiring is easier, plugin authors expect it — but it does mean the framework's reconciliation cost is permanent overhead the team is paying versus Solid's compile-to-DOM-ops approach. The right mitigation isn't to rewrite; it's to be disciplined about memo / store-selector granularity on hot rows (file-tree row, outline item, search-result row, gutter line). The existing RowStatusDot pattern in FilesTree.tsx — pulled into its own component so the parent doesn't re-render on every cache mutation — is the right idea applied locally; it should be the norm, not the exception.

6. What Zed's lessons actually mean for Nexus

Adopt the "snapshot" idiom for multi-store renders

Where a component reads from editor + LSP + git + theme together, write a single useFrameSnapshot hook that pulls a coherent tuple and memoizes it. This is the React analogue of EditorSnapshot, and it eliminates the "three selectors fire on three different ticks" class of jank.

Virtualize the files tree

@tanstack/react-virtual is already in package.json and used in BasesTable.tsx. Apply the same pattern to FilesTree.tsx. This is the single highest-ROI change available for users with large forges, and it directly closes the uniform_list gap the source doc flags.

Treat the WebView like a GPU consumer where possible

The xterm WebGL addon proves the model works. Three more places could plausibly benefit: the graph view (already using force-directed layout), the canvas plugin, and the markdown live-preview syntax highlighting on the active line. Anywhere a primitive can be drawn to a <canvas> instead of a DOM subtree, the browser stops re-running layout on every keystroke.

Publish a benchmark suite

The source doc's analytical power comes from independent measurements. Nexus would benefit from a small in-repo benchmark harness — cold-start trace, "open 50 MB file", "scroll 10k-row file tree", "render outline for 500-heading document" — with output committed under experiments/perf/ and run on a known machine. Without numbers, performance regressions land silently.

7. Bottom line

Nexus is not chasing Zed's frontend ceiling, and shouldn't — the cost of building a from-scratch GPU UI framework is exactly what the Zed doc enumerates (accessibility, IME, RTL, browser-native form controls, no web target). Nexus's design choice — Tauri + React + a disciplined IPC kernel — trades Zed's absolute performance for ecosystem reach, plugin-author ergonomics, and a meaningfully smaller surface area to maintain.

Inside that envelope, the work that pays for itself is concrete and local: virtualize the tree, budget the heavy vendor chunks, formalize per-frame snapshots, ship a benchmark harness. None of these requires changing the architectural bets. All of them close the gap between "this is acceptable" and "this feels right" on the surfaces a user touches dozens of times a day.
